## Descripción general del modelo de permisos

El control de acceso de slash commands opera en dos capas. La primera capa es _defaultMemberPermissions_, un flag a nivel de Discord configurado durante el registro del comando que controla qué usuarios ven el comando en el selector. La segunda capa es el pipeline _@UseGuards_, una verificación del lado del servidor que se ejecuta en el proceso del bot antes de que el handler se ejecute. Ambas capas trabajan juntas: Discord filtra la UI del lado del cliente, y los guards imponen reglas que Discord no puede verificar, como jerarquías de roles personalizadas, allowlists respaldadas por base de datos o restricciones basadas en tiempo.

Ninguna capa por sí sola es suficiente para la seguridad. _defaultMemberPermissions_ es una pista de UI que los administradores del servidor pueden sobrescribir a través de overrides de permisos de canal de Discord. Los guards se ejecutan en el bot pero requieren que la interacción llegue al bot primero. Usar ambas capas garantiza que usuarios no autorizados no puedan ver el comando y que, incluso si de alguna manera lo invocan, el pipeline de guards bloquee la ejecución antes de que cualquier lógica de negocio se ejecute.

El exception handler se ubica al final del pipeline. Cuando un guard deniega la interacción o un handler lanza un error, el framework captura el error, lo normaliza en una _SpraxiumException_, resuelve el layout de respuesta apropiado y envía un mensaje en Discord. Esto significa que los handlers nunca necesitan bloques try/catch crudos para escenarios de error esperados.

## Configurando defaultMemberPermissions

La propiedad _defaultMemberPermissions_ en la configuración de _@SlashCommand_ acepta un bitfield de permisos que determina qué usuarios pueden ver el comando por defecto. Usa el enum _PermissionFlagsBits_ de discord.js para construcción de bitfield con tipado seguro. Los patrones más comunes son requerir un solo permiso, combinar múltiples permisos con OR bit a bit, y configurar `null` para permitir a todos.

```typescript filename="commands/ban.command.ts"
import { SlashCommand, SlashOption } from '@spraxium/common';
import { PermissionFlagsBits } from 'discord.js';

@SlashCommand({
  name: 'ban',
  description: 'Ban a member from the server.',
  defaultMemberPermissions: PermissionFlagsBits.BanMembers,
})
export class BanCommand {
  @SlashOption.User('target', {
    description: 'The member to ban',
    required: true,
  })
  @SlashOption.String('reason', {
    description: 'Reason for the ban',
    required: false,
    maxLength: 512,
  })
  @SlashOption.Integer('days', {
    description: 'Number of days of messages to delete',
    required: false,
    min: 0,
    max: 7,
  })
  build() {}
}
```

## El pipeline de guards

Los guards son clases que implementan la interfaz _SpraxiumGuard_ con un único método _canActivate(ctx)_. El método recibe un _SpraxiumExecutionContext_ que envuelve la interacción y el nombre del comando, y retorna un booleano (o una promise que resuelve a un booleano). Retornar `true` permite que la interacción proceda al handler. Retornar `false` detiene el pipeline y dispara una _GuardDeniedException_.

El _GuardExecutor_ evalúa guards en un orden estricto: guards globales primero, luego guards de nivel de clase, luego guards de nivel de método. Se detiene en el primer resultado `false`. Esto significa que los guards globales actúan como verificaciones base que todo comando debe pasar, los guards de nivel de clase filtran clases handler específicas, y los guards de nivel de método proporcionan el control más fino.

## Escribiendo un guard personalizado

Una clase guard se decora con _@Injectable()_ (implícito a través del sistema DI) e implementa un método _canActivate_. El contexto de ejecución proporciona _getInteraction()_ para la interacción Discord cruda y _getCommandName()_ para el nombre del comando correspondido. Desde la interacción, puedes acceder al usuario, guild, canal, miembro y cualquier otro dato de Discord necesario para la decisión de autorización.

```typescript filename="guards/moderator.guard.ts"
import type { ExecutionContext, SpraxiumGuard } from '@spraxium/common';
import type { ChatInputCommandInteraction } from 'discord.js';

export class ModeratorGuard implements SpraxiumGuard {
  async canActivate(ctx: ExecutionContext): Promise<boolean> {
    const interaction = ctx.getInteraction() as ChatInputCommandInteraction;
    if (!interaction.memberPermissions) return false;
    return interaction.memberPermissions.has('ModerateMembers');
  }
}
```

El guard verifica si el miembro tiene el permiso _ModerateMembers_. Si la interacción es en DMs (sin _memberPermissions_), el guard retorna `false`. Si el miembro tiene el permiso, el guard retorna `true` y el handler procede. Esta es una verificación del lado del servidor que no puede ser evadida por overrides de permisos de Discord.

## Aplicando guards a handlers

Aplica guards a clases handler usando el decorador _@UseGuards_. Puedes pasar una o más clases guard, y se evalúan en el orden en que aparecen. Los guards también pueden aplicarse a nivel de método decorando el método _handle_ directamente.

<CodeTabs defaultIndex={0}>
  <Tab label="Guard de nivel de clase" language="typescript">
```typescript filename="handlers/ban.handler.ts"
import { Ctx, SlashCommandHandler, SlashOpt, UseGuards } from '@spraxium/common';
import type { ChatInputCommandInteraction, User } from 'discord.js';
import { ModeratorGuard } from '../../guards/moderator.guard';
import { BanCommand } from '../commands/ban.command';

@UseGuards(ModeratorGuard)
@SlashCommandHandler(BanCommand)
export class BanHandler {
  async handle(
    @Ctx() interaction: ChatInputCommandInteraction,
    @SlashOpt('target') target: User,
    @SlashOpt('reason') reason: string | null,
    @SlashOpt('days') days: number | null,
  ): Promise<void> {
    const reasonText = reason ?? 'No reason provided';
    await interaction.reply(`${target} has been banned. Reason: ${reasonText}`);
  }
}

````

## Opciones de guard con withOptions

Algunos guards aceptan opciones de configuración que cambian su comportamiento por handler. El framework soporta pasar opciones tipadas a guards usando el helper _withOptions_. Los campos de opción del guard se declaran con el decorador _@GuardOption_ en la clase guard, especificando valores por defecto y si la opción es obligatoria.

```typescript filename="guards/cooldown.guard.ts"
import {
  GuardOption,
  type ExecutionContext,
  type SpraxiumGuard,
} from '@spraxium/common';
import type { ChatInputCommandInteraction } from 'discord.js';

export class CooldownGuard implements SpraxiumGuard {
  @GuardOption({ default: 5000, required: false })
  duration!: number;

  private readonly timestamps = new Map<string, number>();

  async canActivate(ctx: ExecutionContext): Promise<boolean> {
    const interaction = ctx.getInteraction() as ChatInputCommandInteraction;
    const key = `${interaction.user.id}:${ctx.getCommandName()}`;
    const now = Date.now();
    const last = this.timestamps.get(key) ?? 0;

    if (now - last < this.duration) return false;

    this.timestamps.set(key, now);
    return true;
  }
}
```

<Callout tone="info" title="Las respuestas de error se manejan automáticamente">
  Cuando un guard retorna false o un handler lanza un error, el framework
  captura el error y envía una respuesta a Discord automáticamente. Lanza una
  subclase de *SpraxiumException* con un mensaje significativo y deja que el
  exception handler resuelva el layout de respuesta correcto.
</Callout>

## Configuración del exception handler

El comportamiento del exception handler se configura a través de la propiedad _exceptions_ en la configuración de Spraxium. Las opciones clave son _logUnhandled_ (si registrar errores que no son del framework en la consola) y _layouts_ (un mapa de clases de excepción a clases de layout para respuestas de error personalizadas).

```typescript filename="spraxium.config.ts"
import { defineConfig } from '@spraxium/core';

export default defineConfig({
  exceptions: {
    logUnhandled: true,
    layouts: [],
  },
});
```

Cuando _logUnhandled_ es `true` (el valor por defecto), cualquier valor lanzado que no sea una subclase de _SpraxiumException_ se registra con su mensaje y stack trace antes de ser envuelto en una _InternalException_. Configurarlo a `false` suprime la salida en consola pero aún envía la respuesta de error a Discord.

## Combinando permisos y guards efectivamente

El patrón recomendado para control de acceso de slash commands usa _defaultMemberPermissions_ como primer filtro y _@UseGuards_ como segundo. El bitfield de permisos reduce el número de interacciones no autorizadas que llegan al bot, y el guard valida condiciones que Discord no puede verificar.

<CodeTabs defaultIndex={0}>
  <Tab label="commands/ban.command.ts" language="typescript">
```typescript filename="commands/ban.command.ts"
import { SlashCommand, SlashOption } from '@spraxium/common';
import { PermissionFlagsBits } from 'discord.js';

@SlashCommand({
  name: 'ban',
  description: 'Ban a member from the server.',
  defaultMemberPermissions: PermissionFlagsBits.BanMembers,
})
export class BanCommand {
  @SlashOption.User('target', { description: 'The member to ban', required: true })
  @SlashOption.String('reason', { description: 'Reason for the ban', required: false })
  build() {}
}

````
  </Tab>
  <Tab label="handlers/ban.handler.ts" language="typescript">
```typescript filename="handlers/ban.handler.ts"
import { Ctx, SlashCommandHandler, SlashOpt, UseGuards } from '@spraxium/common';
import type { ChatInputCommandInteraction, User } from 'discord.js';
import { ModeratorGuard } from '../../guards/moderator.guard';
import { BanCommand } from '../commands/ban.command';

@UseGuards(ModeratorGuard)
@SlashCommandHandler(BanCommand)
export class BanHandler {
  async handle(
    @Ctx() interaction: ChatInputCommandInteraction,
    @SlashOpt('target') target: User,
    @SlashOpt('reason') reason: string | null,
  ): Promise<void> {
    if (!interaction.guild) {
      await interaction.reply({
        content: 'This command can only be used in a server.',
        flags: 'Ephemeral',
      });
      return;
    }

    const member = await interaction.guild.members.fetch(target.id).catch(() => null);
    if (!member) {
      await interaction.reply({
        content: `Could not find member ${target}.`,
        flags: 'Ephemeral',
      });
      return;
    }

    const reasonText = reason ?? 'No reason provided';
    await interaction.reply(`${target} has been banned. Reason: ${reasonText}`);
  }
}
````

## Checklist de diseño para guards y manejo de errores

<Steps>
  <Step title="Configura defaultMemberPermissions en comandos sensibles">
    Todo comando que ejecuta una acción destructiva o privilegiada debe
    configurar *defaultMemberPermissions*. Esto previene que usuarios no
    autorizados vean el comando en el selector, reduciendo la superficie de
    ataque para invocaciones accidentales o maliciosas. Usa
    `PermissionFlagsBits` para seguridad de tipos.
  </Step>
  <Step title="Usa guards para autorización a nivel de negocio">
    Los guards manejan condiciones que los permisos de Discord no pueden
    expresar: jerarquías de roles en base de datos, feature flags, restricciones
    basadas en tiempo, allowlists específicas por guild. Escribe clases guard
    enfocadas que verifican una condición cada una, y compónlas con múltiples
    entradas *@UseGuards*.
  </Step>
  <Step title="Deja que el exception handler gestione respuestas de error">
    No captures errores en handlers solo para enviar mensajes de error. Lanza
    subclases de *SpraxiumException* con códigos y mensajes significativos, y
    deja que el exception handler resuelva el layout correcto y envíe la
    respuesta. Esto mantiene los handlers limpios y garantiza formato de errores
    consistente en todo el bot.
  </Step>
  <Step title="Configura layouts personalizados para errores esperados">
    Registra clases de layout personalizadas para tipos de excepción que tu bot
    maneja frecuentemente. Una clase de layout puede renderizar un embed
    temático con el branding de tu bot, colores específicos para diferentes
    tipos de error, e instrucciones accionables para el usuario. Esto convierte
    las respuestas de error en una parte pulida de la experiencia del usuario.
  </Step>
  <Step title="Habilita logUnhandled en desarrollo">
    Mantén *logUnhandled* habilitado durante el desarrollo para que errores
    inesperados aparezcan en la consola con stack traces completos. En
    producción, considera si quieres logging en consola o si los errores deben
    fluir a un servicio de logging externo a través de un exception handler
    personalizado.
  </Step>
</Steps>

## Referencia de la API del contexto de ejecución

La clase _SpraxiumExecutionContext_ envuelve la interacción y proporciona una interfaz consistente para guards y exception handlers.

      <TableCell header="Tipo de retorno">Tipo de retorno</TableCell>
      <TableCell header="Descripción">Descripción</TableCell>
    </TableRow>
  </TableHead>
  <TableBody>
    <TableRow>
      <TableCell>*getInteraction()*</TableCell>
      <TableCell>`BaseInteraction | Message`</TableCell>
      <TableCell>
        Retorna la interacción Discord cruda o mensaje que disparó el comando.
      </TableCell>
    </TableRow>
    <TableRow>
      <TableCell>*getCommandName()*</TableCell>
      <TableCell>`string`</TableCell>
      <TableCell>
        Retorna el nombre del comando correspondido según registrado en la
        definición del comando.
      </TableCell>
    </TableRow>
  </TableBody>
</Table>

Los guards deben hacer cast del valor retornado por _getInteraction()_ al tipo esperado (como _ChatInputCommandInteraction_) antes de acceder a propiedades específicas del tipo como _memberPermissions_ o _guild_. El contexto de ejecución no impone el tipo de interacción para mantener flexibilidad entre diferentes sistemas de comando (slash, prefix, message component).

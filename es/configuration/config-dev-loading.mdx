## El bloque dev

El bloque _dev_ controla el watcher de desarrollo local que reinicia el proceso del bot ante cambios en archivos. Ajusta el _entrypoint_ a ejecutar, el intervalo de _debounce_ entre cambios y los directorios a incluir o excluir. El valor por defecto de cada campo es razonable para la mayoría de proyectos, así que solo configura lo que necesites cambiar.

</Table>

<Callout tone="warning" title="Excluye los directorios generados">
  Directorios como .spraxium/ y dist/ cambian durante las operaciones de build.
  Si no los excluyes, el watcher puede entrar en bucles de reinicio cada vez que
  un archivo generado cambie.
</Callout>

```typescript filename="spraxium.config.ts"
import { defineConfig } from '@spraxium/core';

export default defineConfig({
  dev: {
    entrypoint: 'src/main.ts',
    debounce: 500,
    include: ['locales'],
    exclude: ['.spraxium/**', 'dist/**'],
  },
});
```

## Orden de prioridad de nombres de archivo

Spraxium busca el archivo de configuración usando una cadena de prioridad específica por entorno. El primer archivo que exista en el sistema de archivos gana y la búsqueda se detiene. En modo producción, los directorios de salida compilados tienen prioridad. En los demás modos, el archivo fuente TypeScript tiene la mayor prioridad.

## Mapeo de NODE_ENV a ConfigEnv

El loader lee `process.env.NODE_ENV` una vez y lo mapea a un objeto _ConfigEnv_ con cuatro propiedades: _mode_, _isDev_, _isNeutral_ e _isProd_. Este objeto es el único argumento que tu callback recibe cuando usas el patrón de export con función. Cambiar _NODE_ENV_ después de que la config fue evaluada no tiene ningún efecto.

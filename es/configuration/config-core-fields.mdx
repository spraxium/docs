## debug

El flag _debug_ habilita salida de diagnóstico adicional en todos los subsistemas. Cuando está activo, el framework emite líneas de traza adicionales durante la carga de módulos, resolución de providers, registro de comandos y despacho de interacciones. Esta salida está pensada para troubleshooting y no debe permanecer activa en deployments de producción con alto volumen de tráfico.

El flag acepta un booleano simple o cualquier expresión de _ConfigEnv_, lo que permite acotar la verbosidad a un entorno específico.

## logger

El bloque _logger_ controla la política de salida del logger integrado del framework. No afecta la lógica de negocio. Las opciones más configuradas son _maskTokens_ (reemplaza strings con aspecto de token por un placeholder seguro), _commandLogging_ (emite una línea de traza en cada despacho de comando) y _discord_ (envía niveles de log seleccionados a un canal vía webhook).

```typescript filename="config/logger.config.ts"
import type { SpraxiumLoggerConfig } from "@spraxium/core";

export const loggerConfig: SpraxiumLoggerConfig = {
  maskTokens: true,
  commandLogging: true,
  discord: {
    type: "channel",
    webhookUrl: process.env.LOGGER_WEBHOOK!,
    levels: ["error", "warn"],
    channelId: "1462519588775395429",
  },
};
```

## plugins

El array _plugins_ recolecta entradas retornadas por las factories de plugin. Cada entrada porta un string _namespace_ y un payload de configuración tipado. El framework usa el namespace para enrutar la configuración al plugin correcto en runtime. Las factories de plugin y las opciones oficiales se tratan en la página de Registro de Plugins.

## dev

El bloque _dev_ tiene una página de referencia dedicada en esta categoría que cubre todos los campos del watcher, los nombres de archivo de configuración admitidos y cómo se resuelven los entornos al arrancar.

El bloque _prefix_ está documentado en la categoría de Comandos de Prefijo. El bloque _exceptions_ está documentado en la categoría de Excepciones.

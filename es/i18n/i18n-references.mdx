## Qué es una referencia y por qué existe

Una referencia es el puente entre un directorio en el disco y un espacio de nombres en el espacio
de claves de traducción. La declaras dándole un nombre y una ruta. El nombre se convierte en el
primer segmento de todas las claves que provienen de ese directorio. La ruta apunta a un directorio
cuya estructura sigue una de dos convenciones: archivos JSON planos por locale, o subdirectorios
por locale con múltiples archivos JSON dentro. Ambos formatos son válidos y pueden coexistir entre
distintas referencias en el mismo bot.

La razón por la que las referencias existen como concepto es que los bots reales tienen distintas
responsabilidades de traducción que pertenecen a lugares diferentes. Los nombres y descripciones
de slash commands son objetos de dominio. Los mensajes de error compartidos en todo el bot tienen
su propio espacio de nombres. Las cadenas de interfaz para modales, menús de selección y embeds
forman otra capa. Mezclar todo esto en un único archivo por locale crea un alto coste de
mantenimiento y dificulta identificar qué falta dónde.

## Declarando múltiples referencias

```typescript filename="config/i18n.config.ts"
import { defineI18n } from "@spraxium/i18n";

export const i18nConfig = defineI18n({
  defaultLocale: "es-ES",
  references: [
    { name: "slash", path: "./locales/slash" },
    { name: "ui", path: "./locales/ui" },
    { name: "errores", path: "./locales/errores" },
    { name: "embeds", path: "./locales/embeds" },
  ],
});
```

No hay límite en el número de referencias. Cada una se carga de forma independiente y se fusiona
en el mismo registro. Siempre que cada valor de `name` sea único, no puede haber conflictos de
claves entre referencias.

## Estructura de archivos planos

En este diseño, cada locale es un único archivo JSON en la raíz del directorio de la referencia.
El nombre del archivo es el identificador del locale y su contenido es el objeto de traducciones.

<FolderStructure>
  <Folder name="locales" open>
    <Folder name="errores" open>
      <File name="es-ES.json" />
      <File name="en-US.json" />
      <File name="pt-BR.json" />
    </Folder>
  </Folder>
</FolderStructure>

```json filename="locales/errores/es-ES.json"
{
  "noEncontrado": "Recurso no encontrado",
  "prohibido": "No tienes permiso para hacer esto",
  "timeout": "La operación expiró después de {{segundos}} segundos"
}
```

Tras cargar la referencia `{ name: 'errores', path: './locales/errores' }`, las claves disponibles
para `es-ES` son `errores.noEncontrado`, `errores.prohibido` y `errores.timeout`.

## Estructura de directorios anidados

Cuando quieres organizar las traducciones por característica dentro de un locale, puedes usar
subdirectorios. Cada subdirectorio lleva el nombre de un locale y contiene varios archivos JSON.
El nombre del archivo se convierte en un segmento adicional del espacio de nombres.

<FolderStructure>
  <Folder name="locales" open>
    <Folder name="slash" open>
      <Folder name="es-ES" open>
        <File name="ping.json" />
        <File name="ayuda.json" />
        <File name="config.json" />
      </Folder>
      <Folder name="en-US" open>
        <File name="ping.json" />
        <File name="ayuda.json" />
        <File name="config.json" />
      </Folder>
    </Folder>
  </Folder>
</FolderStructure>

```json filename="locales/slash/es-ES/ping.json"
{
  "name": "ping",
  "description": "Verificar la latencia del bot",
  "options": {
    "objetivo": {
      "name": "objetivo",
      "description": "El servidor o servicio a verificar"
    }
  }
}
```

Tras cargar la referencia `{ name: 'slash', path: './locales/slash' }`, las claves disponibles
para `es-ES` incluyen `slash.ping.name`, `slash.ping.description`, `slash.ping.options.objetivo.name`
y todas las demás claves de todos los archivos. La ruta completa siempre es:
`nombreReferencia.nombreArchivo.rutaClaveJSON`.

## Accediendo a claves entre referencias

Todas las referencias son accesibles a través de la misma API del servicio sin especificar qué
referencia contiene una clave. El nombre de la referencia forma parte de la clave en sí misma.

```typescript filename="estado.handler.ts"
import { Ctx, SlashCommandHandler } from "@spraxium/common";
import { I18nService } from "@spraxium/i18n";
import type { ChatInputCommandInteraction } from "discord.js";
import { EstadoCommand } from "../commands/estado.command";

@SlashCommandHandler(EstadoCommand)
export class EstadoHandler {
  constructor(private readonly i18n: I18nService) {}

  async handle(@Ctx() interaction: ChatInputCommandInteraction): Promise<void> {
    const locale = interaction.locale;
    const userId = interaction.user.id;

    const titulo = this.i18n.t("slash.estado.embed.titulo", locale);
    const errPerm = this.i18n.t("errores.prohibido", locale);
    const pieEmbed = await this.i18n.tUser(
      userId,
      "slash.estado.embed.pie",
      {},
    );

    await interaction.reply({ content: `${titulo}: ${pieEmbed}` });
  }
}
```

## Usando claves de referencia en decoradores de slash

Cuando añades un campo `i18n` a un decorador de slash, las rutas de clave apuntan al mapa plano
cargado, que usa el nombre de la referencia como primer segmento.

```typescript filename="config.command.ts"
import { SlashCommand } from "@spraxium/common";

@SlashCommand({
  name: "config",
  description: "Gestionar configuración del bot",
  i18n: {
    name: "slash.config.name",
    description: "slash.config.description",
  },
})
export class ConfigCommand {}
```

El lifecycle del i18n ya ha cargado el mapa plano antes de que el registrador de slash se ejecute.
El puente consulta el mapa de cada locale soportado por el bot y pasa el resultado a
_builder.setNameLocalizations()_. El comando aparece entonces en la interfaz de Discord con nombres
localizados de forma nativa.

## Qué ocurre cuando falta un locale

Cuando se solicita una clave para un locale sin archivo cargado, el registro retorna silenciosamente
al mapa del locale predeterminado. Si la clave tampoco existe allí, se devuelve la cadena de clave
sin procesar. El sistema nunca lanza errores en traducciones ausentes durante las peticiones, lo que te permite
publicar traducciones parciales e ir añadiendo locales de forma incremental.

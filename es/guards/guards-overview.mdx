## Qué es un guard

Un guard es una clase que responde una sola pregunta antes de que cualquier comando o listener corra: esta ejecución debe continuar? Si el guard dice si, el handler procede normalmente. Si el guard dice no, el framework detiene la ejecución inmediatamente y el método del handler nunca corre. Este es el concepto central detrás de los guards en Spraxium, y todo en esta categoría se construye sobre esta decisión simple de permitir o denegar.

En la práctica, esto significa que no necesitas escribir chequeos de permisos, validaciones de role o lógica de rate limit dentro de cada método handler. En su lugar, escribes esa lógica una sola vez en una clase guard y la aplicas donde sea necesario. El guard corre automáticamente antes del handler, y el handler solo maneja lógica de negocio porque la frontera de autorización ya paso. Esto mantiene tus command handlers limpios y tu política de acceso centralizada en clases dedicadas y reutilizables.

Los guards también se integran con la capa de exceptions. Cuando un guard niega ejecución, el framework puede renderizar una respuesta para el usuario a través de exception layouts sin que el handler necesite saber nada al respecto. Esta separación es lo que convierte a los guards en una frontera arquitectural verdadera en lugar de solo una función utilitaria que llamas manualmente al inicio de cada handler.

## La interfaz SpraxiumGuard

Todo guard implementa la interfaz _SpraxiumGuard_, que requiere un único método llamado _canActivate_. Este método recibe el _ExecutionContext_ actual y debe retornar un boolean o una promise que resuelve a boolean. Si el método retorna true, la ejecución continua al siguiente guard o al handler. Si retorna false, la ejecución se detiene y el framework lo trata como una denegación.

```typescript filename="guild-only.guard.ts"
import { Guard } from '@spraxium/common';
import type { ExecutionContext, SpraxiumGuard } from '@spraxium/common';

@Guard()
export class GuildOnlyGuard implements SpraxiumGuard {
  canActivate(ctx: ExecutionContext): boolean {
    return ctx.getGuildId() !== null;
  }
}
```

El método _canActivate_ soporta retornos síncronos y asíncronos. Si tu guard necesita consultar una base de datos, llamar una API externa o hacer cualquier operación async para decidir si la ejecución debe continuar, puedes retornar una _Promise_ y el framework esperara el resultado antes de tomar la decisión. Esta flexibilidad significa que los guards funcionan igualmente bien para chequeos instantáneos como verificar contexto de guild y para chequeos más pesados como consultar una base de datos de suscripciones.

## El ExecutionContext

El _ExecutionContext_ es el objeto que se pasa al método _canActivate_ de todo guard. Encapsula una interaction o message de Discord y provee una API unificada para que tu guard no necesite saber si el comando fue activado por un slash command o un prefix command. Cada método del contexto extrae una información específica sobre quien activo la ejecución y donde ocurrio.

Quizás te preguntes por qué _getUserId_ y _getUser_ existen por separado. La razón es rendimiento: si tu guard solo necesita comparar el ID del usuario contra una lista de owner IDs, llamar _getUserId_ evita resolver el objeto User completo. Si tu guard necesita más información como la fecha de creación de la cuenta, _getUser_ te da el objeto completo. Usa el que tenga sentido para el chequeo específico de tu guard.

## El pipeline de ejecución

Los guards no corren en orden aleatorio. Spraxium los evalua en tres capas, siempre en la misma secuencia: guards globales primero, luego guards de clase, luego guards de método. Si todos los guards en las tres capas retornan true, el método handler ejecuta. Si cualquier guard retorna false o lanza una exception en cualquier punto de la cadena, la ejecución se detiene inmediatamente y todos los guards restantes se saltan.

<Steps>
  <Step title="Guards globales corren primero">
    Se registran durante el bootstrap de la aplicación con *useGlobalGuards* en
    la instancia *SpraxiumApplication*. Se aplican a todo comando y todo
    listener con guard en toda la aplicación, lo que los hace el lugar correcto
    para reglas amplias como bloquear cuentas de bot, restringir acceso durante
    mantenimiento o aplicar rate limit global.
  </Step>
  <Step title="Guards de clase corren segundo">
    Se aplican con *UseGuards* en la clase del handler o listener. Se aplican a
    todo método dentro de esa clase, lo cual es útil cuando todos los comandos
    de un módulo comparten el mismo requisito de acceso. Un ejemplo típico es
    aplicar un guard de role en la clase para que todo subcommand de un handler
    de moderación requiera el role Moderator.
  </Step>
  <Step title="Guards de método corren último">
    Se aplican con *UseGuards* directamente en un método específico. Se aplican
    solo a ese método, lo que permite agregar requisitos más estrictos para
    operaciones sensibles como banear usuarios o resetear datos sin afectar el
    resto de los métodos de la clase.
  </Step>
  <Step title="Handler ejecuta">
    Solo después de que todo guard en toda capa retorno true el framework llama
    al método handler. Si tu código llega al handler, puedes estar seguro de que
    todas las políticas de acceso configuradas ya fueron satisfechas para el
    contexto de ejecución actual.
  </Step>
</Steps>

Este pipeline usa evaluación short-circuit. En el momento que cualquier guard niega la ejecución, el framework deja de evaluar y va directo al manejo de denegación. Esto es importante para rendimiento porque debes colocar chequeos baratos antes de los caros siempre que sea posible. Por ejemplo, un simple chequeo guild-only que lee una propiedad debe ir antes de una consulta a base de datos que verifica estado de suscripción premium.

<Callout tone="info" title="Guard versus lógica de negocio">
  Un guard responde "puede esto correr?" y nada más. El handler responde "que
  deberia pasar?". Mantén estas responsabilidades separadas para que los guards
  se mantengan pequeños, reutilizables y fáciles de testear independientemente.
</Callout>

## Creando un guard con @Guard

Para crear un guard, escribes una clase decorada con _@Guard()_ que implementa la interfaz _SpraxiumGuard_. El decorator marca la clase como guard en el sistema de metadata del framework, y la interfaz exige el método _canActivate_ que el pipeline llama en runtime. Este es el mismo patrón para cada guard que vas a escribir, ya sea chequeando una propiedad simple o consultando un servicio externo.

```typescript filename="guild-only.guard.ts"
import { Guard } from '@spraxium/common';
import type { ExecutionContext, SpraxiumGuard } from '@spraxium/common';

@Guard()
export class GuildOnlyGuard implements SpraxiumGuard {
  canActivate(ctx: ExecutionContext): boolean {
    return ctx.getGuildId() !== null;
  }
}
```

El guard de arriba chequea si el comando fue activado dentro de un guild. Si _getGuildId_ retorna null, el usuario está en DM y el guard niega la ejecución. Nota que el guard no envia ninguna respuesta ni loguea nada cuando niega. Eso es intencional: el único trabajo del guard es la decisión de permitir o denegar, y el framework maneja el resto a través del pipeline de exceptions.

Mantén cada guard enfocado en una sola responsabilidad. Un guard que chequea tanto membership de role como tipo de canal es más dificil de reutilizar que dos guards separados, uno para el role y otro para el canal. Guards más pequeños componen mejor y son más fáciles de testear porque cada uno tiene exactamente una razón para fallar.

## Aplicando guards con @UseGuards

El decorator _@UseGuards_ aplica guards a una clase o método específico. Cuando se coloca en la clase, todo método de esa clase pasa por esos guards. Cuando se coloca en un método, solo ese método específico recibe los guards extra. Puedes usar ambos al mismo tiempo, y los guards de clase siempre corren antes de los de método.

también puedes apilar múltiples guards en una sola llamada de _@UseGuards_. Se evaluarán en el orden que los listas, lo que significa que controlas la secuencia de evaluación explícitamente. Coloca chequeos baratos antes de los caros para que el pipeline pueda hacer short-circuit temprano sin correr trabajo innecesario. Cuando necesitas guards de método además de guards de clase, los guards de clase aún corren primero. Los guards de método agregan una capa extra de restricción que solo aplica a ese método específico, lo cual es útil cuando un módulo de comandos tiene un requisito general de acceso pero un subcommand particular necesita permisos adicionales.

<CodeTabs>
  <Tab label="Alcance de Clase" language="typescript">
```typescript filename="moderation.handler.ts"
import { UseGuards, withOptions } from '@spraxium/common';
import { RoleGuard } from '@spraxium/core';

@UseGuards(withOptions(RoleGuard, { roles: ['Moderator'] }))
export class ModerationHandler {
  async handle() {
    // todo método en esta clase requiere el role Moderator
  }
}

````
  </Tab>
  <Tab label="Guards Apilados" language="typescript">
```typescript filename="admin.handler.ts"
import { UseGuards, withOptions } from '@spraxium/common';
import { GuildOnly, PermissionGuard, RoleGuard } from '@spraxium/core';

@UseGuards(
  GuildOnly,
  withOptions(RoleGuard, { roles: ['Admin'] }),
  withOptions(PermissionGuard, { permissions: ['BanMembers'] }),
)
export class AdminHandler {
  async handle() {
    // chequeo de guild corre primero, luego role, luego permiso
  }
}
````

  </Tab>
  <Tab label="Clase + Método" language="typescript">
```typescript filename="settings.handler.ts"
import { UseGuards, withOptions } from '@spraxium/common';
import { GuildOnly, PermissionGuard } from '@spraxium/core';

@UseGuards(GuildOnly)
export class SettingsHandler {
  async viewSettings() {
    // solo necesita contexto de guild del guard de clase
  }

  @UseGuards(withOptions(PermissionGuard, { permissions: ['ManageGuild'] }))
  async editSettings() {
    // necesita contexto de guild Y permiso ManageGuild
  }
}

````

## Guards globales con useGlobalGuards

Los guards globales corren antes de todo guard de clase y de método en toda la aplicación. Los registras durante el bootstrap en la instancia *SpraxiumApplication* usando el método *useGlobalGuards*. Este es el lugar correcto para reglas que aplican en todas partes, como bloquear cuentas de bot, restringir el bot a IDs específicos de owner durante desarrollo, o aplicar un modo de mantenimiento global.

```typescript filename="src/main.ts"
import { SpraxiumFactory } from '@spraxium/core';
import { NoBotGuard, OwnerOnly } from '@spraxium/core';

const app = await SpraxiumFactory.create({ token: process.env.DISCORD_TOKEN! });

app.useGlobalGuards(NoBotGuard);
app.useGlobalGuards(OwnerOnly, { ownerIds: ['123456789', '987654321'] });

await app.listen();
````

El método acepta una clase guard como primer argumento y un objeto de opciones opcional como segundo. Si el guard necesita opciones como _OwnerOnly_ necesitando un array de owner IDs, las pasas directamente como segundo parámetro. Si el guard no tiene opciones como _NoBotGuard_, pasas solo la clase.

Se intencional sobre lo que va en alcance global. Un guard que solo importa para comandos de moderación no pertenece al registro global porque correria en todo comando incluyendo los de no-moderación. Mueve guards específicos de feature al alcance de clase o método, y reserva el alcance global para reglas verdaderamente universales.

## Entendiendo el pipeline combinado

Cuando guards globales, de clase y de método están todos presentes, forman un único pipeline ordenado que ejecuta como una cadena continua. El framework no los separa en chequeos independientes. Los procesa secuencialmente: todos los guards globales primero, luego todos los de clase, luego todos los de método. Si cualquier guard en cualquier punto de la cadena niega, todo después de el se salta.

```typescript filename="example-pipeline.ts"
// Global: NoBotGuard -> OwnerOnly
// Clase:  RoleGuard({ roles: ['Moderator'] })
// método: PermissionGuard({ permissions: ['BanMembers'] })

// Orden en runtime:
// 1. NoBotGuard.canActivate(ctx)
// 2. OwnerOnly.canActivate(ctx)
// 3. RoleGuard.canActivate(ctx)
// 4. PermissionGuard.canActivate(ctx)
// 5. handler ejecuta
```

Este ordenamiento predecible significa que tu equipo siempre puede entender cuáles guards corren y en qué orden mirando tres lugares: el archivo de bootstrap para guards globales, el decorator de clase para guards de clase, y el decorator de método para guards de método. No hay sorpresas ocultas de ordenamiento o caminos condicionales de evaluación.

<Callout tone="warning" title="Evita duplicar política entre alcances">
  Si un guard ya corre en alcance global, no apliques el mismo guard otra vez en
  alcance de clase o método. Guards duplicados desperdician ciclos de runtime y
  crean confusion sobre donde la política realmente se esta aplicando.
</Callout>

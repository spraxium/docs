## Configurando el logger

La configuración del logger vive en un archivo dedicado `config/logger.config.ts` exportado como `const` con nombre y pasado a _defineConfig_ dentro de `spraxium.config.ts`. Mantener cada preocupación en su propio módulo facilita la lectura y hace la config importable de forma independiente. El framework aplica la configuración automáticamente durante el bootstrap de la aplicación, antes de que cualquier provider comience a loguear.

```typescript filename="config/logger.config.ts"
import type { SpraxiumLoggerConfig } from '@spraxium/core';

export const loggerConfig: SpraxiumLoggerConfig = {
  levels: [
    { name: 'audit', color: 'blueBright' },
    { name: 'payment', color: 'greenBright' },
  ],
  maskTokens: true,
  discord: {
    type: 'channel',
    channelId: '1234567890',
    levels: ['error', 'warn', 'command'],
  },
};
```

```typescript filename="spraxium.config.ts"
import { defineConfig } from '@spraxium/core';
import { loggerConfig } from './config/logger.config';

export default defineConfig((env) => ({
  logger: loggerConfig,
}));
```

El array _levels_ registra niveles de log personalizados con sus colores en consola. Estos niveles quedan disponibles a través de _Logger.extend_ y son reconocidos por ambos transports. La flag _maskTokens_ controla el _TokenMasker_, que está habilitado por defecto. Establecerla en `false` deshabilita el ocultamiento de tokens, lo cual solo deberías hacer en desarrollo local cuando necesitas ver la salida completa.

## El transport Discord

El transport Discord envía entradas de log como embeds a un canal de texto o webhook. Solo reenvía mensajes cuyo nivel coincida con el array _levels_ en su configuración. Esto permite enviar errores y advertencias a Discord mientras mantienes mensajes info y debug solo en la consola.

Dos modos de entrega están disponibles. En modo _channel_, el transport usa el propio client del bot para encontrar el canal por ID y enviar embeds. En modo _webhook_, envía solicitudes HTTP POST a una URL de webhook, lo cual funciona incluso antes de que el client del bot esté listo y no requiere que el bot esté en el servidor destino.

<CodeTabs>
  <Tab label="Modo canal" language="typescript">
```typescript filename="config/logger.config.ts"
import type { SpraxiumLoggerConfig } from '@spraxium/core';

export const loggerConfig: SpraxiumLoggerConfig = {
  discord: {
    type: 'channel',
    channelId: '1234567890123456789',
    levels: ['error', 'warn'],
  },
};
```
  </Tab>
  <Tab label="Modo webhook" language="typescript">
```typescript filename="config/logger.config.ts"
import type { SpraxiumLoggerConfig } from '@spraxium/core';

export const loggerConfig: SpraxiumLoggerConfig = {
  discord: {
    type: 'webhook',

    levels: ['error'],
  },

```

</CodeTabs>

En modo canal, el transport encola mensajes que llegan antes de que el client esté listo. Cuando el client emite el evento _ready_, la cola se vacía automáticamente. La cola tiene un tamaño máximo de 100 entradas para prevenir problemas de memoria durante secuencias de arranque largas. Los mensajes que llegan después de que la cola esté llena se descartan silenciosamente.

El transport también deduplica entradas. Si el mismo nivel, contexto y mensaje se envían dentro de un segundo, el segundo envío se suprime. Esto previene logging duplicado accidental por edge cases en event handlers sin requerir coordinación de tu código.

## Templates de embed personalizados

Por defecto, los embeds Discord usan un formato simple con el nivel como título, el mensaje como descripción, un color que coincide con el nivel y un timestamp ISO. Puedes sobrescribir esto con un template de _embed_ personalizado que usa placeholders `{{key}}`.

````typescript filename="config/logger.config.ts"
import type { SpraxiumLoggerConfig } from '@spraxium/core';

export const loggerConfig: SpraxiumLoggerConfig = {
  discord: {
    type: 'channel',
    channelId: '1234567890123456789',
    levels: ['error', 'warn', 'command'],
    embed: {
      title: '{{level}} Log',
      description: "```\n{{message}}\n```",
      color: 0xff6b6b,
      timestamp: true,
      footer: { text: "Shard {{shard}} \u00b7 {{date}} {{time}}" },
      fields: [
        { name: 'Context', value: '{{context}}', inline: true },
        { name: 'Level', value: '{{level}}', inline: true },
      ],
    },
  },
};
````

Las variables de placeholder disponibles se resuelven de la entrada de log al momento del envío. La variable _shard_ muestra el ID del shard cuando se ejecuta en modo sharded, o "N/A" en modo single-process. Cualquier clave de metadata pasada por el método _log_ también está disponible como placeholder.

## Logging de comandos

El servicio _CommandLogger_ loguea automáticamente cada ejecución de slash command. Se vincula al evento _interactionCreate_ del client y captura el nombre del comando, tag del usuario, ID del usuario, nombre del guild, ID del guild e ID del canal. Cada ejecución emite una entrada de log en el nivel `command` con metadata estructurada que puedes referenciar en templates de embed.

La forma estándar de habilitar el logging de comandos es a través de `spraxium.config.ts`. Establece `commandLogging: true` dentro del bloque `logger` y el framework llama _CommandLogger.bind_ automáticamente durante el wiring del client. Este enfoque se integra con el ciclo de vida completo del framework y no requiere imports adicionales.

```typescript filename="config/logger.config.ts"
import type { SpraxiumLoggerConfig } from '@spraxium/core';

export const loggerConfig: SpraxiumLoggerConfig = {
  maskTokens: true,
  commandLogging: true,
  discord: {
    type: 'channel',
    channelId: '1234567890123456789',
    levels: ['command'],
    embed: {
      title: 'Command Executed',
      description: '`/{{command}}` by **{{user}}**',
      timestamp: true,
      footer: { text: '{{guild}} ({{guildId}})' },
    },
  },
};
```

Si estás usando el logger fuera del framework o necesitas vincular el logging de comandos manualmente, llama _CommandLogger.bind_ directamente con el client Discord después de que esté listo. Las claves de metadata disponibles en templates de embed son `{{command}}`, `{{user}}`, `{{userId}}`, `{{guild}}`, `{{guildId}}` y `{{channel}}`.

```typescript filename="src/main.ts"
import { CommandLogger } from '@spraxium/core';

// Llama después de que el client esté listo
CommandLogger.bind(client);
```

## Transports personalizados

Puedes crear transports para cualquier destino implementando la interfaz _LogTransport_. Un transport necesita una string _name_ y un método _log_ que recibe un objeto _LogEntry_. El método puede retornar void o una Promise. Si lanza o rechaza, el error se captura silenciosamente para prevenir que una falla de transport rompa la aplicación.

```typescript filename="src/transports/file.transport.ts"
import { appendFileSync } from 'node:fs';
import type { LogEntry, LogTransport } from '@spraxium/core';

export class FileTransport implements LogTransport {
  readonly name = 'file';

  constructor(private readonly path: string) {}

  log(entry: LogEntry): void {
    const line = `[${entry.timestamp.toISOString()}] ${entry.level.toUpperCase()} ${entry.context ?? ''} ${entry.message}\n`;
    appendFileSync(this.path, line);
  }
}
```

Registra el transport con _Logger.addTransport_. Si un transport con el mismo nombre ya existe, se reemplaza. Puedes remover un transport por nombre con _Logger.removeTransport_. El _ConsoleTransport_ siempre está registrado por defecto y usa el nombre `"console"`.

```typescript filename="src/main.ts"
import { Logger } from '@spraxium/core';
import { FileTransport } from './transports/file.transport';

Logger.addTransport(new FileTransport('./logs/bot.log'));
```

Si tu transport necesita el client Discord (por ejemplo para buscar nombres de canales o resolver datos de guild), implementa la interfaz _ClientAwareTransport_ en su lugar. Agrega un método _setClient_ que el framework llama automáticamente cuando el client está listo. El _DiscordTransport_ incorporado usa esta interfaz internamente.

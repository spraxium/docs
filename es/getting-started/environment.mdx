## Que hace el modulo de entorno

El paquete *@spraxium/env* reemplaza el patron comun de leer *process.env* directamente y esperar que los valores esten presentes en runtime. En lugar de dispersar accesos al environment por toda la base de codigo, defines una unica clase tipada que declara exactamente que variables necesita tu bot, que tipo debe tener cada una y cuales son opcionales. Cuando el bot arranca, `EnvValidator.validate` lee el archivo *.env*, verifica cada campo contra esas reglas y retorna un objeto completamente tipado o lanza un error claro listando exactamente que esta mal.

El beneficio practico es que las fallas de configuracion aparecen durante el inicio, no despues del deploy cuando un usuario dispara la feature afectada. Una URL de base de datos faltante o un endpoint de webhook malformado se convierte en un error de arranque con un mensaje descriptivo en lugar de un crash enigmatico en runtime. Este comportamiento fail-fast es especialmente valioso en entornos de produccion donde depurar problemas en vivo cuesta significativamente mas tiempo que capturarlos durante la secuencia de boot.

## Definir el schema

Crea una clase en *src/app.env.ts* que extienda `SpraxiumBaseEnv` para heredar los campos base que todo bot Spraxium necesita. La clase base ya incluye *TOKEN* y *NODE_ENV*, asi que solo necesitas agregar los campos especificos de tu aplicacion. Cada campo usa el decorator `@Env()` para vincularlo a una variable de entorno nombrada, y puedes apilar decorators de validacion como `@IsString()` y `@IsUrl()` para imponer reglas de formato automaticamente.

```typescript filename="src/app.env.ts"
import { Env, EnvSchema, IsOptional, IsString, IsUrl, SpraxiumBaseEnv } from '@spraxium/env';

@EnvSchema()
export class AppEnv extends SpraxiumBaseEnv {
  @Env('DATABASE_URL')
  @IsString()
  DATABASE_URL!: string;

  @Env('DISCORD_WEB', { secret: false })
  @IsString()
  @IsUrl()
  DISCORD_WEB!: string;

  @Env('ANALYTICS_TRACKING_ID', { secret: false })
  @IsString()
  @IsOptional()
  ANALYTICS_TRACKING_ID?: string;
}
```

La flag *secret* controla si el valor aparece en logs y salidas de diagnostico, lo cual es importante para mantener credenciales sensibles como tokens y claves de API fuera de los streams de log en produccion. Los campos sin `@IsOptional()` son obligatorios por defecto y fallaran en la validacion si estan ausentes del entorno, asegurando que tu bot nunca arranque con configuracion incompleta.

## Validar antes del boot

Llama a `EnvValidator.validate` como lo primero en *src/main.ts*, antes de crear la instancia de la aplicacion o cargar cualquier modulo. Esto asegura que cada variable de entorno haya sido verificada y tipada antes de que cualquier codigo del framework se ejecute, y te da un objeto completamente tipado que puedes pasar a traves de inyeccion de dependencias o usar directamente en la logica de bootstrap.

```typescript filename="src/main.ts"
import { EnvValidator } from '@spraxium/env';
import { AppEnv } from './app.env';

const env = EnvValidator.validate(AppEnv);
```

Despues de esta llamada, la variable *env* es una instancia completamente tipada de *AppEnv* con todas las validaciones aprobadas. Puedes pasar *env.token* a `SpraxiumFactory.create`, usar *env.DATABASE_URL* para inicializar una conexion de base de datos, o inyectar el objeto entero a traves de DI en cualquier provider que necesite valores de configuracion. Si cualquier campo obligatorio esta ausente o falla en sus reglas de validacion, el proceso termina inmediatamente con un mensaje claro antes de que cualquier modulo o provider sea instanciado.

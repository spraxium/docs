## Por qué importan los módulos

Spraxium escala a través de módulos, y cada feature en tu bot pertenece a exactamente uno de ellos, ya sea para eventos, moderación, comandos o integraciones externas. Cada módulo es responsable de declarar lo que provee, lo que importa de otros módulos y lo que expone de vuelta al resto de la aplicación. Este modelo de fronteras explícitas significa que puedes crecer la base de código agregando nuevos módulos sin aumentar la complejidad de los existentes, porque cada módulo es autocontenido, testeable independientemente y fácil de razonar de forma aislada.

Sin esta estructura, el código tiende a acumularse en archivos compartidos, las dependencias se vuelven implícitas y agregar una nueva feature corre el riesgo de romper algo no relacionado en una parte completamente diferente de la base de código. Los módulos previenen eso forzándote a ser explícito sobre las relaciones desde el principio, dejando claro a primera vista de qué depende cada parte del bot y qué provee a los demás.

## Arquitectura de módulo recomendada

El layout estándar aisla cada feature detrás de una frontera de módulo con su propia carpeta conteniendo como mínimo un archivo de módulo y un archivo de service. Los módulos evolucionan independientemente unos de otros, y el contenedor de DI se encarga de toda la conexión entre ellos al arrancar sin ninguna configuración manual ni funciones factory.

## Declarar el módulo raíz

El archivo de bootstrap *main.ts* es el punto de entrada donde validas el entorno, creas la instancia de la aplicación, cargas el módulo raíz y comienzas a escuchar eventos de Discord. La llamada `app.useModule(AppModule)` es el punto de entrada de composición, y desde ahí Spraxium carga todos los módulos importados recursivamente, resuelve providers a través de inyección de dependencias y vincula cada listener registrado al client de Discord automáticamente.

```typescript filename="src/main.ts"
import { IntentPreset, Logger, SpraxiumFactory } from '@spraxium/core';
import { EnvValidator } from '@spraxium/env';
import { AppEnv } from './app.env';
import { AppModule } from './app.module';

async function main() {
  const logger = new Logger('Main');
  const environment = EnvValidator.validate(AppEnv);
  const app = await SpraxiumFactory.create({ token: environment.token });
  app.useModule(AppModule);
  app.intents(IntentPreset.Standard);

  logger.info('Iniciando aplicación...');
  await app.listen();
}

main();
```

## Construir la raíz de composición

El módulo raíz *AppModule* compone features importando otros módulos pero nunca debe contener lógica de feature directamente. Cuando quieras agregar un nuevo dominio al bot, creas un módulo de feature dedicado y lo importas aquí, manteniendo la raíz delgada y legible sin importar cuán grande o complejo se vuelva el bot con el tiempo.

```typescript filename="src/app.module.ts"
import { Module } from '@spraxium/common';
import { AnalyticsModule } from '@spraxium/analytics';
import { NombreModuloModule } from './modules/nombre-modulo/nombre-modulo.module';

@Module({
  imports: [AnalyticsModule, NombreModuloModule],
})
export class AppModule {}
```

## Referencia de propiedades del módulo

<DropdownGroup>
  <Dropdown title="imports , cargar módulos antes del actual">
    Registra los módulos de los que depende el módulo actual. Spraxium los carga e inicializa en el orden correcto antes de instanciar los providers del módulo que los importa. Úsalo en *AppModule* para componer todos los módulos de feature, y en cualquier módulo de feature que dependa de providers exportados por otro.
  </Dropdown>
  <Dropdown title="providers , registrar clases inyectables en el contenedor">
    Lista las clases decoradas con *@Injectable()* que el contenedor de DI debe instanciar y administrar. Cada provider se crea una vez por módulo y se comparte entre todas las clases que lo inyectan. Úsalo para services, repositorios y cualquier clase de runtime con dependencias.
  </Dropdown>
  <Dropdown title="exports , exponer providers a otros módulos">
    Hace que los providers listados estén disponibles para cualquier módulo que importe este. Sin exports, los providers son privados al módulo que los declara. Úsalo cuando un service deba compartirse entre fronteras de módulo sin convertirlo en global.
  </Dropdown>
  <Dropdown title="listeners , registrar handlers de eventos de Discord">
    Enlaza clases listener al client de Discord para reaccionar a eventos de gateway como *messageCreate* o *guildMemberAdd*. Los listeners se instancian con soporte completo de DI, por lo que pueden inyectar cualquier provider exportado disponible en el módulo.
  </Dropdown>
  <Dropdown title="commands , registrar metadatos de comandos">
    Registra las clases de metadatos de *SlashCommand* y *PrefixCommand* que describen el nombre, descripción y opciones del comando. Spraxium usa estos metadatos para sincronizar comandos con Discord y enrutar interacciones entrantes al handler correcto.
  </Dropdown>
  <Dropdown title="handlers , registrar la lógica de ejecución de comandos">
    Registra las clases *SlashCommandHandler* y *PrefixCommandHandler* que contienen la lógica de ejecución actual. Los handlers reciben el contexto de interacción con acceso completo al DI del módulo, lo que los mantiene enfocados y fáciles de testear de forma aislada.
  </Dropdown>
</DropdownGroup>

## Módulos globales con *@Global()*

Por defecto, los providers de un módulo solo están disponibles para módulos que lo importan explícitamente, que es el comportamiento deseado para la mayoría de los módulos de feature porque mantiene las dependencias visibles y rastreables. Marcar un módulo con *@Global()* hace que sus exports estén disponibles en toda la aplicación sin requerir imports explícitos, pero esto debe reservarse exclusivamente para preocupaciones transversales como logging, analytics o services de configuración compartida que genuinamente necesitan ser accedidos desde todas las partes de la base de código.

<Callout tone="warning" title="Usa @Global() con moderación">
  Marcar demasiados módulos como globales elimina el encapsulamiento que hace valioso al sistema de módulos y dificulta rastrear dependencias a medida que el proyecto crece. Reserva *@Global()* exclusivamente para infraestructura que cada módulo genuinamente necesite , como logging o analytics , y mantén los services específicos de feature dentro de su propio módulo con exports explícitos.
</Callout>

```typescript filename="src/modules/analytics/analytics.module.ts"
import { Global, Module } from '@spraxium/common';
import { AnalyticsService } from './analytics.service';

@Global()
@Module({
  providers: [AnalyticsService],
  exports: [AnalyticsService],
})
export class AnalyticsModule {}
```

Usa módulos globales con moderación porque eluden las fronteras de encapsulamiento que hacen valioso al sistema de módulos en primer lugar. Los services específicos de feature siempre deben permanecer locales a su propio módulo con exports explícitos, y solo infraestructura verdaderamente universal debe ser global. Si te encuentras marcando más de dos o tres módulos como globales, generalmente indica que las fronteras de módulo necesitan ser rediseñadas en lugar de eludidas.

## Reglas para mantener módulos sostenibles

Mantener módulos saludables a lo largo del tiempo requiere seguir unas cuantas directrices prácticas que previenen que la complejidad crezca sigilosamente conforme el bot evoluciona. Mantén una frontera de negocio por carpeta de módulo, y exporta solo lo que otros módulos deben consumir mientras mantienes todos los detalles de implementación internos privados. Separa declaraciones de command y listener de los internals de service, y prefiere muchos módulos pequeños y enfocados compuestos por *AppModule* en lugar de unos pocos módulos grandes y compartidos que intentan hacer demasiado. A medida que el bot madura, es normal que un módulo existente se divida en dos más enfocados cuando un único dominio crece más allá de su alcance original, y este tipo de división es una señal de crecimiento saludable en lugar de una carga de refactorización.

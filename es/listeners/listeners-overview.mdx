## Que es un listener

Un listener es una clase que reacciona a eventos de Discord. Cuando algo pasa en Discord, como un usuario enviando un mensaje, un miembro entrando a un servidor, o el bot terminando su login, Discord.js dispara un evento. Una clase listener en Spraxium captura ese evento y corre tu código en respuesta. Sin listeners, tu bot no podría responder a nada que pase fuera de comandos explícitos.

La diferencia clave entre listeners y comandos es que los activa. Los comandos se activan por un usuario escribiendo un slash command o un prefix command. Los listeners se activan por eventos de Discord que pasan en background, como mensajes siendo enviados, miembros entrando o saliendo, reacciónes siendo agregadas, o el client quedando listo. Algunos de estos eventos involucran interacciónes de usuario que pueden recibir respuestas, y otros son eventos de background que pasan silenciosamente.

Los listeners en Spraxium no son callbacks crudos. Son clases completas manejadas por el contenedor de inyección de dependencias, registradas en módulos y conectadas al client de Discord automáticamente durante el bootstrap. Esto significa que tus clases listener pueden recibir servicios inyectados, participar en el pipeline de guards para eventos soportados y beneficiarse del mismo manejo de exceptions que usan los comandos.

## Un ejemplo básico de listener

El decorator _@Listener_ en la clase declara a cual evento de Discord pertenece este listener, el decorator _@Once_ en el método hace que dispare solo la primera vez que el evento ocurre, y _@On_ crea un binding persistente que dispara cada vez que el evento pasa. La primera pestaña muestra un listener que loguea cuando el bot termina de conectarse, y la segúnda un handler que reacciona a cada mensaje entrante.

<CodeTabs>
  <Tab label="Ready (Once)" language="typescript">
```typescript filename="src/events/listeners/ready.listener.ts"
import { Events, Listener, Once } from '@spraxium/common';
import { Logger } from '@spraxium/core';
import type { Client } from 'discord.js';

@Listener(Events.ClientReady)
export class ReadyListener {
private logger: Logger = new Logger(ReadyListener.name);

@Once()
onReady(client: Client<true>): void {
this.logger.info(`Client is ready as ${client.user.tag}`);
}
}

````
  </Tab>
  <Tab label="Mensajes (On)" language="typescript">
```typescript filename="src/events/listeners/message-create.listener.ts"
import { Events, Listener, On } from '@spraxium/common';
import type { Message } from 'discord.js';

@Listener(Events.MessageCreate)
export class MessageCreateListener {
  @On()
  async onMessage(message: Message): Promise<void> {
    if (message.author.bot) return;
    if (message.content === '!ping') {
      await message.reply('pong');
    }
  }
}
````

  </Tab>
</CodeTabs>

Nota que el parámetro del método recibe el payload del evento de Discord.js directamente. Para _ClientReady_, el payload es el objeto _Client_. Para _MessageCreate_, el payload es el objeto _Message_. Cada evento de Discord tiene su propio tipo de payload, y tu método recibe lo que Discord.js provee para ese evento específico.

## Registrando listeners en un módulo

Los listeners no se activan solo por existir en tu código fuente. Necesitas registrarlos en el array _listeners_ de un módulo para que el framework sepa sobre ellos y pueda conectarlos al client de Discord durante el bootstrap. Sin este registro, tu clase listener sera completamente ignorada.

```typescript filename="src/events/events.module.ts"
import { Module } from "@spraxium/common";
import { ReadyListener } from "./listeners/ready.listener";
import { MessageCreateListener } from "./listeners/message-create.listener";

@Module({
  listeners: [ReadyListener, MessageCreateListener],
})
export class EventsModule {}
```

El módulo en si debe ser importado por tu módulo raiz u otro módulo en la cadena de imports. El framework procesa módulos durante el bootstrap, resuelve sus arrays de listeners, instancia cada clase listener a través del contenedor de DI y luego los conecta al client de Discord. Si un listener depende de un servicio, ese servicio debe estar disponible en el alcance de providers del módulo o exportado desde un módulo importado, igual que cualquier otra dependencia inyectable.

## Como el bootstrap conecta listeners

El proceso de bootstrap maneja listeners en una secuencia específica que importa para entender cuando tu listener se activa. Primero, el framework lee el array _listeners_ de cada módulo. Segúndo, instancia cada clase listener por el contenedor de DI, resolviendo dependencias de constructor. Tercero, el _ListenerRegistry_ interno escanea cada instancia buscando métodos decorados con _@On_ o _@Once_ y los agrupa por nombre de evento. Cuarto, el _ListenerBinder_ toma esos grupos y llama al método _on_ u _once_ del client Discord.js para crear los bindings reales. Finalmente, cuando un evento de Discord dispara, el _ListenerInvoker_ llama a cada handler registrado para ese evento.

Esta estructura mantiene todas las clases listener en una carpeta dedicada y el archivo de módulo en la raiz de events. Conforme tu bot crece, puedes agregar mas archivos de listener sin cambiar la estructura de carpetas. Cada listener maneja un evento, y el archivo de módulo es el único lugar que declara cuales listeners están activos.

<Callout tone="info" title="Una clase por evento">
  Cada clase listener está vinculada a exactamente un evento de Discord a través
  del decorator Listener. Si necesitas escuchar múltiples eventos, crea clases
  listener separadas para cada uno. Esto mantiene cada clase enfocada y fácil de
  encontrar al debuguear.
</Callout>

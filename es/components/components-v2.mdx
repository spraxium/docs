## Qué son los contenedores V2

La API Components V2 de Discord reemplaza el modelo tradicional de layout de mensajes con un sistema de bloques componibles. En lugar de que un mensaje sea una combinación plana de contenido de texto, embeds y action rows, V2 te permite combinar bloques de texto, secciones, galerías de medios, separadores y action rows dentro de un único elemento contenedor. El contenedor se convierte en la unidad de layout principal del mensaje.

El sistema V2 de Spraxium envuelve esta API con la misma arquitectura decorator-handler utilizada en todo el paquete *@spraxium/components*. Declaras un schema de contenedor como una clase decorada, anotas cada elemento estructural con un decorator de propiedad y renderizas el contenedor completo a través de *V2Service* pasando datos de runtime. El resultado es un layout fuertemente tipado y componible que separa la estructura de los datos que fluyen por él en el momento del renderizado.

El paquete ofrece dos estilos de creación para V2. El patrón de schema basado en clases usa *@V2Container* con decorators de propiedad y es ideal para layouts reutilizables y orientados a datos. La API fluida es un builder imperativo que encadena llamadas en una instancia de *V2ContainerFluentBuilder* y es ideal para mensajes de un solo uso o ensamblados dinámicamente. Ambos estilos producen el mismo formato de wire y pueden combinarse en el mismo bot.

## Cuándo usar contenedores V2

Los contenedores V2 son apropiados siempre que el mensaje necesite un layout estructurado que un embed simple no pueda expresar. Los casos de uso comunes incluyen leaderboards con listas de entradas de longitud variable, paneles de dashboard de servidor con thumbnail de ícono y cuadrícula de estadísticas, perfiles con avatar junto a múltiples campos y galerías de imágenes integradas en respuestas de comandos. Cualquier mensaje que se beneficie de separación visual entre secciones, o que requeriría múltiples embeds para aproximarlo, es un buen candidato para un contenedor V2.

<Callout type="warning">
  Los contenedores V2 requieren la flag MessageFlags.IsComponentsV2 al enviar mensajes.
  Establecer esta flag significa que el mensaje no puede llevar embeds tradicionales ni una cadena de contenido simple junto a los contenedores, diseña los mensajes V2 como layouts independientes, no como mejoras de contenido existente.
</Callout>

## Configurar V2Service

*V2Service* es el servicio de renderizado y construcción para todo el trabajo con V2. Está disponible automáticamente mediante inyección una vez que *ComponentsModule* está registrado en tu grafo de módulos. No se necesita configuración adicional más allá de la configuración estándar del plugin *defineComponents*.

Inyecta *V2Service* en cualquier handler o servicio que necesite renderizar contenedores V2.

```typescript filename="src/handlers/my.handler.ts"
import { V2Service } from '@spraxium/components';

@SlashCommandHandler(MyCommand)
export class MyHandler {
  constructor(private readonly v2: V2Service) {}

  async handle(@Ctx() interaction: ChatInputCommandInteraction): Promise<void> {
    // ...
  }
}
```

## Schemas de contenedor basados en clases

El patrón basado en clases declara un contenedor como una clase anotada. Aplica *@V2Container* a la clase para marcarla como contenedor y configurar el color de acento y las opciones de spoiler. Aplica decorators de propiedad a cada campo para registrar componentes hijos en el orden de declaración. En el momento del renderizado, *V2Service.build* lee los metadatos, evalúa todas las funciones fábrica con tus datos de runtime y ensambla el contenedor.

Los nombres de las propiedades en la clase de schema no importan en runtime. Solo se leen los metadatos registrados por los decorators. Declara el tipo de propiedad como `never` para señalar que el campo existe únicamente como un slot de metadatos.

<CodeTabs>
  <Tab label="Schema">
```typescript filename="src/schemas/user-profile.container.ts"
import {
  V2Container, V2Separator, V2Section, V2Text, desc, } from '@spraxium/components';

export interface UserProfileData {
  username: string;
  userId: string;
  joinedAt: number;
  avatarUrl: string;
  bio: string;
}

@V2Container({ accentColor: 0x5865f2 })
export class UserProfileContainer {
  @V2Text((data: UserProfileData) => desc().h2(`👤 ${data.username}`))
  title!: never;

  @V2Separator()
  divider!: never;

  @V2Section({
    text: (data: UserProfileData) =>
      [
        `**ID:** \`${data.userId}\``, `**Se unió:** <t:${Math.floor(data.joinedAt / 1000)}:R>`, `**Bio:** ${data.bio}`, ].join('\n'), thumbnail: {
      url: (data: UserProfileData) => data.avatarUrl, description: 'Avatar del usuario', }, })
  stats!: never;

  @V2Separator({ divider: false })
  spacer!: never;

  @V2Text((data: UserProfileData) => desc().subtext(`Perfil de ${data.username}`))
  footer!: never;
}
```

import { Ctx, SlashCommandHandler } from '@spraxium/common';
import { V2Service } from '@spraxium/components';
import { MessageFlags, type ChatInputCommandInteraction } from 'discord.js';
import { ProfileCommand } from '../commands/profile.command';
import { UserProfileContainer, type UserProfileData } from '../schemas/user-profile.container';

@SlashCommandHandler(ProfileCommand)
export class ProfileHandler {
  constructor(private readonly v2: V2Service) {}

  async handle(@Ctx() interaction: ChatInputCommandInteraction): Promise<void> {
    const user = interaction.user;

    const data: UserProfileData = {
      username: user.username, userId: user.id, joinedAt: user.createdTimestamp, avatarUrl: user.displayAvatarURL({ size: 128 }), bio: 'Sin bio configurada.', };

    const payload = this.v2.buildReply(UserProfileContainer, data);

    await interaction.reply({
      ...payload, flags: payload.flags | MessageFlags.Ephemeral, });
  }
}

## La API fluida

La API fluida es un builder encadenable vinculado a *V2Service*. Llama a *v2.container()* para abrir un builder, encadena las llamadas a componentes hijos y llama a *.toReply()* al final para producir el payload listo para *interaction.reply()*. Cada método de encadenamiento devuelve el mismo builder, por lo que todas las operaciones pueden ser inline.

La API fluida no admite decorators de propiedad ni condiciones *@V2When*, ya que no hay metadatos a nivel de clase para anotar. Usa el patrón basado en clases cuando necesites renderizado condicional o cuando quieras reutilizar el mismo layout en múltiples handlers.

```typescript filename="src/handlers/greet.handler.ts"
import { Ctx, SlashCommandHandler } from '@spraxium/common';
import { V2Service, desc } from '@spraxium/components';
import { MessageFlags, type ChatInputCommandInteraction } from 'discord.js';
import { GreetCommand } from '../commands/greet.command';

@SlashCommandHandler(GreetCommand)
export class GreetHandler {
  constructor(private readonly v2: V2Service) {}

  async handle(@Ctx() interaction: ChatInputCommandInteraction): Promise<void> {
    const user = interaction.user;

    const payload = this.v2
      .container({ accentColor: 0x57f287 })
      .text(desc().h2(`👋 ¡Hola, ${user.username}!`))
      .sep()
      .section(
        [
          `**ID:** \`${user.id}\``, `**Antigüedad de la cuenta:** <t:${Math.floor(user.createdTimestamp / 1000)}:R>`, ].join('\n'), this.v2.thumbnail(user.displayAvatarURL({ size: 128 }), {
          description: `Avatar de ${user.username}`, }), )
      .sep({ divider: false })
      .text(desc().subtext('Desarrollado con Spraxium V2'))
      .toReply();

    await interaction.reply({
      ...payload, flags: payload.flags | MessageFlags.Ephemeral, });
  }
}
```

## Hijos dinámicos con @V2Dynamic

*@V2Dynamic* es la solución cuando el número de componentes hijos depende de datos de runtime. En lugar de una configuración estática, proporcionas una función fábrica que recibe el objeto de datos y retorna un array de specs de hijos creados con las funciones auxiliares *v2text*, *v2sep*, *v2section*, *v2gallery* y *v2row*. El motor recorre este array y renderiza cada spec en secuencia.

Un patrón común es generar una o más líneas de texto por entrada de datos e insertar separadores entre ellas, deteniéndose un separador antes para que la última entrada no quede seguida de un divisor.

```typescript filename="src/schemas/leaderboard.container.ts"
import {
  V2Container, V2Dynamic, V2Separator, V2Text, desc, v2sep, v2text, } from '@spraxium/components';

export interface LeaderboardEntry {
  username: string;
  score: number;
  rank: number;
}

export interface LeaderboardData {
  title: string;
  entries: Array<LeaderboardEntry>;
}

const MEDALS = ['🥇', '🥈', '🥉'];

@V2Container({ accentColor: 0xf1c40f })
export class LeaderboardContainer {
  @V2Text((data: LeaderboardData) => desc().h2(`🏆 ${data.title}`))
  title!: never;

  @V2Separator()
  divider!: never;

  @V2Dynamic((data: LeaderboardData) => {
    if (data.entries.length === 0) {
      return [v2text(desc().subtext('Sin entradas aún.'))];
    }

    return data.entries.flatMap((entry, index) => {
      const medal = MEDALS[index] ?? `**${entry.rank}.**`;
      return [
        v2text(`${medal} **${entry.username}**, ${entry.score.toLocaleString()} pts`), ...(index < data.entries.length - 1 ? [v2sep({ divider: false })] : []), ];
    });
  })
  entries!: never;

  @V2Separator({ divider: false })
  footer!: never;

  @V2Text((data: LeaderboardData) => desc().subtext(`${data.entries.length} jugadores clasificados`))
  summary!: never;
}
```

El handler pasa la lista de entradas como datos y llama a *buildReply* sin necesidad de saber cuántas filas se generarán.

```typescript filename="src/handlers/leaderboard.handler.ts"
@SlashCommandHandler(LeaderboardCommand)
export class LeaderboardHandler {
  constructor(private readonly v2: V2Service) {}

  async handle(@Ctx() interaction: ChatInputCommandInteraction): Promise<void> {
    const entries: Array<LeaderboardEntry> = await fetchTopPlayers(10);

    const payload = this.v2.buildReply(LeaderboardContainer, {
      title: 'Top Jugadores de la Semana', entries, });

    await interaction.reply(payload);
  }
}
```

## Renderizado condicional con @V2When

*@V2When* incluye condicionalmente un hijo según los datos de runtime. Se aplica como un decorator adicional *debajo* del decorator del hijo, lo que significa que se ejecuta al final y agrega un predicado de guardia a la definición de hijo existente. Si el predicado retorna `false` para los datos dados, el hijo se excluye del contenedor completamente, no se renderiza como vacío.

*@V2When* es apilable: múltiples hijos en el mismo contenedor pueden tener condiciones independientes. El orden asignado en el momento del registro se preserva independientemente de qué hijos sean filtrados.

```typescript filename="src/schemas/member-card.container.ts"
import {
  V2Container, V2Separator, V2Text, V2When, desc, } from '@spraxium/components';

export interface MemberCardData {
  username: string;
  boostLevel: number;
  isAdmin: boolean;
  bio: string;
}

@V2Container({ accentColor: 0x5865f2 })
export class MemberCardContainer {
  @V2Text((data: MemberCardData) => desc().h2(data.username))
  title!: never;

  @V2Separator()
  divider!: never;

  @V2When((data: MemberCardData) => data.isAdmin)
  @V2Text('⚙️ **Administrador del Servidor**')
  adminBadge!: never;

  @V2When((data: MemberCardData) => data.boostLevel > 0)
  @V2Text((data: MemberCardData) => `⭐ **Server Booster**, Nivel ${data.boostLevel}`)
  boostBadge!: never;

  @V2Text((data: MemberCardData) => data.bio)
  bio!: never;
}
```

Cuando `isAdmin` es `false`, la fila del emblema de administrador no se renderiza. Los hijos restantes se cierran naturalmente sin ningún espacio vacío.

## Incrustar action rows con @V2Row

*@V2Row* coloca una action row de botones o select menu directamente dentro de un contenedor V2. Pasa las clases de tus componentes en el array `components` de la misma forma que lo harías con *ButtonService* o *SelectService*. El servicio detecta automáticamente si la primera clase es un select menu y enruta al builder correcto.

  Button, LinkButton, V2Container, V2Row, V2Separator, V2Text, desc, } from '@spraxium/components';

@Button({ customId: 'panel_confirm', label: 'Confirm', style: 'success', emoji: '✅' })
export class ConfirmButton {}

@Button({ customId: 'panel_cancel', label: 'Cancel', style: 'danger', emoji: '✖' })
export class CancelButton {}

@V2Container({ accentColor: 0xed4245 })
export class ConfirmPanelContainer {
  @V2Text((data: { action: string }) => desc().h2(`Confirmar: ${data.action}`))
  title!: never;

  @V2Text('Esta acción no se puede deshacer. ¿Estás seguro?')
  body!: never;

  @V2Separator({ divider: false })
  spacer!: never;

  @V2Row({ components: [ConfirmButton, CancelButton] })

## El DescriptionBuilder

La función auxiliar *desc()* devuelve un *DescriptionBuilder* que encadena helpers de Markdown con formato de Discord. Está disponible en los patrones basados en clases y fluido en cualquier lugar donde se acepte un valor de contenido de texto. El builder es un acumulador de string liviano; llamar a *.toString()* retorna el string final, que el motor V2 llama automáticamente antes de pasar el contenido a los builders de discord.js.

## Construir el payload de reply

Tanto *buildReply* como el método fluido *.toReply()* retornan un objeto `{ components, flags }`. Hacer spread de este objeto en *interaction.reply()* establece `IsComponentsV2` automáticamente. Puedes combinar flags adicionales usando OR a nivel de bits.

```typescript filename="enviando un reply V2"
const payload = this.v2.buildReply(MyContainer, data);

await interaction.reply({
  ...payload, flags: payload.flags | MessageFlags.Ephemeral, });
```

Si necesitas actualizar un mensaje V2 existente, usa *interaction.editReply()* o pasa el payload a *message.edit()* directamente. La flag es necesaria tanto en el envío inicial como en las ediciones posteriores, ya que Discord identifica un mensaje como V2 por la presencia de esa flag.

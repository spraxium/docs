## Drivers de bloqueo

Cuando ejecutas múltiples instancias de tu bot (por ejemplo, detrás de un balanceador de carga o en shards separados), cada instancia intentará ejecutar cada job planificado. Sin un mecanismo de bloqueo, el mismo job se ejecutaría múltiples veces simultáneamente. Los drivers de schedule resuelven esto adquiriendo un bloqueo temporal antes de cada ejecución de job, solo la instancia que adquiere el bloqueo ejecuta el job.

El framework incluye dos drivers. El _MemoryDriver_ es el predeterminado y simplemente adquiere siempre el bloqueo, lo que significa que no hay protección contra ejecuciones duplicadas. Es adecuado para despliegues de instancia única donde la ejecución duplicada no es posible. El _RedisScheduleDriver_ usa Redis SET NX con TTL para bloqueo distribuido y scripts Lua para liberación atómica.

## Configuración de drivers

El driver se configura en el archivo de configuración del schedule. El _MemoryDriver_ no necesita parámetros, funciona inmediatamente. El _RedisScheduleDriver_ necesita las opciones de conexión a Redis que proporcionas desde tu capa de infraestructura. Ambos drivers implementan la misma interfaz, así que cambiar de uno a otro solo requiere modificar una línea de configuración.

<CodeTabs>
  <Tab label="Memory (predeterminado)" language="typescript">
```typescript filename="config/schedule.config.ts"
import { defineSchedule } from '@spraxium/schedule';

export default defineSchedule({
  timezone: 'UTC',
});
```
  </Tab>
  <Tab label="Redis (producción)" language="typescript">
```typescript filename="config/schedule.config.ts"
import { defineSchedule, RedisScheduleDriver } from '@spraxium/schedule';

export default defineSchedule({
  timezone: 'UTC', defaultLockTtlMs: 30_000, driver: new RedisScheduleDriver({
    host: process.env.REDIS_HOST ?? 'localhost', port: Number(process.env.REDIS_PORT ?? 6379), }),
});
```
  </Tab>
</CodeTabs>

## API de ScheduleService

El _ScheduleService_ es la fachada pública para el control de jobs en runtime. Puedes inyectarlo en cualquier servicio o handler para pausar, reanudar, destruir o ejecutar manualmente jobs planificados. Todos los métodos usan el nombre del job como identificador, que corresponde al campo _name_ en las opciones del decorator o al nombre del método si no se especificó un nombre.

## Comando de administración de ejemplo

Un caso de uso común es exponer el control de jobs a través de un prefix command o slash command para que los administradores del bot puedan gestionar jobs planificados sin reiniciar el bot. El siguiente ejemplo muestra un handler de prefix command que pausa, reanuda o dispara un job por nombre.

```typescript filename="src/modules/admin/handlers/schedule-admin.handler.ts"
import { Ctx, PrefixArg, PrefixCommandHandler } from "@spraxium/common";
import { ScheduleService } from "@spraxium/schedule";
import type { Message } from "discord.js";
import { ScheduleAdminCommand } from "../commands/schedule-admin.command";

@PrefixCommandHandler(ScheduleAdminCommand)
export class ScheduleAdminHandler {
  constructor(private readonly schedule: ScheduleService) {}

  async handle(
    @Ctx() message: Message,
    @PrefixArg("action") action: string,
    @PrefixArg("jobName") jobName: string,
  ): Promise<void> {
    switch (action) {
      case "pause":
        this.schedule.pause(jobName);
        await message.reply(`Job "${jobName}" paused.`);
        break;
      case "resume":
        this.schedule.resume(jobName);
        await message.reply(`Job "${jobName}" resumed.`);
        break;
      case "run":
        await this.schedule.runNow(jobName);
        await message.reply(`Job "${jobName}" executed.`);
        break;
      default:
        await message.reply("Valid actions: pause, resume, run");
    }
  }
}
```

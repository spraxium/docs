## ScheduleConfig

El objeto _ScheduleConfig_ controla el comportamiento global del módulo de schedule. Se pasa a la fábrica _defineSchedule_ cuando registras el módulo en tu configuración de Spraxium. Todos los campos son opcionales con valores predeterminados razonables, así que la configuración mínima es una llamada sin argumentos.

## Configuración dividida

Para proyectos grandes, es útil extraer la configuración del schedule a un archivo separado. Esto mantiene el archivo principal de configuración limpio y te permite cambiar la configuración del schedule según el entorno sin modificar la configuración raíz. La separación también facilita compartir la configuración del schedule entre múltiples bots si es necesario.

<CodeTabs>
  <Tab label="Archivo de configuración" language="typescript">
```typescript filename="config/schedule.config.ts"
import { defineSchedule, MemoryDriver, RedisScheduleDriver } from '@spraxium/schedule';

const isProduction = process.env.NODE_ENV === 'production';

export default defineSchedule({
  driver: isProduction
    ? new RedisScheduleDriver({
        host: process.env.REDIS_HOST ?? 'localhost',
        port: Number(process.env.REDIS_PORT ?? 6379),
      })
    : new MemoryDriver(),
  timezone: 'America/Sao_Paulo',
  defaultLockTtlMs: 30_000,
  disableAll: process.env.DISABLE_JOBS === 'true',
});
```
  </Tab>
  <Tab label="Configuración principal" language="typescript">
```typescript filename="spraxium.config.ts"
import { defineConfig } from '@spraxium/core';
import schedule from './config/schedule.config';

export default defineConfig({
  plugins: [
    schedule, ],
});
```
  </Tab>
</CodeTabs>

## Estructura de carpetas

Los archivos de configuración de schedule siguen el mismo patrón que otros archivos de configuración del framework. El archivo de configuración del schedule vive en la carpeta _config_ junto con los demás archivos de configuración. Los servicios de jobs viven dentro de sus respectivas carpetas de módulos.

## Patrones combinados de jobs

Un único servicio puede combinar múltiples tipos de jobs cuando estos están lógicamente relacionados. Por ejemplo, un servicio de moderación podría tener un job cron para limpieza diaria, un job de intervalo para monitoreo de spam y un job post-online para cargar la configuración de moderación desde la base de datos. Agrupar jobs relacionados en un solo servicio mantiene la cohesión y facilita el testing.

Los nombres de job deben ser únicos dentro de toda la aplicación, no solo dentro del servicio. El framework lanza un error en tiempo de inicio si detecta nombres de job duplicados. Usa un prefijo específico del módulo en los nombres de tus jobs para evitar colisiones, como `moderation:cleanup` o `analytics:daily-report`.

Cuando un servicio tiene múltiples jobs, puedes controlarlos independientemente a través del _ScheduleService_. Pausar un job no afecta a los demás jobs en el mismo servicio. Esta granularidad te permite deshabilitar jobs específicos en runtime sin afectar a otros jobs.

```typescript filename="src/modules/moderation/services/moderation-jobs.service.ts"
import { Injectable } from "@spraxium/common";
import {
  AfterOnline,
  Cron,
  CronExpression,
  Interval,
  IntervalExpression,
} from "@spraxium/schedule";

@Injectable()
export class ModerationJobsService {
  @Cron(CronExpression.EVERY_DAY_AT_3AM, {
    name: "moderation:cleanup",
    timezone: "America/New_York",
  })
  async cleanupExpiredBans(): Promise<void> {
    // Eliminar bans expirados de la base de datos
  }

  @Interval(IntervalExpression.EVERY_10_SECONDS, {
    name: "moderation:spam-check",
  })
  async checkSpamQueues(): Promise<void> {
    // Procesar mensajes en cola para detecci\u00f3n de spam
  }

  @AfterOnline(5000, {
    name: "moderation:load-config",
  })
  async loadModerationConfig(): Promise<void> {
    // Cargar configuraci\u00f3n de moderaci\u00f3n al estar online
  }
}
```

## Flags de features

La opción _disabled_ en los decorators de jobs y el campo _disableAll_ en la configuración proporcionan feature flags integrados para el sistema de schedule. Los feature flags te permiten deshabilitar jobs en entornos específicos sin eliminar el código, lo cual es útil durante pruebas, mantenimiento o despliegue gradual de nuevos jobs.

Puedes vincular las opciones _disabled_ a variables de entorno para controlar qué jobs se ejecutan en cada despliegue. Esto es especialmente útil cuando quieres deshabilitar jobs intensivos en recursos en entornos de staging o deshabilitar todos los jobs temporalmente durante una ventana de mantenimiento de base de datos.

El campo _disableAll_ en la configuración actúa como un interruptor maestro que impide que cualquier job sea planificado al inicio. Esto es diferente a pausar jobs individualmente con _ScheduleService.pause_, porque _disableAll_ previene completamente que los jobs se creen en el registro.

```typescript filename="src/modules/analytics/services/analytics-jobs.service.ts"
import { Injectable } from "@spraxium/common";
import { Cron, CronExpression } from "@spraxium/schedule";

@Injectable()
export class AnalyticsJobsService {
  @Cron(CronExpression.EVERY_HOUR, {
    name: "analytics:hourly",
    disabled: process.env.DISABLE_ANALYTICS === "true",
  })
  async hourlyReport(): Promise<void> {
    // Generar reporte por hora
  }
}
```

## Registro del módulo

La configuración final registra el módulo de schedule en el array de plugins del config raíz. El orden de los plugins no afecta a la ejecución de jobs, los jobs siempre se planifican después de que todos los módulos hayan completado su inicialización. Esto garantiza que la inyección de dependencias esté completamente resuelta antes de que cualquier job se ejecute.

```typescript filename="spraxium.config.ts"
import { defineConfig } from "@spraxium/core";
import schedule from "./config/schedule.config";

export default defineConfig({
  plugins: [schedule],
});
```

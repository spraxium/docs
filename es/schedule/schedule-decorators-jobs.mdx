## Jobs cron e intervalo

Los decorators _@Cron_ y _@Interval_ crean jobs que se ejecutan repetidamente. Los jobs cron se ejecutan en horarios específicos definidos por una expresión cron, mientras que los jobs de intervalo se ejecutan cada N milisegundos continuamente. Ambos soportan las opciones _name_ y _disabled_ para control de identidad y estado. Los jobs cron adicionalmente aceptan _timezone_ y _runOnInit_.

El decorator _@Cron_ acepta una expresión de 5 o 6 campos como primer argumento. Puedes escribir la expresión directamente como string o usar el enum helper _CronExpression_ que proporciona más de 45 presets con nombre. La opción _runOnInit_ dispara el job inmediatamente al inicio del bot además del schedule cron normal, lo cual es útil para jobs que deben capturar datos perdidos durante el tiempo de inactividad.

El decorator _@Interval_ acepta un número de milisegundos como primer argumento o un valor del enum _IntervalExpression_. Los intervalos comienzan inmediatamente después del inicio del bot y se repiten indefinidamente hasta que el bot se detiene o el job se destruye manualmente a través del _ScheduleService_.

@Injectable()
export class AnalyticsJobsService {
@Cron(CronExpression.EVERY_DAY_AT_MIDNIGHT, {
name: 'daily-stats', timezone: 'America/New_York', })

// Agregar estad\u00edsticas del d\u00eda y enviar al canal de anal\u00edticas
}
}

## Jobs timeout y post-online

Los decorators _@Timeout_ y _@AfterOnline_ crean jobs que se ejecutan una única vez. Los timeouts se disparan después de un delay desde el inicio del bot, mientras que los jobs post-online esperan a que el bot esté completamente conectado a Discord antes de contar el delay. Esta distinción es importante para tareas que necesitan del cliente de Discord completamente inicializado.

El decorator _@Timeout_ es útil para lógica de inicialización diferida que no necesita la conexión de Discord. Por ejemplo, podrías usarlo para pre-calentar cachés, programar tareas de mantenimiento únicas o registrar que el bot inició correctamente. El delay se cuenta desde el momento en que el módulo de schedule completa su fase de arranque.

El decorator _@AfterOnline_ está diseñado específicamente para tareas que dependen de la conexión a Discord. La fase de ready garantiza que el cliente tiene acceso al caché de servidores, caché de canales y datos de usuario. Úsalo para tareas de sincronización, actualizaciones de estado o cualquier trabajo que necesite consultar la API de Discord.

<CodeTabs>
  <Tab label="Job timeout" language="typescript">
```typescript filename="src/modules/cache/services/warmup-jobs.service.ts"
import { Injectable } from '@spraxium/common';
import { Timeout, TimeoutExpression } from '@spraxium/schedule';

@Injectable()
export class WarmupJobsService {
@Timeout(TimeoutExpression.AFTER_5_SECONDS, {
name: 'cache-warmup', })
async warmupCaches(): Promise<void> {
// Pre-cargar datos usados frecuentemente en cach\u00e9
}
}

````

  }

## Opciones de decorator

Todos los decorators de schedule aceptan un objeto de opciones como segundo argumento. Las opciones disponibles varían según el tipo de decorator, pero _name_ y _disabled_ están disponibles en los cuatro. La opción _name_ es especialmente importante porque el _ScheduleService_ usa el nombre del job para control en runtime.

## Helpers de enum de expresiones

El paquete schedule exporta tres enums que proporcionan expresiones de schedule con nombre como alternativa a valores codificados. Usar estos enums hace que las definiciones de jobs sean más legibles y previene errores tipográficos en expresiones cron. Los enums están completamente tipados, así que tu editor los autocompleta y valida.

El enum _CronExpression_ proporciona más de 45 presets que cubren patrones comunes como _EVERY_HOUR_, _EVERY_DAY_AT_MIDNIGHT_, _EVERY_MONDAY_AT_MIDNIGHT_ y _EVERY_WEEK_. El enum _IntervalExpression_ proporciona 26 presets desde _EVERY_SECOND_ hasta _EVERY_24_HOURS_. El enum _TimeoutExpression_ proporciona 20 presets desde _AFTER_1_SECOND_ hasta _AFTER_24_HOURS_.

Los enums se resuelven a valores primitivos en tiempo de compilación, así que no hay sobrecarga en runtime. Puedes mezclar valores de enum con valores literales en la misma base de código sin conflictos. Usa los enums cuando los presets nombrados coincidan con tu intención, y los valores literales cuando necesites expresiones personalizadas como `0 */6 * * 1-5` para cada 6 horas entre semana.

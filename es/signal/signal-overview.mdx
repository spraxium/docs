## Por qué existe Signal en arquitecturas de bot

_Signal_ fue diseñado para equipos que operan bots en operaciones reales donde múltiples servicios externos necesitan informar al bot sobre cambios de dominio sin crear dependencias rígidas de runtime. En lugar de conectar cada API, worker o panel administrativo directamente a los internos del bot, publicas eventos estandarizados a través de webhooks de Discord y dejas que el bot los consuma a través de una ruta segura única. Esto mantiene limpios los límites de responsabilidad porque los sistemas externos se enfocan en producir eventos de negocio mientras el bot se enfoca en reaccionar a ellos.

El protocolo usa la propia infraestructura de webhooks de Discord como capa de transporte. Los servicios externos envían envelopes JSON firmados a un canal privado de Discord, y el módulo Signal del bot escucha mensajes nuevos en ese canal. Este diseño evita exponer APIs de servicio internas, evita configurar servidores de transporte personalizados, y aprovecha la confiabilidad y garantías de entrega existentes de Discord. La separación entre productor y consumidor es la razón core por la que Signal escala mejor que handlers de webhook ad-hoc dispersos en módulos de funcionalidades.

## Resumen de arquitectura

El ecosistema Signal se divide en un receptor (el bot) y uno o más emisores (servicios externos). El emisor construye un envelope firmado, lo publica como mensaje webhook en un canal privado de Discord, y el listener del bot lo recoge. El receptor valida el envelope a través de un estricto pipeline de diez reglas antes de despachar el evento a un handler registrado.

Esta arquitectura mantiene limpio el proceso del bot. Nunca expone puertos HTTP entrantes para la ingesta de eventos. En su lugar, aprovecha el evento _messageCreate_ existente de Discord, al que el bot ya está suscrito. El módulo Signal simplemente agrega un filtro que identifica mensajes webhook de fuentes aprobadas y los enruta a través del pipeline de validación.

## El contrato del envelope

Cada mensaje Signal sigue una forma única de envelope. Esta estructura estandarizada asegura que la validación de transporte ocurra antes de que cualquier código de negocio se ejecute. El envelope es un objeto JSON con siete campos, cada uno sirviendo un rol específico en el pipeline de validación y despacho.

## Modelo de seguridad HMAC

La capa HMAC es la frontera de seguridad que le dice al bot si un envelope fue producido por un emisor autorizado y si el contenido permaneció intacto en tránsito. La firma se computa concatenando los campos _nonce_, _sentAt_ y _payload_ en un string único (`${nonce}.${sentAt}.${JSON.stringify(payload)}`), luego hasheándolo con HMAC-SHA256 usando una clave secreta compartida.

En el lado del bot, la misma computación se realiza y el resultado se compara contra la firma recibida usando una función de comparación timing-safe. Si los valores no coinciden, el envelope es rechazado antes de que cualquier lógica de negocio se ejecute. Esto protege contra tres categorías de ataques: manipulación del payload en tránsito, replay con datos modificados, y envelopes falsificados por servicios no autorizados.

En términos prácticos, HMAC permite a tu equipo confiar en la automatización proveniente de dashboards externos, sistemas de billing o backoffices de moderación sin exponer APIs de servicio internas al proceso del bot. El secreto compartido debe ser el mismo tanto en el emisor como en el receptor, y nunca debe committearse al control de versiones.

<Callout
  tone="warning"
  title="El secreto compartido debe coincidir exactamente"
>
  El valor de <code>hmacSecret</code> en <code>defineSignal</code> en el lado
  del bot debe ser idéntico al valor de <code>hmacSecret</code> en el
  constructor de
  <code>SignalClient</code>. Si divergen incluso por un carácter, cada envelope
  será rechazado como no confiable.
</Callout>

## Pipeline de validación de diez reglas

Antes de que cualquier handler de dominio se ejecute, cada mensaje entrante pasa por un estricto pipeline de validación de diez reglas implementado en _SignalValidator_. Cada regla se evalúa secuencialmente, y el primer fallo detiene el pipeline inmediatamente. Este diseño de fallo rápido asegura que los mensajes inválidos consuman tiempo mínimo de procesamiento.

## Pipeline de procesamiento completo

Después de que el validador aprueba un envelope, el _SignalProcessor_ continúa con tres pasos adicionales antes de que el handler de dominio se ejecute. Comprender el pipeline completo te ayuda a depurar problemas de entrega y razonar sobre el orden de operaciones durante el análisis de incidentes.

<Steps>
  <Step title="Validación de diez reglas">
    El *SignalValidator* verifica cada mensaje entrante contra diez reglas
    secuenciales. El primer fallo rechaza el mensaje y registra una entrada de
    diagnóstico cuando el modo debug está habilitado. Solo los mensajes que
    pasan las diez reglas proceden a la siguiente etapa.
  </Step>
  <Step title="Deduplicación de nonce">
    El *NonceCache* verifica si este nonce ya fue visto dentro de los últimos 60
    segundos. Si lo fue, el mensaje se trata como replay y se descarta
    silenciosamente. Si es nuevo, el nonce se registra con un timestamp. Las
    entradas expiradas se expulsan automáticamente en la siguiente verificación.
  </Step>
  <Step title="Resolución de handler y enrutamiento">
    El *SignalRouter* busca el handler registrado para este nombre de evento. Si
    no existe handler, el mensaje se registra como no enrutable y se descarta.
    Esto no debería ocurrir porque la regla R6 ya verificó que el evento estaba
    registrado, pero la verificación del router agrega defensa en profundidad.
  </Step>
  <Step title="Validación Zod del payload">
    Si el handler resuelto tiene un esquema Zod adjunto vía las opciones de
    *@OnSignal*, el procesador ejecuta *safeParse* sobre el payload. Si la
    validación falla, el mensaje se rechaza con un log de diagnóstico. Si no hay
    esquema configurado, el payload crudo se pasa sin cambios.
  </Step>
  <Step title="Eliminación opcional del mensaje">
    Si *deleteAfterProcessing* está habilitado en la config de Signal, el
    procesador intenta eliminar el mensaje webhook del canal de Discord. Los
    fallos de eliminación se registran como advertencias pero no bloquean la
    ejecución del handler.
  </Step>
  <Step title="Ejecución del handler">
    El procesador invoca el handler resuelto con el payload validado y el
    envelope completo. Los errores lanzados por el handler se capturan,
    registran, y no crashean el pipeline de Signal. Otros eventos continúan
    procesándose normalmente.
  </Step>
</Steps>

## Cuándo evitar Signal

Si el productor y consumidor viven en el mismo grafo de módulos y no necesitas fronteras asíncronas, las llamadas directas de servicio son más simples de razonar y más fáciles de trazar. Signal debería adoptarse cuando sus garantías operativas, modelo de seguridad y beneficios de desacoplamiento son genuinamente útiles, no como valor por defecto para cada interacción interna. Para loops de feedback síncronos orientados al usuario donde el llamador necesita una respuesta inmediata, los endpoints HTTP o llamadas directas a la API de Discord siguen siendo la mejor opción.

## Responsabilidades del lado del bot

En el lado receptor, *Signal* es responsable de proteger tu runtime de bot antes de que cualquier mutación de dominio ocurra. El *SignalRegistry* descubre clases listener durante el arranque, el *SignalValidator* impone diez reglas de transporte, el *NonceCache* previene ataques de replay, y el *SignalProcessor* orquesta todo el pipeline. Tus handlers solo se ejecutan después de que todas estas capas hayan aprobado el envelope entrante.

Esta separación es el beneficio arquitectónico clave. Tus handlers de dominio se mantienen enfocados en la intención de negocio, aplicando cambios de roles, actualizando configuraciones, publicando anuncios, mientras la capa de protocolo posee las responsabilidades de confianza, formato y enrutamiento. Ningún handler necesita verificar si el webhook está autorizado o si la firma es válida porque esas verificaciones ya ocurrieron upstream.

## Configuración del plugin

El módulo Signal del lado del bot lee su configuración desde una entrada de plugin producida por la factory *defineSignal*. Esta factory se exporta desde `@spraxium/signal` y produce una entrada *SpraxiumPlugin* tipada que debe agregarse al array *plugins* en `spraxium.config.ts`. La configuración define las fronteras de confianza y controles operativos para el pipeline receptor.

### Patrón de config separada

El enfoque recomendado es extraer la config de Signal a un archivo dedicado dentro de `config/` e importar la constante en `spraxium.config.ts`. Esto mantiene la política operativa explícita y permite a los equipos de seguridad auditar las fronteras de Signal sin parsear configuración no relacionada de la aplicación.

<CodeTabs>
  <Tab label="config/signal.config.ts" language="typescript">
```typescript filename="config/signal.config.ts"
import { defineSignal } from '@spraxium/signal';

export const signalPlugin = defineSignal({
  channelId: process.env.SIGNAL_CHANNEL_ID!, allowedWebhookIds: [process.env.SIGNAL_WEBHOOK_ID!], hmacSecret: process.env.SIGNAL_HMAC_SECRET!, deleteAfterProcessing: true, debug: false, });
```
  </Tab>
  <Tab label="spraxium.config.ts" language="typescript">
```typescript filename="spraxium.config.ts"
import { defineConfig } from '@spraxium/core';
import { signalPlugin } from './config/signal.config';

export default defineConfig(() => ({
  plugins: [signalPlugin], }));
```

## El decorador @SignalListener

El decorador *@SignalListener()* marca una clase como contenedor de handlers de eventos Signal. Cumple el mismo rol que *@Listener()* cumple para eventos de Discord y *@PrefixCommand()* cumple para prefix commands. Durante el arranque, el *SignalRegistry* escanea instancias de providers e identifica clases decoradas con *@SignalListener()*, luego inspecciona cada método buscando metadata de *@OnSignal()*.

Internamente, *@SignalListener()* aplica *@Injectable()* automáticamente, así que la clase participa en inyección de dependencias sin necesitar un decorador separado. La clase debe estar listada en el array *providers* de un módulo que importe *SignalModule*.

```typescript filename="src/modules/signal/listeners/guild-config.signal.ts"
import { SignalListener, OnSignal } from '@spraxium/signal';
import type { SignalEnvelope } from '@spraxium/signal';

@SignalListener()
export class GuildConfigSignal {
  @OnSignal('config.update')
  async onConfigUpdate(payload: unknown, envelope: SignalEnvelope): Promise<void> {
    // Lógica de dominio anclada a envelope.guildId
  }
}
```

## El decorador @OnSignal

El decorador *@OnSignal()* registra un método como handler para un evento Signal específico. El primer argumento es el nombre del evento, que debe coincidir exactamente con el nombre de evento usado por el emisor. El segundo argumento opcional es un objeto de opciones que puede incluir un esquema Zod para validación de payload.

Cuando el pipeline de Signal recibe un envelope con un nombre de evento coincidente, el método decorado se invoca con dos argumentos: el payload (opcionalmente validado por Zod) y el *SignalEnvelope* completo. El envelope te da acceso a *guildId*, *nonce*, *sentAt* y otros metadatos para scoping y auditoría.

```typescript filename="src/modules/signal/listeners/guild-config.signal.ts"
import { SignalListener, OnSignal } from '@spraxium/signal';
import type { SignalEnvelope } from '@spraxium/signal';
import { z } from 'zod';

const ConfigUpdateSchema = z.object({
  prefix: z.string().min(1).max(5), language: z.enum(['en', 'pt', 'es']).optional(), });

@SignalListener()
export class GuildConfigSignal {
  @OnSignal('config.update', { schema: ConfigUpdateSchema })
  async onConfigUpdate(
    payload: z.infer<typeof ConfigUpdateSchema>, envelope: SignalEnvelope, ): Promise<void> {
    // payload está validado y tipado, envelope proporciona contexto del guild
  }
}
```

## Nombre de evento como contrato estable

El primer argumento en *@OnSignal('config.update')* es la identidad del evento usada por el enrutamiento de Signal. Es la clave que mapea envelopes entrantes al listener correcto, es un contrato productor-consumidor entre el código emisor y los listeners del bot, y es una referencia operativa para logs, dashboards y análisis de incidentes.

Trata los nombres de evento como APIs de dominio estables. Si la semántica del payload debe cambiar de forma no retrocompatible, prefiere introducir un nuevo nombre de evento como `config.update.v2` en lugar de reutilizar silenciosamente el mismo. Renombrar un evento sin coordinar productores y listeners causa que los mensajes se ignoren silenciosamente o se despachen al handler incorrecto.

<Callout tone="warning" title="Renombrar nombres de evento es un cambio disruptivo">
  Si renombras un evento sin actualizar tanto el emisor como el listener del bot, los mensajes
  serán ignorados silenciosamente porque el router no puede encontrar un handler coincidente
  para el nombre anterior.
</Callout>

## Conexión de módulo y listener

La estructura de módulo para Signal sigue el patrón estándar de Spraxium. Crea un módulo de funcionalidades que importe *SignalModule* y registre tus clases listener en el array *providers*. El *SignalModule* es un módulo global que proporciona los servicios core (*SignalValidator*, *SignalRouter*, *SignalProcessor*, *SignalRegistry*, *NonceCache*), así que solo necesitas importarlo una vez en cualquier módulo que contenga listeners.

<CodeTabs>
  <Tab label="Módulo" language="typescript">
```typescript filename="src/modules/signal/signal.module.ts"
import { Module } from '@spraxium/common';
import { SignalModule } from '@spraxium/signal';
import { GuildConfigSignal } from './listeners/guild-config.signal';
import { BillingSignal } from './listeners/billing.signal';

@Module({
  imports: [SignalModule], providers: [GuildConfigSignal, BillingSignal], })
export class AppSignalModule {}
```
  </Tab>
  <Tab label="Listener" language="typescript">
```typescript filename="src/modules/signal/listeners/billing.signal.ts"
import { SignalListener, OnSignal } from '@spraxium/signal';
import type { SignalEnvelope } from '@spraxium/signal';
import { z } from 'zod';

const BillingSchema = z.object({
  action: z.enum(['activated', 'cancelled', 'upgraded']), planId: z.string(), userId: z.string(), });

@SignalListener()
export class BillingSignal {
  @OnSignal('billing.status.changed', { schema: BillingSchema })
  async onBillingChange(
    payload: z.infer<typeof BillingSchema>, envelope: SignalEnvelope, ): Promise<void> {
    // Aplicar cambios de roles basados en estado de billing
  }
}
```

## Flujo del ciclo de vida del lado del bot

<Steps>
  <Step title="Descubrimiento y registro de listeners">
    Durante el arranque, el *SignalRegistry* registra una función scanner con *ModuleLoader.instanceScanners*. A medida que el module loader instancia cada provider, el scanner verifica si la clase de la instancia está decorada con *@SignalListener()*. Para las clases que coinciden, extrae todos los métodos decorados con *@OnSignal()* y los registra con el *SignalRouter*.
  </Step>
  <Step title="Suscripción a messageCreate de Discord">
    Después del descubrimiento, el método *SignalRegistry.onBoot()* lee la config de Signal desde *ConfigStore* y suscribe un handler al evento *messageCreate* del cliente Discord. Este handler filtra mensajes entrantes y pasa solo tráfico webhook de canales configurados al *SignalProcessor*.
  </Step>
  <Step title="Pipeline de validación de envelope">
    Para cada mensaje webhook entrante, el *SignalProcessor* invoca *SignalValidator.validate()* que ejecuta las diez reglas secuenciales. Tipo de origen, identidad del webhook, frontera de canal, parsing JSON, versión de protocolo, registro de evento, guild ID, nonce, desviación de reloj y firma HMAC se verifican en orden.
  </Step>
  <Step title="Deduplicación de nonce y despacho de handler">
    Después de la validación, el procesador verifica el *NonceCache* para prevenir procesamiento de replay. Si el nonce es nuevo, resuelve el handler a través del *SignalRouter*, opcionalmente valida el payload con el esquema Zod del handler, elimina el mensaje si está configurado, e invoca el handler con el payload y el envelope.
  </Step>
  <Step title="Shutdown graceful">
    Cuando el bot se apaga, *SignalRegistry.onShutdown()* remueve el listener de *messageCreate* del cliente Discord y desregistra el scanner de *ModuleLoader.instanceScanners*. Esta limpieza previene fugas de memoria y listeners fantasma durante hot reloads.
  </Step>
</Steps>

## Inyección de dependencias en listeners

Dado que *@SignalListener()* aplica *@Injectable()* automáticamente, tus clases listener participan en el sistema estándar de inyección de dependencias. Puedes inyectar servicios, repositorios y otros providers a través de parámetros del constructor, igual que cualquier otra clase inyectable en Spraxium.

```typescript filename="src/modules/signal/listeners/guild-config.signal.ts"
import { SignalListener, OnSignal } from '@spraxium/signal';
import type { SignalEnvelope } from '@spraxium/signal';
import { GuildSettingsService } from '../../services/guild-settings.service';

@SignalListener()
export class GuildConfigSignal {
  constructor(private readonly guildSettings: GuildSettingsService) {}

  @OnSignal('config.update')
  async onConfigUpdate(payload: { prefix: string }, envelope: SignalEnvelope): Promise<void> {
    await this.guildSettings.updatePrefix(envelope.guildId, payload.prefix);
  }
}
```

## Diagnósticos de debug

Cuando *debug* está en `true` en la config de Signal, el framework emite mensajes de diagnóstico detallados para cada paso del pipeline de validación y procesamiento. Cada mensaje incluye un prefijo *[SignalValidator]* o *[SignalProcessor]* y reporta la regla exacta que pasó o falló, el estado del nonce, el resultado de resolución del handler, y el resultado de validación Zod.

Esta salida de diagnóstico es invaluable durante la integración inicial e investigación de incidentes. Sin embargo, debe deshabilitarse en producción en estado estable porque incrementa significativamente el volumen de logs en canales con tráfico Signal frecuente. Un patrón práctico es habilitar debug solo en entornos neutral/staging o temporalmente durante análisis de incidentes.

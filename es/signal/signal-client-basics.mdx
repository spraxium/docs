## Rol del emisor

_Signal Client_ existe para publicar eventos desde servicios externos hacia una frontera de integración orientada al bot. Incluso cuando el emisor es un endpoint de API, un worker cron o una Server Action de Next.js, el objetivo operativo siempre es el mismo: impulsar el comportamiento del bot de forma controlada y auditable. El paquete se encarga de la construcción del envelope, firmado HMAC, rotación de webhooks y absorción de rate-limits para que el código del productor se enfoque en lógica de negocio.

Esta perspectiva previene que los equipos conviertan el client en una herramienta de mensajería genérica. El client es más valioso cuando tus servicios necesitan una forma segura y consistente de notificar dominios del bot sobre cambios de estado sin acoplarse a los internos del gateway de Discord.

## Contrato del constructor

_SignalClient_ recibe un solo objeto _SignalClientConfig_ y un _FallbackStore_ personalizado opcional. El constructor crea un _SignalEnvelopeBuilder_ interno para firmado HMAC, un _WebhookPool_ para entrega round-robin y, cuando está habilitado, un _FallbackWorker_ que se inicia inmediatamente. Todos los campos además de `webhookUrls` y `hmacSecret` tienen valores por defecto sensatos, así que una configuración mínima solo requiere esos dos valores.

```typescript filename="signal-client.setup.ts"
import { SignalClient } from '@spraxium/signal-client';

const client = new SignalClient({
  webhookUrls: [
    process.env.SIGNAL_WEBHOOK_URL_1!,
    process.env.SIGNAL_WEBHOOK_URL_2!,
  ],
  hmacSecret: process.env.SIGNAL_HMAC_SECRET!,
  rateLimitStrategy: 'skip',
});
```

El secreto compartido en `hmacSecret` debe coincidir exactamente con la configuración del lado del bot. Si los valores divergen, el bot rechaza cada envelope en la verificación HMAC (regla R10) y ninguna ejecución de listener ocurre.

## Construcción y firmado del envelope

Cada llamada a _send_ delega internamente a _SignalEnvelopeBuilder_, que produce un envelope completo, criptográficamente firmado. El builder genera un UUID aleatorio como nonce, captura el timestamp actual como _sentAt_, y computa la firma HMAC-SHA256 sobre la concatenación `nonce.sentAt.JSON.stringify(payload)`. Esta fórmula es idéntica a la usada por _@spraxium/signal_ en el lado receptor, así que ambos extremos coinciden en el contrato de firmado sin ninguna negociación adicional.

El envelope resultante siempre lleva versión de protocolo `1`, el nombre del evento, el ID de guild objetivo y el payload junto con los campos criptográficos. El código del productor nunca necesita interactuar con el builder directamente porque _send_ encapsula todo el flujo.

## El método send

_send_ es la API principal. Acepta un nombre de evento, un ID de guild, un payload opcional y _SendOptions_ opcionales. El flujo interno construye el envelope, selecciona el siguiente webhook disponible del pool y entrega vía HTTP POST. Si el webhook retorna 429, el pool lo marca como rate-limited e inmediatamente prueba el siguiente candidato. Cuando todos los webhooks se agotan, el comportamiento depende de la _rateLimitStrategy_ configurada.

```typescript filename="emit-event.ts"
await client.send('billing.status.changed', guildId, {
  plan: 'premium',
  expiresAt: '2025-12-31',
});
```

La interfaz _SendOptions_ expone dos campos opcionales. El campo _webhookIndex_ fuerza la entrega a través de un webhook específico, saltando la rotación round-robin. El campo _skipFallback_ es usado internamente por el _FallbackWorker_ para prevenir loops de re-enqueue durante intentos de retry.

## Rotación del pool de webhooks

_WebhookPool_ distribuye los envíos entre los webhooks configurados usando un cursor round-robin. Cuando un webhook retorna HTTP 429, se marca como rate-limited por la duración especificada en el header `Retry-After`, y el pool automáticamente lo salta hasta que el cooldown expire. Agregar múltiples URLs de webhook mejora significativamente la resiliencia ante ráfagas porque el client siempre tiene endpoints alternativos para probar.

El pool expone varios métodos de inspección que son útiles para monitoreo y debugging. Puedes acceder al pool en cualquier momento a través de la propiedad _pool_ en la instancia del client.

## Estrategias de rate-limit

Cuando todos los webhooks en el pool están rate-limited simultáneamente, el client debe decidir qué hacer. El campo _rateLimitStrategy_ controla este comportamiento. Ambas estrategias son opciones válidas dependiendo de la arquitectura de tu servicio.

<Table>
  <TableHead>
    <TableRow>
      <TableCell header>Estrategia</TableCell>
      <TableCell header>Comportamiento</TableCell>
      <TableCell header>Mejor para</TableCell>
    </TableRow>
  </TableHead>
  <TableBody>
    <TableRow>
      <TableCell>`'skip'`</TableCell>
      <TableCell>
        Lanza *AllWebhooksRateLimitedError* inmediatamente. El error incluye el
        estado completo del pool con timestamps de cooldown para que el caller
        decida cuándo reintentar.
      </TableCell>
      <TableCell>
        Servicios que ya tienen su propia orquestación de retry o
        infraestructura de colas.
      </TableCell>
    </TableRow>
    <TableRow>
      <TableCell>`'wait'`</TableCell>
      <TableCell>
        Espera hasta que el webhook más próximo se enfríe (vía
        *msUntilAvailable*), luego reintenta la entrega exactamente una vez. Si
        el retry también falla, lanza error.
      </TableCell>
      <TableCell>
        Absorción de ráfagas cortas dentro del client sin lógica de retry
        externa.
      </TableCell>
    </TableRow>
  </TableBody>
</Table>

El valor por defecto es `'skip'` porque mantiene al client no bloqueante. Para la mayoría de contextos de Server Action o API handler donde una breve latencia es aceptable, `'wait'` proporciona mejor ergonomía al absorber picos de rate-limit transitorios de forma transparente.

## Tipos de error

El client expone dos clases de error que proporcionan información estructurada de fallas. Ambas extienden `Error` y llevan campos legibles por máquina para manejo programático.

_SignalClientError_ se lanza en respuestas HTTP no-2xx, no-429. Expone el _statusCode_ y el _body_ de la respuesta como propiedades, haciendo sencillo registrar fallas de transporte con contexto completo. Este error indica que el endpoint del webhook tuvo un problema, no una situación de rate-limit.

_AllWebhooksRateLimitedError_ extiende _SignalClientError_ con un `statusCode` de `429` y expone un array _poolStatus_. Cada entrada en el array es un snapshot _WebhookStatus_ con el _index_, _url_, flag _available_ y timestamp _rateLimitedUntil_. Este error solo se lanza bajo la estrategia `'skip'` o cuando `'wait'` agota su único reintento.

## Emisor práctico: Server Actions de Next.js

Un patrón de producción común es un panel de administración construido con Next.js donde una Server Action actualiza datos de negocio y emite un evento Signal para el bot. Este enfoque mantiene mutaciones de UI y reacciones del bot débilmente acopladas preservando un workflow de operador único. La Server Action se encarga de la autorización y la persistencia dentro de tu aplicación web, y _Signal Client_ maneja la entrega asíncrona hacia el runtime del bot.

<CodeTabs>
  <Tab label="Server Action">
    ```typescript filename="app/actions/update-prefix.ts"
    'use server';

    import { SignalClient } from '@spraxium/signal-client';

    const client = new SignalClient({
      webhookUrls: [process.env.SIGNAL_WEBHOOK_URL_1!],
      hmacSecret: process.env.SIGNAL_HMAC_SECRET!,
      rateLimitStrategy: 'wait',
    });

    export async function updatePrefix(guildId: string, prefix: string) {
      // persistir en tu base de datos primero
      await db.guilds.update(guildId, { prefix });

      // luego emitir el evento para el bot
      await client.send('config.update', guildId, {
        prefix,
        source: 'admin-panel',
      });
    }
    ```

  </Tab>
  <Tab label="Route Handler de API">
    ```typescript filename="app/api/signal/route.ts"
    import { SignalClient } from '@spraxium/signal-client';
    import { NextResponse } from 'next/server';

    const client = new SignalClient({
      webhookUrls: [
        process.env.SIGNAL_WEBHOOK_URL_1!,
        process.env.SIGNAL_WEBHOOK_URL_2!,
      ],
      hmacSecret: process.env.SIGNAL_HMAC_SECRET!,
      rateLimitStrategy: 'skip',
    });

    export async function POST(req: Request) {
      const { event, guildId, payload } = await req.json();
      await client.send(event, guildId, payload);
      return NextResponse.json({ ok: true });
    }
    ```

## Directrices de modelado de eventos

Prefiere nombres orientados al dominio que describan hechos de negocio en lugar de acciones de transporte. Un nombre como _billing.status.changed_ es más fácil de versionar y razonar que etiquetas orientadas a acciones que filtran detalles de implementación interna. Mantén los nombres de evento en formato minúsculas separadas por punto para ser consistente con las convenciones usadas en todo el sistema de Signal.

Los payloads deben ser intencionalmente pequeños y estables. Incluye solo campos que el bot realmente necesita para tomar decisiones, y evoluciona el schema con cambios aditivos siempre que sea posible. Remover o renombrar un campo existente es un cambio disruptivo que requiere coordinar despliegues tanto del emisor como del listener simultáneamente.

Usa el filtro _events_ en _FallbackConfig_ para enfocar las garantías de confiabilidad en los eventos más críticos. Los eventos que no aparecen en la lista del filtro se entregan por la ruta normal, pero no se encolan en el fallback store en caso de falla.

## Recorrido del flujo de entrega

<Steps>
  <Step title="Construir el envelope firmado">
    El método *send* delega a *SignalEnvelopeBuilder*. El builder genera un
    nonce vía `crypto.randomUUID()`, captura el `Date.now()` actual como
    *sentAt*, computa la firma HMAC-SHA256 y retorna un envelope completo con
    versión de protocolo `1`.
  </Step>
  <Step title="Elegir el siguiente webhook">
    *WebhookPool.next* retorna el siguiente webhook disponible en orden
    round-robin. Si se proporcionó *SendOptions.webhookIndex*, el pool retorna
    esa entrada específica vía *get* en su lugar, saltando la rotación. Si todos
    los webhooks están rate-limited y no se forzó un índice, el flujo salta al
    paso cinco.
  </Step>
  <Step title="Entregar vía HTTP POST">
    El client envía una solicitud POST a la URL del webhook con el envelope
    serializado en JSON envuelto dentro de un campo de contenido de mensaje de
    Discord. Si la respuesta es 2xx, la entrega está completa y el método
    retorna exitosamente.
  </Step>
  <Step title="Manejar respuestas 429">
    Cuando el webhook retorna HTTP 429, el pool marca ese índice como
    rate-limited por la duración parseada del header `Retry-After` (por defecto
    1000 ms). El client inmediatamente prueba el siguiente webhook disponible.
    Esta rotación continúa hasta que un webhook tenga éxito o todos se agoten.
  </Step>
  <Step title="Aplicar estrategia de rate-limit">
    Cuando no quedan webhooks, la *rateLimitStrategy* decide el resultado. Bajo
    `'skip'`, un *AllWebhooksRateLimitedError* se lanza inmediatamente. Bajo
    `'wait'`, el client espera por el cooldown más próximo y reintenta una vez.
    Si el fallback está habilitado y el error es capturable, el envelope se
    encola en el fallback store para retry posterior.
  </Step>
</Steps>

## Cuándo este modelo no es la herramienta correcta

Si requieres confirmación síncrona inmediata de la lógica del bot dentro del mismo ciclo de vida de la solicitud, Signal usualmente no es el contrato correcto. Las llamadas directas entre servicios siguen siendo más simples y más fáciles de rastrear para comunicación in-process dentro de una sola frontera de servicio. Signal está diseñado para entrega de eventos asíncrona, fire-and-forget, donde el productor no necesita una respuesta del consumidor.

De manera similar, si tus eventos requieren garantías estrictas de ordenamiento, una cola de mensajes dedicada como RabbitMQ o Redis Streams es una mejor opción. Signal entrega mensajes tan rápido como el pool de webhooks lo permite, y el ordenamiento de retry después de fallas es best-effort en lugar de estrictamente secuencial.

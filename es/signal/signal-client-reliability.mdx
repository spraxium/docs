## Modelo de confiabilidad

*Signal Client* combina rotación de webhooks con persistencia opcional de fallback para que el tráfico en ráfagas y las fallas temporales no se conviertan en actualizaciones perdidas del lado del bot. El diseño es intencionalmente en capas: primero intenta entrega en vivo a través del pool, luego encola los envelopes fallidos y los reintenta con backoff exponencial. Esto le da a los equipos de bot un camino práctico hacia la confiabilidad sin forzar a cada servicio productor a implementar su propio scheduler de reintentos.

El *FallbackWorker* es la primitiva central de confiabilidad. Cuando el sistema de fallback está habilitado y una llamada a *send* falla, el worker persiste el envelope en un *FallbackStore* y lo reintenta en lotes en segundo plano. Después de agotar el número máximo de intentos, el envelope se mueve a una partición dead-letter para inspección manual.

## Elegir un fallback store

La interfaz *FallbackStore* define un contrato de persistencia con ocho métodos: *enqueue*, *claimBatch*, *markProcessed*, *reschedule*, *moveToDeadLetter*, *has*, *pendingCount* y *deadLetterCount*. Dos implementaciones integradas cubren los escenarios de despliegue más comunes. También puedes implementar la interfaz tú mismo para backends personalizados.

<CodeTabs>
  <Tab label="Store de Archivo">
    ```typescript filename="signal-client-file.ts"
    import { SignalClient } from '@spraxium/signal-client';

    const client = new SignalClient({
      webhookUrls: [process.env.SIGNAL_WEBHOOK_URL_1!], hmacSecret: process.env.SIGNAL_HMAC_SECRET!, rateLimitStrategy: 'wait', fallback: {
        enabled: true, store: 'file', filePath: '.spraxium-signal-fallback.json', retryIntervalMs: 5_000, maxAttempts: 10, batchSize: 20, events: ['config.update', 'billing.status.changed'], }, });
    ```
  </Tab>
  <Tab label="Store Redis">
    ```typescript filename="signal-client-redis.ts"
    import { createClient } from 'redis';
    import { SignalClient, RedisFallbackStore } from '@spraxium/signal-client';

    const redis = createClient({ url: process.env.REDIS_URL });
    await redis.connect();

    const client = new SignalClient(
      {
        webhookUrls: [process.env.SIGNAL_WEBHOOK_URL_1!], hmacSecret: process.env.SIGNAL_HMAC_SECRET!, rateLimitStrategy: 'wait', fallback: {
          enabled: true, store: 'redis', retryIntervalMs: 5_000, maxAttempts: 10, }, }, new RedisFallbackStore(redis), );
    ```
  </Tab>
</CodeTabs>

*FileFallbackStore* mantiene entradas pendientes, dead-letter y procesadas en un solo archivo JSON. Carga el archivo de forma lazy en el primer acceso y escribe después de cada mutación. La lista de procesados se recorta automáticamente a las últimas 5000 entradas cuando excede 10000, previniendo crecimiento ilimitado. Este store funciona bien para desarrollo local y servicios pequeños de instancia única.

*RedisFallbackStore* persiste entradas como claves individuales de Redis bajo el prefijo `spraxium:signal:fallback:pending:`, con entradas dead-letter bajo `spraxium:signal:fallback:deadletter:` e IDs procesados en un set en `spraxium:signal:fallback:processed`. Redis es preferible para emisores de múltiples instancias porque el estado de fallback se comparte entre réplicas sin coordinación del sistema de archivos.

## Backoff exponencial y jitter

Cuando un intento de retry falla, el worker reprograma el envelope usando una fórmula de backoff exponencial con jitter aditivo. La fórmula previene tormentas de retry al distribuir los intentos en intervalos progresivamente más largos, mientras el componente de jitter previene que múltiples envelopes reintenten en el mismo instante.

La fórmula exacta es: `retryIntervalMs * 2^min(attempt, 8) + random() * retryIntervalMs * 0.3`. El exponente tiene un tope en 8 para prevenir retrasos absurdamente largos en conteos de intentos altos. Con el *retryIntervalMs* por defecto de `5000` y *maxAttempts* de `10`, el backoff máximo antes del jitter es aproximadamente 1280 segundos (unos 21 minutos) al octavo intento.

## Manejo de dead-letter

Cuando un envelope agota los *maxAttempts* configurados, el worker lo mueve a la partición dead-letter vía *moveToDeadLetter*. La entrada original se remueve de la cola pendiente y se almacena con la última razón de error adjunta. Los envelopes dead-letter nunca se reintentan automáticamente, por lo que requieren investigación manual.

En el store de archivo, las entradas dead-letter se acumulan en el array `deadLetter` del archivo JSON. En el store Redis, se mueven a claves individuales bajo `spraxium:signal:fallback:deadletter:`. Ambos stores rastrean el conteo vía *deadLetterCount*, que deberías monitorear como parte de tus verificaciones de salud operacional.

El método *stats* en el *FallbackWorker* retorna tanto los conteos de *pending* como de *deadLetter* en una sola llamada. Exponer estos datos a tu sistema de monitoreo es la forma más simple de detectar fallas de entrega persistentes antes de que afecten el comportamiento del bot visible para los usuarios.

```typescript filename="fallback-monitoring.ts"
const { pending, deadLetter } = await client.fallback!.stats();

if (deadLetter > 0) {
  logger.warn(`Signal fallback: ${deadLetter} envelopes dead-letter requieren investigación`);
}

if (pending > 100) {
  logger.warn(`Signal fallback: ${pending} envelopes pendientes, posible falla de entrega sostenida`);
}
```

## Ciclo de vida de una entrada de fallback

<Steps>
  <Step title="Enqueue del envelope en caso de falla">
    Cuando una llamada a *send* lanza una excepción y el fallback está habilitado, el worker verifica si el evento coincide con el filtro *events* (o si no hay filtro establecido, todos los eventos califican). Luego deduplica por nonce para prevenir doble-enqueue y crea una *FallbackEntry* con el envelope, nombre de evento, ID de guild, cero intentos y un *nextAttemptAt* establecido a ahora más *retryIntervalMs*.
  </Step>
  <Step title="Reclamo de lote y retry">
    En cada ciclo de procesamiento (cada *retryIntervalMs*), el worker llama a *claimBatch* para extraer hasta *batchSize* entradas cuyo *nextAttemptAt* haya pasado. Para cada entrada, llama a *send* en el client con `skipFallback: true` para prevenir loops de re-enqueue. Si el envío tiene éxito, la entrada se marca como procesada.
  </Step>
  <Step title="Reprogramación exponencial en caso de falla">
    Cuando un intento de retry falla, el worker computa el próximo tiempo de intento usando la fórmula de backoff exponencial con jitter. El conteo de intentos de la entrada se incrementa, la razón de error se almacena en *lastError*, y la entrada se reprograma vía *reschedule*. Esto continúa hasta que la entrega tenga éxito o el límite de intentos se alcance.
  </Step>
  <Step title="Dead-letter al agotar intentos">
    Cuando el conteo de intentos alcanza *maxAttempts*, el worker llama a *moveToDeadLetter* con la razón del error final. La entrada se remueve permanentemente de la cola pendiente y se coloca en la partición dead-letter. Desde este punto, solo intervención manual o un script personalizado puede reprocesarla.
  </Step>
</Steps>

## Shutdown graceful

Llama a *destroy* en el client durante la secuencia de shutdown de tu aplicación para detener el *FallbackWorker* limpiamente. Esto limpia el timer interno de `setInterval` y previene loops de retry parciales durante despliegues rolling. El worker usa `timer.unref()` para evitar mantener el proceso de Node.js vivo, pero destruir explícitamente el client asegura una limpieza determinista.

```typescript filename="graceful-shutdown.ts"
process.on('SIGTERM', () => {
  client.destroy();
  process.exit(0);
});
```

Si omites la llamada a destroy, los ciclos de retry pendientes pueden ejecutarse parcialmente durante el shutdown, potencialmente produciendo entregas duplicadas. En entornos containerizados donde los shutdowns ocurren frecuentemente, esto es especialmente importante de manejar correctamente.

## Runbook operativo

<Steps>
  <Step title="Observar el comportamiento del pool continuamente">
    Rastrea el estado del pool vía `client.pool.status()` para entender qué endpoints de webhook están sanos, throttled o indisponibles. Correlaciona patrones de rate-limit con la carga de webhooks del lado de Discord para determinar si necesitas agregar más webhooks o reducir la frecuencia de envío.
  </Step>
  <Step title="Monitorear el crecimiento de la cola de fallback">
    Cuando el fallback está habilitado, conteos de *pending* crecientes son tu alerta temprana de fallas de transporte sostenidas. Expón la salida del método *stats* a tu sistema de monitoreo y establece alertas para umbrales apropiados a tu volumen de eventos. Un conteo de pending que se mantiene por encima de cero por más de unos minutos usualmente indica un problema sistémico.
  </Step>
  <Step title="Investigar la acumulación de dead-letter">
    El crecimiento de dead-letter debe activar investigación inmediata. Revisa el *lastError* almacenado en cada entrada dead-letter para determinar si las fallas son transitorias (rate limits, errores de red) o permanentes (URL de webhook inválida, secreto cambiado). Para el store de archivo, inspecciona el array `deadLetter` en el archivo JSON directamente. Para Redis, escanea claves bajo `spraxium:signal:fallback:deadletter:*`.
  </Step>
  <Step title="Ajustar ventanas de retry por criticidad">
    Eventos de alto impacto como billing o escalamiento de moderación justifican más *maxAttempts* y menor *retryIntervalMs* para reintentos iniciales más rápidos. Eventos de bajo impacto deben usar configuraciones más ligeras para prevenir que la presión de cola ahogue el tráfico crítico. Usa el filtro *events* para aplicar fallback solo a los nombres de evento que lo ameriten.
  </Step>
  <Step title="Destruir emisores durante el shutdown">
    Siempre llama a `client.destroy()` en tu handler de SIGTERM o shutdown graceful. Esto detiene el timer del fallback worker y previene procesamiento duplicado durante despliegues rolling. En entornos serverless donde el proceso puede congelarse en lugar de terminarse, considera hacer flush del estado pendiente antes de que el handler retorne.
  </Step>
</Steps>

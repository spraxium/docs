## Qué es un prefix command

Un prefix command es un par de clases que reacciona a un mensaje de texto que comienza con un carácter configurable como `!` o `?`. Cuando un usuario envía `!roll 20`, el framework detecta el prefijo, extrae el nombre del comando, parsea los argumentos, ejecuta guards y despacha el mensaje al handler correcto. Este es el modelo tradicional de comandos de bots Discord que precede a los slash commands, y sigue siendo útil para workflows rápidos de operación, herramientas internas de moderación y bots donde los usuarios ya conocen los atajos.

La diferencia clave entre prefix commands y slash commands es lo que los activa. Los slash commands se activan por el sistema de interacciones nativo de Discord con autocomplete y validación. Los prefix commands se activan por mensajes de texto crudos, lo que significa que el framework debe manejar el parsing, la coerción de argumentos y el reporte de errores por su cuenta. Algunos bots necesitan ambos, y Spraxium permite registrar prefix y slash commands en el mismo módulo sin conflictos.

Los prefix commands en Spraxium no son un legado olvidado. Son clases completas manejadas por el contenedor de inyección de dependencias, registradas en módulos y conectadas al client de Discord automáticamente durante el bootstrap. Esto significa que tus handlers pueden recibir servicios inyectados, participar en el pipeline de guards, beneficiarse del seguimiento de cooldown y usar el mismo manejo de excepciones que los slash commands.

## Un ejemplo básico de comando

Aquí hay dos clases que muestran el patrón de dos clases. La primera es una definición de comando que declara el nombre, aliases y schema de argumentos a través de decorators _@PrefixArg_. La segunda es un handler que recibe los argumentos parseados y el _Message_ crudo vía _@Ctx_. Juntas, cubren la estructura mínima que necesitas para crear un prefix command funcional.

<CodeTabs>
  <Tab label="Comando" language="typescript">
```typescript filename="src/modules/fun/commands/ping.command.ts"
import { PrefixCommand } from '@spraxium/common';

@PrefixCommand({
name: 'ping', aliases: ['p'], description: 'Check bot latency.', category: 'Fun', usage: '!ping', cooldown: 3,
})
export class PingCommand {
build() {}
}

````
  </Tab>
  <Tab label="Handler" language="typescript">
```typescript filename="src/modules/fun/handlers/ping-command.handler.ts"
import { Ctx, PrefixCommandHandler } from '@spraxium/common';
import type { Message } from 'discord.js';
import { PingCommand } from '../commands/ping.command';

@PrefixCommandHandler(PingCommand)
export class PingHandler {
  async handle(@Ctx() message: Message): Promise<void> {
    const latency = Date.now() - message.createdTimestamp;
    await message.reply(`Pong! Latency: ${latency}ms`);
  }
}
````

  </Tab>
</CodeTabs>

Nota que la clase de comando es puramente declarativa: describe cómo se llama el comando, qué aliases acepta y qué cooldown impone. La clase handler contiene la lógica de ejecución y recibe dependencias a través de decorators. Para comandos con argumentos, agregas decorators _@PrefixArg_ en el método _build_ del comando y decorators _@PrefixArg_ correspondientes en los parámetros del método _handle_ del handler. La siguiente página cubre schemas de argumentos en detalle.

## Registrando comandos en un módulo

Los prefix commands no se activan solo por existir en tu código fuente. Necesitas registrar la clase de comando en el array _commands_ del módulo y la clase handler en el array _handlers_ para que el framework sepa sobre ellos y pueda conectarlos durante el bootstrap. Sin ambos registros, tu comando será invisible para el parser o no tendrá handler para despacho.

```typescript filename="src/modules/fun/fun.module.ts"
import { Module } from '@spraxium/common';
import { PingCommand } from './commands/ping.command';
import { PingHandler } from './handlers/ping.handler';

@Module({
  commands: [PingCommand],
  handlers: [PingHandler],
})
export class FunModule {}
```

El módulo debe importarse en tu módulo raíz o en otro módulo dentro de la cadena de imports. Si un handler depende de un servicio, ese servicio debe estar disponible en el alcance de providers del módulo o exportado desde un módulo importado, como cualquier otra clase inyectable. Un módulo de prefix típico organiza commands, handlers y guards en carpetas separadas dentro del directorio de dominio.

<FolderStructure title="Layout típico de módulo prefix">
  <Folder name="src" open>

Esta estructura mantiene las definiciones de comando y los handlers en carpetas dedicadas, con el archivo de módulo en la raíz del dominio. Conforme tu bot crece, puedes agregar más archivos de comando y handler sin cambiar la estructura de carpetas. El archivo de módulo es el único lugar que declara qué commands y handlers están activos.

<Callout tone="info" title="Dos clases por comando">
  Cada prefix command requiere una clase de comando y una clase handler
  registradas en el mismo módulo. La clase de comando declara la estructura y
  los metadatos. La clase handler contiene la lógica de ejecución y las
  dependencias inyectables. Si olvidas registrar cualquiera de las dos, el
  comando no funcionará en runtime.
</Callout>

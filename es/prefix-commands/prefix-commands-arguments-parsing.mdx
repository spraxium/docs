## Cómo la entrada se convierte en argv

Tras la detección del prefijo, el parser elimina el prefijo correspondido y tokeniza el texto restante. El primer token se convierte en el nombre del comando y todo lo demás se convierte en el array argv. La tokenización soporta strings entre comillas dobles, así que `"hello world"` se trata como un único token en lugar de dos. Esto permite que los usuarios pasen argumentos con múltiples palabras sin escapado.

El tokenizador divide por espacios en blanco por defecto. Cuando encuentra unas comillas dobles de apertura, recopila todos los caracteres hasta las comillas de cierre y los emite como un único token. Si las comillas de cierre están ausentes, el resto de la entrada se convierte en el token actual. Este comportamiento es intencional y previene desalineamiento silencioso de argumentos.

Para comandos con subcomandos, el primer token argv se verifica contra nombres de subcomandos registrados antes de que comience el parsing de argumentos. Si coincide con un subcomando, el framework desplaza el array argv y usa el schema de argumentos específico del subcomando en lugar del schema raíz. Esto significa que el parser resuelve el enrutamiento antes de resolver los argumentos.

## Decorators de método @PrefixArg

El namespace *@PrefixArg* proporciona 14 decorators de método tipados que apilas en un método *build* (para comandos simples) o en métodos *@PrefixSubcommand* (para comandos con subcomandos). Cada decorator declara un argumento posicional con su nombre, tipo y reglas de validación opcionales. Los decorators se apilan de arriba hacia abajo para determinar el orden posicional.

<CodeTabs>
  <Tab label="Comando simple" language="typescript">
```typescript filename="src/modules/fun/commands/roll.command.ts"
import { PrefixArg, PrefixCommand } from '@spraxium/common';

@PrefixCommand({ name: 'roll', cooldown: 3 })
export class RollCommand {
  @PrefixArg.Integer('sides', { required: false, min: 2, max: 1000 })
  @PrefixArg.Integer('count', { required: false, min: 1, max: 25 })
  build() {}
}
```
  </Tab>
  <Tab label="Subcomando" language="typescript">
```typescript filename="src/modules/moderation/commands/mod.command.ts"
import { PrefixArg, PrefixCommand, PrefixSubcommand } from '@spraxium/common';

@PrefixCommand({ name: 'mod' })
export class ModCommand {
  @PrefixSubcommand({ name: 'ban', description: 'Banear a un usuario' })
  @PrefixArg.User('target')
  @PrefixArg.Rest('reason', { required: false })
  ban() {}
}
```
  </Tab>
</CodeTabs>

## Tipos de argumento soportados

El framework proporciona 14 tipos de argumento que cubren los patrones de entrada más comunes en bots Discord. Cada tipo tiene su propio parser que convierte el token de texto crudo en una salida tipada. Si la conversión falla, el parser lanza una *ArgumentException* con un mensaje de error descriptivo.

<DropdownGroup>
  <Dropdown title="PrefixArg.String">
    Entrega el token crudo como `string`, sin coerción de tipo. Acepta restricciones *minLength* y *maxLength* para controlar la longitud. Úsalo para texto libre, etiquetas o campos donde no necesitas conversión numérica ni parsing especial.
  </Dropdown>
  <Dropdown title="PrefixArg.Integer">
    Convierte el token con `parseInt` y retorna un `number`. Acepta *min* y *max* para validar rango tras la coerción. Lanza *ArgumentException* si el valor no puede interpretarse como entero.
  </Dropdown>
  <Dropdown title="PrefixArg.Number">
    Convierte el token con `parseFloat` y retorna un `number`. Acepta *min* y *max* para límites numéricos. Úsalo cuando debas aceptar decimales como `3.14`.
  </Dropdown>
  <Dropdown title="PrefixArg.Boolean">
    Convierte valores verdaderos (`true`, `yes`, `1`) a `true` y valores falsos (`false`, `no`, `0`) a `false`. Lanza *ArgumentException* para cualquier token fuera de esos conjuntos. Es ideal para flags de activación y desactivación.
  </Dropdown>
  <Dropdown title="PrefixArg.User">
    Acepta una mención de usuario como `<@123456>` o un ID snowflake crudo y retorna el ID como `string`. El parser elimina el formato de mención automáticamente. Úsalo cuando necesitas resolver una acción sobre un usuario.
  </Dropdown>
  <Dropdown title="PrefixArg.Channel">
    Acepta una mención de canal como `<#456789>` o un snowflake y retorna el ID como `string`. El parser limpia el formato de mención antes de devolver el valor. Es útil para comandos que reciben un canal objetivo.
  </Dropdown>
  <Dropdown title="PrefixArg.Role">
    Acepta una mención de rol como `<@&789012>` o un snowflake y retorna el ID como `string`. El parser normaliza el valor para que el handler reciba siempre el mismo formato. Úsalo para permisos, asignaciones o validaciones por rol.
  </Dropdown>
  <Dropdown title="PrefixArg.Member">
    Se comporta igual que *PrefixArg.User* y retorna el ID como `string`. La diferencia es semántica, para indicar que esperas un miembro del servidor. Esto ayuda a que el contrato del comando sea más explícito para quien lo mantiene.
  </Dropdown>
  <Dropdown title="PrefixArg.Snowflake">
    Valida que el token sea una cadena numérica de 17 a 20 dígitos y la retorna como `string`. A diferencia de *PrefixArg.User*, no acepta formato de mención. Es útil para comandos administrativos que trabajan con IDs crudos.
  </Dropdown>
  <Dropdown title="PrefixArg.Url">
    Valida el token mediante `new URL()` y lo retorna como `string` si la entrada es válida. Lanza *ArgumentException* cuando la URL no pasa la validación. Úsalo para enlaces de imágenes, webhooks o recursos externos.
  </Dropdown>
  <Dropdown title="PrefixArg.Duration">
    Interpreta duraciones como `5m`, `1h30m` o `2d` y retorna milisegundos como `number`. Las unidades soportadas son `ms`, `s`, `m`, `h`, `d` y `w`. Lanza *ArgumentException* si el formato de duración es inválido.
  </Dropdown>
  <Dropdown title="PrefixArg.Color">
    Acepta colores hexadecimales como `#FF0000` o `FF0000` y retorna un entero `number`. Lanza *ArgumentException* cuando el valor no es un hexadecimal válido de 6 dígitos. Es útil para colores de embeds o configuración visual.
  </Dropdown>
  <Dropdown title="PrefixArg.Emoji">
    Parsea emojis personalizados como `<:name:id>` o emojis unicode estándar. Retorna un objeto `ParsedEmoji` con `name`, `id` y `animated`, donde `id` es `null` para unicode. Úsalo en comandos de reacciones o configuración por emojis.
  </Dropdown>
  <Dropdown title="PrefixArg.Rest">
    Captura todos los tokens restantes y los une en una sola `string`. Es ideal para motivos, comentarios o texto libre de varias palabras. Solo puede existir una vez por schema y debe declararse al final.
  </Dropdown>
</DropdownGroup>

## Opciones de validación

Todo decorator *@PrefixArg* acepta un objeto de configuración opcional con reglas de validación. Estas reglas se aplican durante la fase de parsing antes de que el handler reciba el valor. Si cualquier validación falla, el parser lanza una *ArgumentException* que incluye el nombre del argumento, la restricción esperada y la entrada recibida.

<DropdownGroup>
  <Dropdown title="required (boolean, predeterminado true)">
    Cuando vale `true`, el parser lanza *ArgumentException* si el token no existe o llega vacío. Cuando vale `false`, el valor ausente se inyecta como `undefined` y se omiten validaciones posteriores. Es útil para argumentos opcionales con fallback en el handler.
  </Dropdown>
  <Dropdown title="min / max (number)">
    Define límites numéricos para *Integer* y *Number* después de la coerción. Si el valor queda fuera de rango, se lanza *ArgumentException* con detalle del límite incumplido. No aplica para tipos de texto ni para entidades de Discord.
  </Dropdown>
  <Dropdown title="minLength / maxLength (number)">
    Define límites de longitud para *String* y *Rest*. La validación se ejecuta sobre el texto recibido antes de lógica adicional del handler. Es una defensa práctica para evitar entradas excesivamente largas.
  </Dropdown>
  <Dropdown title="choices (string[])">
    Restringe la entrada a un conjunto cerrado de valores permitidos. El parser compara el token crudo antes de la coerción de tipo. Si no coincide con ninguna opción, se lanza *ArgumentException*.
  </Dropdown>
  <Dropdown title="validation.pattern (RegExp)">
    Aplica una expresión regular sobre el token crudo antes de cualquier coerción. Sirve para validar estructuras, prefijos o formatos específicos de dominio. Si no hay match, se lanza *ArgumentException*.
  </Dropdown>
  <Dropdown title="validation.validate (función)">
    Ejecuta una validación personalizada que recibe el valor crudo y devuelve `true` o un mensaje de error. Cuando devuelve texto de error, ese mensaje se incorpora a *ArgumentException*. Es ideal para reglas de negocio que no entran en min, max o regex.
  </Dropdown>
</DropdownGroup>

## Orden del pipeline de validación

El argument parser aplica validaciones en un orden específico para cada argumento. Comprender este orden te ayuda a diseñar schemas que producen mensajes de error claros, porque la primera verificación que falla es lo que el usuario ve. El pipeline se interrumpe en la primera falla, así que las verificaciones posteriores solo se ejecutan si todas las anteriores pasan.

<Steps>
  <Step title="Verificación de presencia">
    Si el argumento es obligatorio y el token crudo está ausente o vacío, el parser lanza una *ArgumentException* inmediatamente. Los argumentos opcionales con tokens ausentes reciben `undefined` y saltan todas las verificaciones restantes. Esta es la primera barrera que impide que valores vacíos lleguen a los parsers específicos de tipo.
  </Step>
  <Step title="Validación personalizada">
    Si un regex *validation.pattern* está definido, la entrada cruda debe coincidir con él. Si una función *validation.validate* está definida, recibe la entrada cruda y debe retornar `true` o un string de error. Ambas verificaciones se ejecutan sobre el texto crudo antes de cualquier coerción de tipo, permitiendo validar restricciones de formato independientemente del parsing de tipo.
  </Step>
  <Step title="Verificación de choices">
    Si un array *choices* está definido, la entrada cruda debe coincidir exactamente con uno de los valores permitidos. Esto funciona como una restricción de enum y es útil para valores de argumento que deben provenir de un conjunto fijo como nombres de estado, tipos de acción o selecciones de modo.
  </Step>
  <Step title="Coerción de tipo y límites">
    El parser convierte el token crudo en el tipo objetivo usando el parser específico del tipo. Para tipos numéricos, las verificaciones de límites (*min*, *max*) se ejecutan tras la coerción exitosa. Para tipos string, las verificaciones de longitud (*minLength*, *maxLength*) se aplican. Si la coerción o los límites fallan, el parser lanza una *ArgumentException*.
  </Step>
</Steps>

## Decorator de parámetro @PrefixArg

En la clase de handler, la función *@PrefixArg* (sin el namespace) es un decorator de parámetro que inyecta un valor de argumento parseado por nombre. El nombre debe coincidir con un argumento declarado en el schema del comando. El decorator *@Ctx* inyecta el objeto *Message* crudo. Juntos, permiten que el handler reciba parámetros completamente tipados sin parsing manual.

```typescript filename="src/modules/fun/handlers/roll-command.handler.ts"
import { Ctx, PrefixArg, PrefixCommandHandler } from '@spraxium/common';
import type { Message } from 'discord.js';
import { RollCommand } from '../commands/roll.command';

@PrefixCommandHandler(RollCommand)
export class RollHandler {
  async handle(
    @Ctx() message: Message, @PrefixArg('sides') sides: number | undefined, @PrefixArg('count') count: number | undefined, ): Promise<void> {
    const faces = sides ?? 6;
    const rolls = count ?? 1;
    const results = Array.from({ length: rolls }, () =>
      Math.floor(Math.random() * faces) + 1, );
    await message.reply(`Rolled ${rolls}d${faces}: [${results.join(', ')}]`);
  }
}
```

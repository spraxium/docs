## Por qué importan las excepciones explícitas

Los clientes HTTP dependen de status codes y formatos de respuesta estables. Lanzar clases _HttpError_ explícitas vuelve los fallos observables, testeables y más fáciles de monitorear en producción. Cada clase de error exportada por el paquete lleva un status code semántico y un mensaje por defecto, permitiendo que el servidor normalice respuestas sin lógica condicional en cada handler.

Cuando los controllers retornan errores genéricos, los clientes pierden contexto semántico y las estrategias de retry se vuelven frágiles. El paquete evita esto ofreciendo una clase dedicada para cada escenario de fallo, todas extendiendo la misma base, garantizando serialización consistente en toda la superficie de la API.

## Clase base HttpError

Cada error de la jerarquía extiende _HttpError_, que a su vez extiende la clase nativa _Error_. Agrega la propiedad read-only _statusCode_ y sobreescribe el constructor para aceptar un mensaje y un status HTTP numérico. Este contrato es la única interfaz que lee la capa de normalización del servidor, así que cualquier clase que extienda _HttpError_ participa automáticamente en el manejo centralizado sin configuración adicional.

La clase base también define `this.name` con el nombre concreto de la clase, haciendo stack traces y dashboards de monitoreo inmediatamente distinguibles entre tipos de error. Las subclases solo necesitan llamar `super(message, statusCode)` con sus propios defaults para mantener el contrato.

## ValidationError y detalles estructurados

_ValidationError_ es la única clase de error que lleva más que un mensaje. Incluye un array _details_ donde cada entrada mapea un nombre de propiedad a las constraints que fallaron. Esta estructura es producida automáticamente por _ValidationPipe_ cuando la validación de body ejecuta reglas de _class-validator_.

La interfaz _ValidationDetail_ expone dos campos: _property_ (nombre del campo en el DTO) y _constraints_ (registro de nombres de regla a mensajes legibles). Como el pipeline usa `whitelist: true` y `forbidNonWhitelisted: true`, cualquier campo inesperado en el body también dispara error de validación, impidiendo que payloads contaminados alcancen la lógica de negocio.

Los clientes reciben una estructura predecible donde `details` siempre es un array, incluso para un solo campo inválido. Esto permite que consumidores de la API iteren el array y mapeen cada entrada a errores inline de formulario o resúmenes de error sin condiciones especiales en la forma de respuesta.

## Comportamiento de normalización en servidor

*HttpServer* registra dos handlers terminales al final del pipeline Hono: *notFound* y *onError*. Juntos garantizan que toda salida posible produce una respuesta JSON con la misma estructura.

El handler *notFound* se activa para cualquier request sin ruta registrada y retorna `{ ok: false, error: 'Not Found' }` con status 404. El handler *onError* cubre excepciones lanzadas con esta precedencia:

1. *ValidationError* retorna status 400 con `{ ok: false, error: 'Validation failed', details: [...] }`;
2. cualquier otra subclase de *HttpError* retorna su propio status code con `{ ok: false, error: message }`;
3. excepciones desconocidas se loguean y retornan `{ ok: false, error: 'Internal server error' }` con status 500.

Como este mapeo es centralizado, los controllers deben lanzar la clase de error más específica disponible y nunca construir payloads JSON de error manuales en el código del handler. La capa de normalización se encarga del resto.

## Contrato de formato de respuesta

Toda respuesta de error sigue el mismo envelope sin importar el status code. El campo `ok` siempre es `false`, el campo `error` lleva la string de mensaje, y `details` aparece únicamente en respuestas de *ValidationError*. Esto permite que consumidores de la API verifiquen `ok` primero y luego ramifiquen por `error` o `details` sin tratamiento especial por endpoint.

## Clases de error personalizadas

Puedes crear clases de error propias extendiendo *HttpError* o cualquier subclase existente. El único requisito es llamar `super` con mensaje y status code. Cualquier clase personalizada que extienda *HttpError* es automáticamente capturada y serializada por la capa de normalización del servidor sin cambios en middleware o configuración.

## Rechazo de guard como frontera de error

*GuardExecutor* ejecuta guards globales en secuencia antes de cualquier handler de controller. Cuando un guard retorna `false`, el executor interrumpe la ejecución y retorna `{ ok: false, error: 'Unauthorized' }` con status 401. Esta respuesta se inyecta directamente, no a través del handler *onError*, lo que significa que rechazos de guard son finales y no pueden capturarse o modificarse por middleware posterior.

Como el comportamiento de guard está fuera del camino de normalización de *HttpError*, las implementaciones de guard deben mantener semántica de rechazo simple. El formato de respuesta 401 sigue el mismo envelope que todos los demás errores, así que los clientes no necesitan manejo especial para fallos de guard.

## Checklist de diseño de excepciones

<Steps>
  <Step title="Lanzar errores semánticos cerca de la regla de negocio">
    Usa clases *HttpError* específicas en el punto donde se toma la decisión de dominio, no solo en el borde del controller. Lanzar *ConflictError* dentro de la lógica de service es más claro que retornar un flag y convertirlo en error en el handler después.
  </Step>
  <Step title="Mantener mensajes accionables y estables">
    Los mensajes de error deben orientar a clientes y operaciones sin exponer detalles internos sensibles. Redacción estable también mejora deduplicación de alertas y lectura de dashboards a lo largo del tiempo.
  </Step>
  <Step title="Usar validación DTO para garantía de forma">
    Deja que *ValidationPipe* y *class-validator* impongan garantías estructurales antes de la lógica de dominio. El pipeline usa `whitelist: true` y `forbidNonWhitelisted: true` por defecto, rechazando propiedades inesperadas automáticamente.
  </Step>
  <Step title="Preferir subclases personalizadas a HttpError directo">
    Crea subclases con nombres descriptivos como *GuildLockedError* o *BridgeUnavailableError* en lugar de lanzar *ForbiddenError* con un string. Clases con nombre son más fáciles de buscar en logs, testear en aserciones y filtrar en dashboards.
  </Step>
  <Step title="Confiar en el mapeo centralizado de respuesta">
    Permite que *HttpServer* normalice errores en JSON de forma consistente en lugar de crear payloads ad-hoc por handler. Un mapeo unificado es más fácil de testear y evolucionar entre equipos, reduciendo drift accidental en el formato de respuesta.
  </Step>
</Steps>

<Callout tone="warning" title="Nunca expongas detalles internos de error a los clientes">
  Las excepciones desconocidas son capturadas por la capa de normalización y retornadas como respuesta genérica 500. El mensaje original del error solo se loguea en el servidor. No sobreescribas este comportamiento para reenviar stack traces o mensajes internos, ya que eso filtra detalles de implementación a consumidores externos.
</Callout>

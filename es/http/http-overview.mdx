## Qué controla este paquete

_@spraxium/http_ es la frontera de transporte entre tu bot de Discord y clientes HTTP externos. Inicia un servidor basado en Hono, aplica middleware y guards compartidos, transforma metadata de controllers en rutas ejecutables y expone una abstracción _BotBridge_ para que los controllers interactúen con guilds, miembros y APIs de moderación sin depender de un modo específico de runtime de Discord. Todo este pipeline, desde la resolución del plugin hasta el shutdown gradual, es gestionado por un único servicio llamado _HttpServer_.

El paquete sigue un modelo de tres capas. La configuración del plugin define comportamiento como host, port, modo de sharding y política de seguridad. La metadata del módulo declara los controllers, services, guards y middleware que forman la superficie API. _HttpServer_ lee ambas capas al arrancar y produce una aplicación Hono completamente cableada conectada a `@hono/node-server`. Esta separación significa que las decisiones de configuración viven fuera del código, los contratos de ruta viven en decorators, y el cableado de runtime queda oculto detrás de un ciclo de vida predecible.

Entender este modelo de ownership es la clave para usar el paquete de forma efectiva. Cada otra página en la categoría HTTP se construye sobre esta base, así que dedicar tiempo aquí evita confusión cuando llegues a los temas de routing, seguridad o bridge más adelante.

## Hono como base HTTP

El paquete usa Hono como framework HTTP en lugar de Express o Fastify. Hono es un framework web ligero, compatible con edge, que funciona en cualquier runtime JavaScript estándar. _HttpServer_ crea una instancia `new Hono()` durante el arranque, registra todas las cadenas de middleware y handlers de ruta sobre ella, y luego pasa `app.fetch` a `@hono/node-server` mediante la función `serve()` con el host y port configurados.

Este diseño significa que controllers y middleware interactúan con los tipos _Context_ y _Next_ de Hono. Decorators de parámetros como _@HttpParam_, _@HttpQuery_ y _@HttpHeader_ leen desde `ctx.req`, mientras que los handlers pueden retornar objetos plain (auto-serializados a JSON) o instancias nativas de `Response` para control total. Entender esta base Hono ayuda cuando necesitas acceder a funcionalidades fuera de la superficie de decorators, como leer cuerpos de request crudos o definir headers de respuesta personalizados mediante `ctx.res`.

## Modelo de ownership global

<Callout tone="warning" title="Usa un único camino de ownership HTTP">
  No crees un módulo wrapper local solo para HTTP. Usa tu AppModule raíz como
  fuente de metadata en defineHttp({module}) e importa *HttpModule* una sola vez
  dentro de AppModule. Múltiples raíces de metadata causan comportamiento
  ambiguo de guards, middleware y ciclo de vida.
</Callout>

El modelo de ownership único significa una entrada de plugin en `spraxium.config.ts`, un decorator _@HttpClientModule_ en tu AppModule y un import de _HttpModule_ en el grafo de módulos. Este patrón mantiene el ownership del ciclo de vida explícito: el servidor hace bind una vez, la política de guards se declara una vez y el orden del middleware es visible en un solo lugar. Cuando los equipos dividen responsabilidades, agregan controllers y services a la misma metadata del módulo en vez de crear árboles de módulos que compiten entre sí.

Esta restricción existe porque _HttpServer_ lee la metadata de _HttpClientModule_ desde la clase referenciada en `defineHttp({ module })`. Si creas un segundo módulo con su propio _HttpClientModule_, el servidor lo ignora porque solo se resuelve la referencia raíz. Mantener todo en AppModule evita desviaciones silenciosas de configuración.

## Modos de arranque: directo vs sharded

_HttpServer_ soporta dos rutas de arranque dependiendo del flag `sharding` en _HttpConfig_. Entender la diferencia es importante porque afecta cuándo arranca el servidor, qué implementación de bridge se selecciona y si un _Client_ de Discord está disponible en el contenedor de DI.

En modo directo (`sharding: false`), el servidor arranca durante _onBoot_ y requiere un _Client_ de Discord inyectado. Si el client falta, el arranque se aborta con un error logueado. _BridgeFactory_ crea un _DirectBotBridge_ que accede a guilds, miembros y bans a través del cache del client y métodos API directamente.

En modo sharded (`sharding: true`), el servidor no arranca durante _onBoot_. En cambio, un bloque estático en _HttpServer_ registra un hook con _ParentHookRegistry_. Cuando el proceso padre genera shards, este hook recibe la instancia de _ShardingManager_ y arranca el servidor fuera del ciclo de vida normal de boot. _BridgeFactory_ crea un _ShardedBotBridge_ que usa `manager.broadcastEval()` para alcanzar guilds a través de shards. Este modo también almacena la referencia del manager en _HttpRegistry_ para que sobreviva al contexto de inicialización estática.

## Flujo de runtime paso a paso

<Steps>
  <Step title="Resolver configuración del plugin">
    *HttpServer* lee la entrada del plugin HTTP desde *ConfigStore* mediante `getPluginConfig(defineHttp)`. Ese objeto porta host, port, modo de sharding, política de seguridad y la referencia a la clase módulo. Si no se encuentra config o `enabled` es `false`, el servidor omite el arranque por completo y registra una advertencia.
  </Step>
  <Step title="Seleccionar estrategia de arranque">
    En modo directo, *onBoot* requiere un *Client* de Discord inyectado. En modo sharded, un hook estático de *ParentHookRegistry* recibe el *ShardingManager* y llama a `server.start()` fuera del ciclo de vida normal. Ambas rutas convergen en el mismo método `start()` que construye la aplicación Hono.
  </Step>
  <Step title="Crear BotBridge">
    *BridgeFactory.create()* verifica si un *ShardingManager* o *Client* está disponible y retorna la bridge correspondiente. La instancia de bridge se agrega al mapa de dependencias de DI para que controllers y services puedan inyectar *BotBridge* como parámetro de constructor.
  </Step>
  <Step title="Aplicar capa global de seguridad">
    El servidor registra middleware en este orden fijo: eliminación de `X-Powered-By`, *SecurityHeadersMiddleware*, *CorsMiddleware*, *BodyLimitMiddleware*, luego cualquier middleware global personalizado de la metadata de *HttpClientModule*. Cada middleware se vincula a la ruta `'*'` para que se ejecute antes de todos los handlers de ruta.
  </Step>
  <Step title="Ejecutar guards globales">
    Si *HttpClientModule* declara guards, *GuardExecutor* los envuelve en un middleware de Hono que se ejecuta secuencialmente. Cuando cualquier guard retorna `false`, la ejecución se detiene y el servidor responde con `{ ok: false, error: 'Unauthorized' }` con status 401. Ningún middleware o handler posterior se ejecuta.
  </Step>
  <Step title="Registrar controllers y services">
    *RouteRegistry* resuelve services primero usando ordenamiento topológico para manejar el orden de dependencias, luego instancia controllers con el mapa de dependencias resuelto. *RouteBuilder* lee la metadata de ruta de cada controller y vincula handlers en Hono con cadenas mergeadas de middleware a nivel de clase y método.
  </Step>
  <Step title="Resolver parámetros y validar DTOs">
    Cuando un request llega a un handler, *ParamResolver* lee la metadata de decorators de parámetro y extrae valores de params, query, body, headers o el *Context* completo de Hono. Si un decorator *@Body* incluye una clase DTO, *ValidationPipe* transforma el body crudo en una instancia tipada y lo valida con reglas de *class-validator* antes de que el handler se ejecute.
  </Step>
  <Step title="Normalizar errores y retornar respuesta">
    Handlers que retornan objetos plain se auto-serializan a JSON con status 200 (o un status personalizado de *@HttpStatus*). Si un handler retorna un `Response` nativo, Hono lo pasa sin cambios. El handler *onError* normaliza excepciones lanzadas: *ValidationError* retorna 400 con detalles, cualquier *HttpError* retorna su propio código de status, y excepciones desconocidas retornan 500 tras loguear.
  </Step>
  <Step title="Shutdown gradual">
    Durante *onShutdown*, el servidor cierra la instancia subyacente de `Server` de Node.js y espera que las conexiones activas se drenen. El handler de shutdown también se registra en `SIGINT` y `SIGTERM` en modo sharded para asegurar que el servidor se detenga limpiamente cuando el proceso padre termina.
  </Step>
</Steps>

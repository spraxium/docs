## Para que sirve la capa de exceptions

En Spraxium, las exceptions no sirven solo para interrumpir el flujo de control cuando algo sale mal. Son el mecanismo que transforma tanto fallas tecnicas como de dominio en comportamiento predecible para el usuario y senales operacionales utiles que ayudan a tu equipo a entender que paso y por que. Sin una capa de exceptions estructurada, cada handler termina inventando su propio estilo de respuesta de error, lo que lleva a UX inconsistente, logs ruidosos y resolucion de incidentes mas lenta en toda la aplicacion.

A medida que el bot crece, las fallas dejan de ser casos extremos y pasan a ser parte de la experiencia normal del producto. Los usuarios encontraran negaciones de permisos, limites de cooldown, argumentos invalidos y recursos no disponibles de forma regular. Una capa dedicada de exceptions asegura que todos estos escenarios produzcan respuestas consistentes, significativas y faciles de mantener conforme el bot evoluciona con el tiempo.

## La clase base SpraxiumException

Todas las exceptions del framework extienden la clase `SpraxiumException`, que provee un constructor unificado con campos para identificar el error, transportar contexto y controlar como el pipeline lo procesa. Cuando creas una instancia pasas un objeto de opciones con los campos listados en la tabla a continuacion, y el pipeline usa esos campos para resolver layouts, decidir si responde al usuario y determinar si registra la falla en logs.

```typescript filename="shop.command.ts"
import { SpraxiumException } from '@spraxium/core';

throw new SpraxiumException({
  code: 'CUSTOM_ERROR',
  message: 'Something went wrong for {{userId}}.',
  props: { userId: '123456' },
  shouldReply: true,
  shouldLog: false,
});
```

Para cualquier cosa mas alla de un throw puntual, extiende `SpraxiumException` en una clase con nombre para que la exception lleve una identidad estable que layouts y reglas de monitoreo puedan referenciar. El constructor acepta el mismo objeto de opciones y lo reenvía a `super`, permitiendo que cada subclase defina su propio code, mensaje por defecto e interfaz *props* tipada.

```typescript filename="order-limit.exception.ts"
import { SpraxiumException } from '@spraxium/core';

interface OrderLimitProps {
  limit: number;
  userId: string;
}

export class OrderLimitExceededException extends SpraxiumException {
  constructor(props: OrderLimitProps) {
    super({
      code: 'ORDER_LIMIT_EXCEEDED',
      message: 'You have exceeded the {{limit}} order limit for today.',
      props,
    });
  }
}

// Uso en cualquier parte del codigo de dominio
throw new OrderLimitExceededException({ limit: 5, userId: interaction.user.id });
```

### Usando exceptions built-in

Las exceptions built-in se importan desde *@spraxium/core* y se lanzan con un objeto de props opcional que adjunta contexto al error. El pipeline resuelve el layout basado en el mapeo de configuracion, renderiza la respuesta al usuario y registra la falla si la politica de logging lo requiere, todo automaticamente y sin codigo adicional en el punto de lanzamiento.

```typescript filename="shop.command.ts"
import {
  CooldownException,
  PermissionDeniedException,
  MaintenanceException,
} from '@spraxium/core';

// El usuario debe esperar antes de usar el command de nuevo
throw new CooldownException({ seconds: 30 });

// El usuario no tiene el permiso requerido
throw new PermissionDeniedException({ permission: 'BanMembers' });

// La funcionalidad esta en mantenimiento con hora estimada de retorno
throw new MaintenanceException({
  feature: 'economy',
  eta: '15 minutes',
});
```

## Error como contrato, no como accidente

Tratar una exception como contrato significa que cada falla relevante tiene intencion explicita detras de ella, en lugar de ser una solucion improvisada o una respuesta generica. En vez de retornar mensajes de error vagos, defines clases de exception y layouts que representan escenarios concretos como permisos insuficientes, violaciones de cooldown, argumentos invalidos o recursos no disponibles. Este enfoque crea un vocabulario estable en todo el equipo donde los ingenieros pueden leer codigo e inmediatamente entender que fallo, por que fallo y como el usuario deberia ser informado.

El objetivo practico es predictibilidad en todo tu bot. El mismo tipo de falla siempre deberia producir la misma clase de exception, el mismo formato de respuesta y la misma senal operacional sin importar donde en el codigo se origina el error. Cuando ese contrato se mantiene de forma consistente, la depuracion se vuelve mas rapida, la experiencia del usuario permanece coherente y el equipo puede razonar sobre fallas sin leer cada handler individualmente para entender el comportamiento de error.

## Donde las exceptions entran en el flujo de runtime

En la practica, una exception se lanza dentro de codigo de dominio como un service, guard, handler o listener, y luego es resuelta por el pipeline central que decide la estrategia de rendering y el comportamiento de logging. Entender este flujo es esencial porque determina donde se originan tus clases de exception, como se seleccionan los layouts para presentar errores a los usuarios y que tipo de datos operacionales se registran para diagnosticos y monitoreo.

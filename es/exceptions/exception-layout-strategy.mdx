## Que es un layout de exception

Cuando un usuario ejecuta un comando y algo sale mal, lo que ve en el chat es el layout. Es la respuesta visual que Spraxium envia en nombre del error: un mensaje de texto, un embed con titulo y descripcion, una respuesta efimera, o cualquier combinacion de esas opciones. La clase de exception es la senal interna que dice que fallo y lleva los datos de contexto. El layout es el template que consume ese contexto y lo transforma en el mensaje de Discord que el usuario va a leer.

Esta separacion hace que el codigo de dominio se mantenga enfocado en reglas de negocio y nunca necesite saber como lucen los errores para el usuario. Un guard que deniega acceso lanza `PermissionDeniedException`. No construye un embed, no compone un string de mensaje ni decide si la respuesta debe ser efimera. Todo eso vive en el layout, que puede ser cambiado o sobreescrito de forma independiente sin tocar ninguna logica de dominio.

## Orden de precedencia en runtime

Spraxium resuelve layouts en un orden fijo, y conocer este orden es esencial para evitar confusion cuando la personalizacion local y la configuracion global coexisten en la misma aplicacion. Cada nivel en la cadena de precedencia puede sobreescribir al que esta debajo, lo que da control granular sobre como se presentan los errores sin crear ambiguedad sobre cual layout realmente renderizara.

1. Override por instancia en el objeto de exception
2. Mapeo en el config que corresponda al nombre de la clase de exception
3. Decorator a nivel de clase con `@WithLayout()`
4. La entrada por defecto en el mapa de layout de la configuracion
5. El *DefaultExceptionLayout* integrado del framework

<Callout tone="warning" title="Consecuencia practica">
  Si mapeas una clase en el mapa de layout del config, ese mapeo tiene prioridad sobre
  el decorator a nivel de clase, lo que permite que la configuracion de la aplicacion sobreescriba
  valores por defecto de biblioteca cuando sea necesario.
</Callout>

## Cuando usar @WithLayout

Usa el decorator cuando la propia clase de exception deberia llevar un layout por defecto semantico como parte de su definicion. Este enfoque funciona bien en paquetes compartidos o codigo de biblioteca donde quieres que la clase venga con un fallback razonable sin depender de la configuracion de la aplicacion consumidora. Los equipos que prefieren convenciones class-first tambien se benefician de este patron porque reduce la posibilidad de olvidar agregar un mapeo de layout al crear una nueva clase de exception.

## Cuando mapear en configuracion

Mapea en *spraxium.config.ts* cuando la politica de layout esta orientada por decisiones de producto como branding, tono de lenguaje o estrategia especifica por entorno. Este es generalmente el mejor default para gobernanza a nivel de aplicacion porque centraliza toda la logica de presentacion de error en un lugar, facilitando la gestion de cambios ya que puedes intercambiar la politica de layout sin modificar ninguna clase de exception de dominio.

## Cuando usar override por instancia

Usa el override por instancia solo para casos contextuales donde la misma clase de exception necesita renderizar de forma diferente dependiendo de condiciones de runtime. Ejemplos tipicos incluyen comunicacion temporal de incidentes, migraciones activas o experimentos de feature-flag donde la respuesta de error necesita cambiar sin modificar definiciones de clase o archivos de configuracion. Como este nivel tiene la mayor prioridad en la cadena de resolucion, tratalo como un camino de excepcion intencional y documenta por que existe para evitar crear comportamiento implicito que es dificil de rastrear al depurar.

## Ejemplo completo con embed personalizado

El siguiente ejemplo muestra como crear una clase de exception con un layout a nivel de decorator y luego sobreescribirlo en la configuracion de la aplicacion. El mapeo en config tiene prioridad sobre el decorator, lo que significa que el layout con branding del equipo renderiza en produccion mientras el decorator sirve como fallback para cualquier consumidor que no configure un mapeo especifico.

```typescript filename="policy.exception.ts"
import { WithLayout } from '@spraxium/common';
import { defineExceptionLayout, SpraxiumException } from '@spraxium/core';
import { EmbedBuilder } from 'discord.js';

const PolicyLayout = defineExceptionLayout((exception) => ({
  embeds: [
    new EmbedBuilder()
      .setColor(0xffb020)
      .setTitle('Accion bloqueada por politica')
      .setDescription(String(exception.props.reason ?? 'Razon no proporcionada')),
  ],
  ephemeral: true,
}));

@WithLayout(PolicyLayout)
export class PolicyException extends SpraxiumException {
  constructor(props: { reason: string }) {
    super({
      code: 'POLICY_BLOCKED',
      message: 'Action blocked: {{reason}}',
      props,
    });
  }
}
```

Y en la configuracion de la aplicacion:

```typescript filename="spraxium.config.ts"
exceptions: {
  logUnhandled: true,
  layouts: {
    default: AppDefaultLayout,
    PolicyException: TeamBrandPolicyLayout,
  },
}
```

## Regla de decision recomendada

Al decidir que enfoque usar, sigue este orden de preferencia para mantener la arquitectura de manejo de errores limpia y sostenible. Usa el mapa de layout del config como la politica primaria para tu aplicacion, usa el decorator como fallback semantico a nivel de clase para bibliotecas y paquetes compartidos, y reserva overrides por instancia exclusivamente para comportamiento genuinamente contextual que no puede expresarse a traves de los otros dos mecanismos.

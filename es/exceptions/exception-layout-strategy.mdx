## Qué es un layout de exception

Cuando un usuario ejecuta un comando y algo sale mal, lo que ve en el chat es el layout. Es la respuesta visual que Spraxium envía en nombre del error: un mensaje de texto, un embed con título y descripción, una respuesta efímera, o cualquier combinación de esas opciones. La clase de exception es la señal interna que dice que falló y lleva los datos de contexto. El layout es el template que consume ese contexto y lo transforma en el mensaje de Discord que el usuario va a leer.

Esta separación hace que el código de dominio se mantenga enfocado en reglas de negocio y nunca necesite saber cómo lucen los errores para el usuario. Un guard que deniega acceso lanza `PermissionDeniedException`. No construye un embed, no compone un string de mensaje ni decide si la respuesta debe ser efímera. Todo eso vive en el layout, que puede ser cambiado o sobreescrito de forma independiente sin tocar ninguna lógica de dominio.

El método *build* retorna un objeto *ExceptionLayoutPayload*. Todos los campos son opcionales y se pueden combinar libremente. Un layout puede retornar solo texto, solo embeds, o una combinación con componentes interactivos.

## Orden de precedencia en runtime

Spraxium resuelve layouts en un orden fijo, y conocer este orden es esencial para evitar confusión cuando la personalización local y la configuración global coexisten en la misma aplicación. Cada nivel en la cadena de precedencia puede sobreescribir al que está debajo, lo que da control granular sobre cómo se presentan los errores sin crear ambigüedad sobre cuál layout realmente renderizará.

1. Override por instancia en el objeto de exception
2. Mapeo en el config que corresponda al nombre de la clase de exception
3. Decorator a nivel de clase con `@WithLayout()`
4. La entrada por defecto en el mapa de layout de la configuración
5. El *DefaultExceptionLayout* integrado del framework

<Callout tone="warning" title="Consecuencia práctica">
  Si mapeas una clase en el mapa de layout del config, ese mapeo tiene prioridad sobre
  el decorator a nivel de clase, lo que permite que la configuración de la aplicación sobreescriba
  valores por defecto de biblioteca cuando sea necesario.
</Callout>

## Cuando usar @WithLayout

Usa el decorator cuando la propia clase de exception debería llevar un layout por defecto semantico como parte de su definición. Este enfoque funciona bien en paquetes compartidos o código de biblioteca donde quieres que la clase venga con un fallback razonable sin depender de la configuración de la aplicación consumidora. Los equipos que prefieren convenciones class-first también se benefician de este patrón porque reduce la posibilidad de olvidar agregar un mapeo de layout al crear una nueva clase de exception.

## Cuando mapear en configuración

Mapea en *spraxium.config.ts* cuando la política de layout esta orientada por decisiones de producto como branding, tono de lenguaje o estrategia específica por entorno. ## Cuando usar override por instancia

Usa el override por instancia solo para casos contextuales donde la misma clase de exception necesita renderizar de forma diferente dependiendo de condiciones de runtime. Ejemplos típicos incluyen comunicaciÃ³n temporal de incidentes, migraciones activas o experimentos de feature-flag donde la respuesta de error necesita cambiar sin modificar definiciones de clase o archivos de configuración. Como este nivel tiene la mayor prioridad en la cadena de resolución, tratalo como un camino de excepciÃ³n intencional y documenta por que existe para evitar crear comportamiento implícito que es dificil de rastrear al depurar.

## Ejemplo completo con embed personalizado

El siguiente ejemplo muestra como crear una clase de exception con un layout a nivel de decorator y luego sobreescribirlo en la configuración de la aplicación. El mapeo en config tiene prioridad sobre el decorator, lo que significa que el layout con branding del equipo renderiza en producción mientras el decorator sirve como fallback para cualquier consumidor que no configure un mapeo específico.

```typescript filename="policy.exception.ts"
import { WithLayout } from '@spraxium/common';
import { defineExceptionLayout, SpraxiumException } from '@spraxium/core';
import { EmbedBuilder } from 'discord.js';

const PolicyLayout = defineExceptionLayout((exception) => ({
  embeds: [
    new EmbedBuilder()
      .setColor(0xffb020)
      .setTitle('Acción bloqueada por política')
      .setDescription(String(exception.props.reason ?? 'Razón no proporcionada')),
  ],
  ephemeral: true,
}));

@WithLayout(PolicyLayout)
export class PolicyException extends SpraxiumException {
  constructor(props: { reason: string }) {
    super({
      code: 'POLICY_BLOCKED',
      message: 'Action blocked: {{reason}}',
      props,
    });
  }
}
```

Y en la configuración de la aplicación:

```typescript filename="spraxium.config.ts"
exceptions: {
  logUnhandled: true,
  layouts: {
    default: AppDefaultLayout,
    PolicyException: TeamBrandPolicyLayout,
  },
}
```

## Regla de decisión recomendada

Al decidir que enfoque usar, sigue este orden de preferencia para mantener la arquitectura de manejo de errores limpia y sostenible. Usa el mapa de layout del config como la política primaria para tu aplicación, usa el decorator como fallback semantico a nivel de clase para bibliotecas y paquetes compartidos, y reserva overrides por instancia exclusivamente para comportamiento genuinamente contextual que no puede expresarse a traves de los otros dos mecanismos.

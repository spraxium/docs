## Por qué configurar exceptions en el core

La configuración central de exceptions impide que los equipos manejen la misma categoría de falla de formas diferentes entre modulos. Al definir la política en *spraxium.config.ts*, el comportamiento de runtime permanece coherente incluso cuando múltiples features y equipos evolucionan en paralelo, porque cada exception sigue las mismas reglas de resolución sin importar donde se origina. Esto también reduce significativamente el riesgo de refactorización, ya que cambiar la estrategia de fallback o actualizar un mapeo de layout requiere tocar una única ubicación de política en lugar de buscar a traves de decenas de archivos de command y listener dispersos por el proyecto.

## Configuración minima

Una baseline pequeña ya da control significativo sobre como tu bot maneja errores en producción. Definir un fallback de layout por defecto y una política de logging de errores no manejados asegura que cada exception en el sistema produzca una respuesta para el usuario y una señal operacional, incluso cuando no has mapeado esa clase específica de exception a un layout dedicado. Esto previene respuestas vacías o rotas que confunden a los usuarios durante escenarios de falla inesperados.

```typescript filename="spraxium.config.ts"
import { defineConfig } from '@spraxium/core';

export default defineConfig({
  exceptions: {
    logUnhandled: true,
    layouts: {
      default: DefaultFriendlyLayout,
    },
  },
});
```

## Configuración con mapeo por clase

Cuando tu aplicación tiene múltiples dominios como moderación, billing y onboarding, mapear layouts por clase de exception preserva contexto y tono para cada tipo de falla. Cada familia de falla puede retornar lenguaje y orientación apropiada a la acción del usuario, lo que mejora significativamente la experiencia de error comparada con un único mensaje genérico para cada escenario en el bot.

```typescript filename="spraxium.config.ts"
import { defineConfig } from '@spraxium/core';
import {
  DefaultFriendlyLayout,
  CooldownLayout,
  PermissionDeniedLayout,
  ArgumentLayout,
} from './layouts/exceptions.layouts';

export default defineConfig({
  exceptions: {
    logUnhandled: true,
    layouts: {
      default: DefaultFriendlyLayout,
      CooldownException: CooldownLayout,
      PermissionDeniedException: PermissionDeniedLayout,
      ArgumentException: ArgumentLayout,
    },
  },
});
```

## Orden de resolución de layout

La selección de layout sigue un orden estable que determina cuál layout renderiza la respuesta cuando múltiples opciones de personalización coexisten. Entender este orden previene ambigüedad cuando overrides locales y configuraciones de fallback global están activos al mismo tiempo, y te permite apilar políticas de forma predecible desde lo más específico a lo más general.

1. Override por instancia a traves del objeto de exception directamente
2. Entrada en el mapa de layout del config que corresponda al nombre de la clase de exception
3. Layout a nivel de decorator de clase definido con `@WithLayout()`
4. El layout por defecto definido en la configuración
5. El layout por defecto integrado del framework como fallback final

<Callout tone="warning" title="Los nombres de clase son claves">
  El mapa de layout usa nombres de clase de exception como claves. Si una clase es renombrada durante
  refactorización, actualiza el mapeo para evitar comportamiento de fallback accidental que cambia
  silenciosamente la experiencia del usuario.
</Callout>

## Estrategia por entorno

Una baseline comun de equipo es mantener la opción de logging de errores no manejados como verdadera en staging y producción mientras se preservan layouts amigables en todos los entornos. Esto mantiene los diagnósticos fuertes para operaciones sin filtrar detalles internos hacia los usuarios, porque el layout comunica con usuarios mientras el logging comunica con el equipo responsable de monitoreo y respuesta a incidentes.

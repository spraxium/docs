## Failure paths

When a prefix command fails at any stage, the framework produces a specific exception that you can intercept and format. Understanding the failure taxonomy helps you design user-friendly error responses instead of letting raw errors leak into chat. Each failure type maps to a distinct exception class with its own layout hook.

<DropdownGroup>
  <Dropdown title="CommandNotFoundException (unknown command)">
    Thrown when the prefix is recognized but the command name does not match any
    entry in the registry. The exception exposes the attempted *command* name
    and the matched *prefix* as properties. You can intercept it with an
    exception layout to send a helpful reply, or enable *replyOnNotFound* in
    *PrefixConfig* for automatic feedback without a custom layout.
  </Dropdown>
  <Dropdown title="ArgumentException (missing or invalid argument)">
    Thrown when a required argument is absent or when type coercion or
    validation fails for a provided value. The exception exposes the argument
    *name*, the *expected* constraint, the *received* raw input, and the *usage*
    string from the command definition. Register a layout for
    *ArgumentException* to display consistent error messages across all
    commands.
  </Dropdown>
  <Dropdown title="ForbiddenException or custom (guard denial)">
    Thrown when a guard's *canActivate* method returns `false` or throws. If the
    guard returns `false`, the framework throws a *ForbiddenException*
    automatically. If the guard throws its own exception, that exception
    propagates unchanged and can be caught by a matching layout.
  </Dropdown>
  <Dropdown title="Cooldown active (silently blocked)">
    When a user triggers a command before the cooldown expires, the framework
    silently blocks execution without throwing an exception. You can read the
    remaining seconds from *PrefixCooldownManager.check* and send a custom reply
    from a cooldown guard if you want user-facing feedback.
  </Dropdown>
</DropdownGroup>

## Exception layouts

Exception layouts let you define custom responses for specific exception types. Instead of writing try-catch blocks in every handler, you register a layout once and the framework applies it automatically whenever that exception type is thrown during prefix command execution. This keeps your handlers clean and your error formatting consistent across the entire bot.

A layout receives the original _Message_ and the thrown exception, so you can build rich error embeds with context about what went wrong. The _defineExceptionLayout_ helper creates a layout factory that the framework discovers through your module registration.

<CodeTabs>
  <Tab label="Argument layout" language="typescript">
```typescript filename="src/modules/prefix/layouts/argument-exception.layout.ts"
import { defineExceptionLayout } from '@spraxium/core';
import { ArgumentException } from '@spraxium/common';

export const ArgumentExceptionLayout = defineExceptionLayout(
  ArgumentException,
  async (message, exception) => {
    await message.reply({
      content: `Invalid argument: **${exception.argName}** - ${exception.message}`,
    });
  },
);
```
  </Tab>
  <Tab label="Not found layout" language="typescript">
```typescript filename="src/modules/prefix/layouts/not-found.layout.ts"
import { defineExceptionLayout } from '@spraxium/core';
import { CommandNotFoundException } from '@spraxium/common';

export const CommandNotFoundLayout = defineExceptionLayout(
  CommandNotFoundException,
  async (message, exception) => {
    await message.reply({
      content: `Command \`${exception.commandName}\` not found. Use \`!help\` to see available commands.`,
    });
  },
);
```
  </Tab>
</CodeTabs>

Register layouts in your module alongside commands and handlers. The framework picks them up during module initialization and applies them to the correct exception types automatically.

```typescript filename="src/modules/prefix/prefix.module.ts"
import { Module } from '@spraxium/common';
import { RollCommand } from './commands/roll.command';
import { RollHandler } from './handlers/roll.handler';
import { ArgumentExceptionLayout } from './layouts/argument-exception.layout';
import { CommandNotFoundLayout } from './layouts/not-found.layout';

@Module({
  commands: [RollCommand],
  handlers: [RollHandler],
  layouts: [ArgumentExceptionLayout, CommandNotFoundLayout],
})
export class PrefixModule {}
```

## Dispatcher introspection

The _PrefixDispatcher_ exposes methods for inspecting the registered command tree at runtime. The _getCommandInfo_ method returns an array of _PrefixCommandInfo_ objects with command names, descriptions, subcommand lists, and argument schemas. This is the foundation for building dynamic help commands.

The dispatcher also exposes _getResolved_ for the fully resolved command-handler bindings, _size_ for the total number of registered entries, _commandCount_ for distinct top-level commands, and _guildPrefixes_ for the guild prefix manager. These introspection methods are useful for admin dashboards, health checks, and debugging tools.

```typescript filename="src/modules/prefix/handlers/help-command.handler.ts"
import { Ctx, PrefixCommandHandler } from '@spraxium/common';
import { PrefixDispatcher } from '@spraxium/core';
import type { Message } from 'discord.js';
import { HelpCommand } from '../commands/help.command';

@PrefixCommandHandler(HelpCommand)
export class HelpHandler {
  constructor(private readonly dispatcher: PrefixDispatcher) {}

  async handle(@Ctx() message: Message): Promise<void> {
    const commands = this.dispatcher.getCommandInfo();
    const lines = commands.map((cmd) => {
      const subs = cmd.subcommands?.map((s) => s.name).join(', ');
      return subs
        ? `**${cmd.name}** - subcommands: ${subs}`
        : `**${cmd.name}**`;
    });
    await message.reply(lines.join('\n') || 'No commands registered.');
  }
}
```

## Guards with prefix commands

Prefix commands support the same guard system used by slash commands and listeners. Decorate a handler class with _@UseGuards_ to attach one or more guards that run before the handler executes. If any guard returns `false` or throws, the handler is skipped. Guards are useful for permission checks, role restrictions, and channel-type filtering.

Guards receive the execution context, which for prefix commands includes the original _Message_ object. You can use the same guard classes across prefix commands, slash commands, and listeners without modification. This is one of the benefits of the unified guard architecture.

<CodeTabs>
  <Tab label="Guard" language="typescript">
```typescript filename="src/modules/moderation/guards/moderator.guard.ts"
import { CanActivate, ExecutionContext } from '@spraxium/common';

export class ModeratorGuard implements CanActivate {
  async canActivate(context: ExecutionContext): Promise<boolean> {
    const message = context.getMessage();
    const member = message.member;
    if (!member) return false;
    return member.permissions.has('ModerateMembers');
  }
}
```
  </Tab>
  <Tab label="Handler with guard" language="typescript">
```typescript filename="src/modules/moderation/handlers/ban-command.handler.ts"
import { Ctx, PrefixArg, PrefixCommandHandler, UseGuards } from '@spraxium/common';
import type { Message } from 'discord.js';
import { ModeratorGuard } from '../guards/moderator.guard';
import { ModCommand } from '../commands/mod.command';

@UseGuards(ModeratorGuard)
@PrefixCommandHandler(ModCommand, { subcommand: 'ban' })
export class BanHandler {
  async handle(
    @Ctx() message: Message,
    @PrefixArg('target') targetId: string,
    @PrefixArg('reason') reason: string | undefined,
  ): Promise<void> {
    await message.guild?.members.ban(targetId, { reason: reason ?? 'No reason provided' });
    await message.reply(`Banned <@${targetId}>.`);
  }
}

```
  </Tab>
</CodeTabs>

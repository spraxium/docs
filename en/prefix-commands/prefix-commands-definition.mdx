## Separating structure from execution

In Spraxium, prefix commands use a two-class pattern. The _@PrefixCommand_ decorator marks a class as a command definition that describes the name, aliases, argument schema, and metadata. The _@PrefixCommandHandler_ decorator marks a separate class that contains the _handle_ method with the actual execution logic. This separation keeps the command contract stable while letting you evolve handler behavior independently.

The command class acts as a metadata container. It declares what the command is called, what arguments it expects, and how it appears in help output. The handler class is where you inject services, interact with Discord, and produce responses. This split also enables multiple handlers for the same command through the subcommand routing mechanism, where each subcommand gets its own focused handler.

Both classes must be registered in the same module. Command classes go in the _commands_ array and handler classes go in the _handlers_ array. If only commands are registered, nothing executes because there is no handler to call. If only handlers are registered, there is no command schema for the registry to index, so the parser cannot resolve the command name.

## A simple command

The simplest prefix command needs one command class with an argument schema and one handler class with a _handle_ method. The command class uses _@PrefixArg_ method decorators on a _build_ method to declare positional arguments. The handler class uses _@PrefixArg_ parameter decorators on the _handle_ method to receive the parsed values.

<CodeTabs>
  <Tab label="Command" language="typescript">
```typescript filename="src/modules/fun/commands/roll.command.ts"
import { PrefixArg, PrefixCommand } from '@spraxium/common';

@PrefixCommand({
  name: 'roll',
  aliases: ['r', 'dice'],
  description: 'Roll one or more dice.',
  category: 'Fun',
  usage: '!roll [sides] [count]',
  cooldown: 3,
})
export class RollCommand {
  @PrefixArg.Integer('sides', { required: false, min: 2, max: 1000 })
  @PrefixArg.Integer('count', { required: false, min: 1, max: 25 })
  build() {}
}

````
  </Tab>
  <Tab label="Handler" language="typescript">
```typescript filename="src/modules/fun/handlers/roll-command.handler.ts"
import { Ctx, PrefixArg, PrefixCommandHandler } from '@spraxium/common';
import type { Message } from 'discord.js';
import { RollCommand } from '../commands/roll.command';

@PrefixCommandHandler(RollCommand)
export class RollHandler {
  async handle(
    @Ctx() message: Message,
    @PrefixArg('sides') sides: number | undefined,
    @PrefixArg('count') count: number | undefined,
  ): Promise<void> {
    const faces = sides ?? 6;
    const rolls = count ?? 1;
    const results = Array.from({ length: rolls }, () =>
      Math.floor(Math.random() * faces) + 1, );
    const total = results.reduce((a, b) => a + b, 0);
    await message.reply(`Rolled ${rolls}d${faces}: [${results.join(', ')}] = ${total}`);
  }
}
````

  </Tab>
</CodeTabs>

The _@PrefixArg_ method decorators on the command class define the argument schema. They stack top-to-bottom to determine positional order: the first decorator becomes position 0, the second becomes position 1, and so on. In the handler, _@PrefixArg('name')_ parameter decorators inject the parsed value by name. The _@Ctx_ decorator injects the raw _Message_ object.

## Subcommand routing

For command families like `!mod warn`, `!mod kick`, and `!mod ban`, use _@PrefixSubcommand_ to define each route on the command class and connect each route to a dedicated handler via _@PrefixCommandHandler_ with a _subcommand_ option. The first argv token after the command name is matched against registered subcommand names, and the framework routes automatically.

<CodeTabs>
  <Tab label="Command" language="typescript">
```typescript filename="src/modules/moderation/commands/mod.command.ts"
import { PrefixArg, PrefixCommand, PrefixSubcommand } from '@spraxium/common';

@PrefixCommand({
  name: 'mod',
  aliases: ['m'],
  description: 'Moderation commands.',
  category: 'Moderation',
  usage: '!mod <warn|kick|ban> <user> [reason]',
})
export class ModCommand {
  @PrefixSubcommand({ name: 'warn', description: 'Warn a user' })
  @PrefixArg.User('target')
  @PrefixArg.Rest('reason', { required: false })
  warn() {}

  @PrefixSubcommand({ name: 'kick', description: 'Kick a user' })
  @PrefixArg.User('target')
  @PrefixArg.Rest('reason', { required: false })
  kick() {}

  @PrefixSubcommand({ name: 'ban', description: 'Ban a user' })
  @PrefixArg.User('target')
  @PrefixArg.Rest('reason', { required: false })
  ban() {}
}

````
  </Tab>
  <Tab label="Warn Handler" language="typescript">
```typescript filename="src/modules/moderation/handlers/mod-warn-command.handler.ts"
import { Ctx, PrefixArg, PrefixCommandHandler, UseGuards } from '@spraxium/common';
import type { Message } from 'discord.js';
import { ModCommand } from '../commands/mod.command';
import { ModeratorGuard } from '../guards/moderator.guard';

@UseGuards(ModeratorGuard)
@PrefixCommandHandler(ModCommand, { subcommand: 'warn' })
export class ModWarnHandler {
  async handle(
    @Ctx() message: Message,
    @PrefixArg('target') userId: string,
    @PrefixArg('reason') reason: string | undefined,
  ): Promise<void> {
    const text = reason ?? 'No reason provided';
    await message.reply(`Warning <@${userId}>. Reason: ${text}`);
  }
}
````

  </Tab>
</CodeTabs>

Each subcommand method on the command class defines its own argument schema. The _warn_ subcommand expects a user mention followed by optional rest text, while _kick_ and _ban_ follow the same shape. Each subcommand gets its own handler class, which means you can apply different guards, inject different services, and handle errors independently per subcommand.

## @PrefixCommand configuration

The _@PrefixCommand_ decorator accepts a configuration object that controls how the command appears in help output, how it is triggered, and what operational limits it has. All fields except _name_ are optional, but providing _description_, _usage_, and _category_ significantly improves the output of any help command implementation.

<DropdownGroup>
  <Dropdown title="name (string, required)">
    The primary command trigger that users type after the prefix. When a user sends `!ping`, the parser strips the prefix, reads `ping`, and looks for a registered command with that name. Every command must have a unique name; if two commands share the same name, the last one registered will shadow the first.
  </Dropdown>
  <Dropdown title="aliases (string[])">
    Alternative trigger names that resolve to the same command and handler. Setting `aliases: ['p', 'latency']` means `!p` and `!latency` work identically to `!ping`. Aliases share the same cooldown pool as the primary name, so a user on cooldown from `!ping` is also blocked from using `!p`.
  </Dropdown>
  <Dropdown title="description (string)">
    A short line of text describing what the command does, used in help command output. The framework does not display this automatically; it is available through the dispatcher's *getCommandInfo* method so your own help handler can format it. Keeping descriptions concise and consistent across commands improves the quality of bot help embeds.
  </Dropdown>
  <Dropdown title="usage (string)">
    A human-readable string showing the expected invocation format, like `!ban <user> [reason]`. The framework stores this on the command metadata and returns it as part of *getCommandInfo*, but it does not parse or validate it automatically. Use it in help embeds and in *ArgumentException* messages so users know the correct syntax when parsing fails.
  </Dropdown>
  <Dropdown title="category (string)">
    A grouping label for organizing commands in help output. Commands in the same category are typically displayed together in help embeds. The framework does not enforce categories or assign special behavior based on them; they are purely metadata that your help handler reads from *getCommandInfo*.
  </Dropdown>
  <Dropdown title="cooldown (number)">
    A per-command cooldown in seconds that overrides the global *defaultCooldown* from *PrefixConfig*. When a user runs the command, the framework records the timestamp and blocks subsequent runs until the cooldown expires. Setting `cooldown: 0` disables the cooldown for this command regardless of the global default.
  </Dropdown>
</DropdownGroup>

## Module registration

Both command and handler classes must be registered in a module for the framework to discover them during bootstrap. The command class goes in the _commands_ array and the handler class goes in the _handlers_ array. The framework processes both arrays during module loading, registers the command metadata in the _PrefixRegistry_, and wires the handler instances through the DI container.

```typescript filename="src/modules/moderation/moderation.module.ts"
import { Module } from '@spraxium/common';
import { ModCommand } from './commands/mod.command';
import { RollCommand } from '../fun/commands/roll.command';
import { ModWarnHandler } from './handlers/mod-warn.handler';
import { ModKickHandler } from './handlers/mod-kick.handler';
import { ModBanHandler } from './handlers/mod-ban.handler';
import { RollHandler } from '../fun/handlers/roll.handler';

@Module({
  commands: [ModCommand, RollCommand],
  handlers: [ModWarnHandler, ModKickHandler, ModBanHandler, RollHandler],
})
export class ModerationModule {}
```

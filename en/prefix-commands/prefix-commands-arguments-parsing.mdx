## How message input becomes argv

After prefix detection, the parser strips the matched prefix and tokenizes the remaining text. The first token becomes the command name and everything after it becomes the argv array. Tokenization supports double-quoted strings, so `"hello world"` is treated as a single token instead of two. This lets users pass multi-word arguments without escaping.

The tokenizer splits on whitespace by default. When it encounters an opening double quote, it collects all characters until the closing quote and emits them as one token. If a closing quote is missing, the rest of the input becomes the current token. This behavior is intentional and prevents silent argument misalignment.

For commands with subcommands, the first argv token is checked against registered subcommand names before argument parsing begins. If it matches a subcommand, the framework shifts the argv array and uses the subcommand-specific argument schema instead of the root schema. This means the parser resolves routing before it resolves arguments.

## @PrefixArg method decorators

The _@PrefixArg_ namespace provides 14 typed method decorators that you stack on a _build_ method (for simple commands) or on _@PrefixSubcommand_ methods (for subcommand commands). Each decorator declares one positional argument with its name, type, and optional validation rules. Decorators stack top-to-bottom to determine positional order.

<CodeTabs>
  <Tab label="Simple Command" language="typescript">
```typescript filename="src/modules/fun/commands/roll.command.ts"
import { PrefixArg, PrefixCommand } from '@spraxium/common';

@PrefixCommand({ name: 'roll', cooldown: 3 })
export class RollCommand {
@PrefixArg.Integer('sides', { required: false, min: 2, max: 1000 })
@PrefixArg.Integer('count', { required: false, min: 1, max: 25 })
build() {}
}

````
  </Tab>
  <Tab label="Subcommand" language="typescript">
```typescript filename="src/modules/moderation/commands/mod.command.ts"
import { PrefixArg, PrefixCommand, PrefixSubcommand } from '@spraxium/common';

@PrefixCommand({ name: 'mod' })
export class ModCommand {
  @PrefixSubcommand({ name: 'ban', description: 'Ban a user' })
  @PrefixArg.User('target')
  @PrefixArg.Rest('reason', { required: false })
  ban() {}
}
````

  </Tab>
</CodeTabs>

## Supported argument types

The framework provides 14 argument types that cover the most common Discord bot input patterns. Each type has its own parser that converts the raw text token into a typed output. If conversion fails, the parser throws an _ArgumentException_ with a descriptive error message.

<DropdownGroup>
  <Dropdown title="PrefixArg.String">
    Passes the raw token as a `string` with no coercion. Accepts *minLength* and *maxLength* constraints for bounding character count. Use this for free-form text fields, tags, or any input that does not need numeric or Discord-specific conversion.
  </Dropdown>
  <Dropdown title="PrefixArg.Integer">
    Parses the token with `parseInt` and returns a `number`. Accepts *min* and *max* constraints to bound the parsed integer. Throws an *ArgumentException* if the token cannot be parsed as a whole number.
  </Dropdown>
  <Dropdown title="PrefixArg.Number">
    Parses the token with `parseFloat` and returns a `number`. Accepts *min* and *max* constraints for bounds checking. Use this instead of *PrefixArg.Integer* when decimal values like `3.14` are valid input.
  </Dropdown>
  <Dropdown title="PrefixArg.Boolean">
    Converts truthy tokens (`true`, `yes`, `1`) to `true` and falsy tokens (`false`, `no`, `0`) to `false`. Throws an *ArgumentException* for any token that does not match either set. This is useful for toggle flags in moderation or configuration commands.
  </Dropdown>
  <Dropdown title="PrefixArg.User">
    Accepts a user mention like `<@123456>` or a raw snowflake ID and returns the extracted `string` ID. The parser strips the mention formatting automatically. Use this when you need the user ID to look up or target a member.
  </Dropdown>
  <Dropdown title="PrefixArg.Channel">
    Accepts a channel mention like `<#456789>` or a raw snowflake ID and returns the `string` ID. The parser strips the mention formatting. Use this for commands that accept a target channel as input.
  </Dropdown>
  <Dropdown title="PrefixArg.Role">
    Accepts a role mention like `<@&789012>` or a raw snowflake ID and returns the `string` ID. The parser strips the mention formatting. Use this for commands that assign or check roles.
  </Dropdown>
  <Dropdown title="PrefixArg.Member">
    Behaves identically to *PrefixArg.User*, accepting a user mention or snowflake and returning the `string` ID. The distinction is semantic, signaling to readers that you expect a server member rather than any user.
  </Dropdown>
  <Dropdown title="PrefixArg.Snowflake">
    Validates that the token is a 17 to 20 digit numeric string and returns it as a `string`. Unlike *PrefixArg.User*, this type rejects mention formatting and only accepts bare snowflake IDs. Use it for admin commands that target arbitrary Discord entities by raw ID.
  </Dropdown>
  <Dropdown title="PrefixArg.Url">
    Validates the token using `new URL()` and returns it as a `string` if parsing succeeds. Throws an *ArgumentException* for any input that is not a valid URL. Use this for commands that accept image links, webhook URLs, or any web address.
  </Dropdown>
  <Dropdown title="PrefixArg.Duration">
    Parses duration strings like `5m`, `1h30m`, or `2d` into a `number` of milliseconds. Supported units are `ms`, `s`, `m`, `h`, `d`, and `w`. Throws an *ArgumentException* if the token cannot be interpreted as a valid duration expression.
  </Dropdown>
  <Dropdown title="PrefixArg.Color">
    Accepts hex color strings like `#FF0000` or `FF0000` and returns an integer (`number`) representation. Throws an *ArgumentException* for input that is not a valid 6-digit hex color. Use this for embed color fields or role color setters.
  </Dropdown>
  <Dropdown title="PrefixArg.Emoji">
    Parses a custom Discord emoji like `<:name:id>` or a standard unicode emoji and returns a `ParsedEmoji` object with `name`, `id`, and `animated` fields. The `id` field is `null` for unicode emojis. Use this for reaction management or emoji-based configuration commands.
  </Dropdown>
  <Dropdown title="PrefixArg.Rest">
    Collects all remaining tokens after the previous argument position and joins them into a single `string`. This is useful for free-text fields like ban reasons or message content where the input can span multiple words. Only one *PrefixArg.Rest* can appear per schema and it must be the last declared argument.
  </Dropdown>
</DropdownGroup>

## Validation options

Every _@PrefixArg_ decorator accepts an optional configuration object with validation rules. These rules are applied during the parsing phase before the handler receives the value. If any validation fails, the parser throws an _ArgumentException_ that includes the argument name, the expected constraint, and the received input.

<DropdownGroup>
  <Dropdown title="required (boolean, default true)">
    When `true`, the parser throws an *ArgumentException* if the argument token
    is missing or empty. When `false`, a missing token results in `undefined`
    being injected and all further validation is skipped. Marking an argument as
    optional is useful for fields like reason text or count modifiers that have
    sensible defaults in the handler.
  </Dropdown>
  <Dropdown title="min / max (number)">
    Numeric bounds applied to *Integer* and *Number* values after coercion. If
    the parsed number falls below *min* or above *max*, the parser throws an
    *ArgumentException* with the constraint and the received value. These
    options are not applicable to string or Discord entity types.
  </Dropdown>
  <Dropdown title="minLength / maxLength (number)">
    Character length bounds applied to *String* and *Rest* values. The check
    runs on the raw token before any further processing. Use *maxLength* to
    protect against unexpectedly long inputs in free-text fields.
  </Dropdown>
  <Dropdown title="choices (string[])">
    An allowlist of valid raw input values. The parser checks the raw token
    against this array before coercion and throws an *ArgumentException* if the
    token is not included. This is useful for enum-style arguments like status
    names, action types, or mode selectors.
  </Dropdown>
  <Dropdown title="validation.pattern (RegExp)">
    A regular expression that the raw token must match before any other
    processing. The check runs on the raw string, so it can validate format
    constraints like specific prefixes, patterns, or structures that the
    built-in type parsers do not enforce. If the regex does not match, an
    *ArgumentException* is thrown.
  </Dropdown>
  <Dropdown title="validation.validate (function)">
    A custom function that receives the raw token and returns either `true` to
    pass or an error string to fail. The error string is included in the
    *ArgumentException* message so users see your custom feedback. Use this for
    domain-specific rules that cannot be expressed with a regex or the other
    constraint options.
  </Dropdown>
</DropdownGroup>

## Validation pipeline order

The argument parser applies validations in a specific order for each argument. Understanding this order helps you design schemas that produce clear error messages, because the first failing check is what the user sees. The pipeline short-circuits on the first failure, so later checks only run if all earlier checks pass.

<Steps>
  <Step title="Presence check">
    If the argument is required and the raw token is missing or empty, the
    parser throws an *ArgumentException* immediately. Optional arguments with
    missing tokens receive `undefined` and skip all remaining checks. This is
    the first gate that prevents empty values from reaching type-specific
    parsers.
  </Step>
  <Step title="Custom validation">
    If a *validation.pattern* regex is defined, the raw input must match it. If
    a *validation.validate* function is defined, it receives the raw input and
    must return `true` or an error string. Both checks run on the raw text
    before any type coercion happens, so you can validate format constraints
    independently from type parsing.
  </Step>
  <Step title="Choices check">
    If a *choices* array is defined, the raw input must match one of the allowed
    values exactly. This acts as an enum constraint and is useful for argument
    values that must come from a fixed set like status names, action types, or
    mode selections.
  </Step>
  <Step title="Type coercion and bounds">
    The parser converts the raw token into the target type using the
    type-specific parser. For numeric types, bounds checks (*min*, *max*) run
    after successful coercion. For string types, length checks (*minLength*,
    *maxLength*) are applied. If coercion or bounds fail, the parser throws an
    *ArgumentException*.
  </Step>
</Steps>

## @PrefixArg parameter decorator

In the handler class, the _@PrefixArg_ function (without the namespace) is a parameter decorator that injects a parsed argument value by name. The name must match an argument declared in the command schema. The _@Ctx_ decorator injects the raw _Message_ object. Together, they let the handler receive fully typed parameters without manual parsing.

```typescript filename="src/modules/fun/handlers/roll-command.handler.ts"
import { Ctx, PrefixArg, PrefixCommandHandler } from '@spraxium/common';
import type { Message } from 'discord.js';
import { RollCommand } from '../commands/roll.command';

@PrefixCommandHandler(RollCommand)
export class RollHandler {
  async handle(
    @Ctx() message: Message,
    @PrefixArg('sides') sides: number | undefined,
    @PrefixArg('count') count: number | undefined,
  ): Promise<void> {
    const faces = sides ?? 6;
    const rolls = count ?? 1;
    const results = Array.from(
      { length: rolls },
      () => Math.floor(Math.random() * faces) + 1,
    );
    await message.reply(`Rolled ${rolls}d${faces}: [${results.join(', ')}]`);
  }
}
```

## @Cron and @Interval

The two repeating job types handle the vast majority of scheduling needs. _@Cron_ runs on calendar-based schedules using standard 5-field or 6-field cron expressions. _@Interval_ repeats at a fixed millisecond offset from the previous execution. Both decorators accept an options object for naming, disabling, and controlling whether the job runs immediately on startup.

The _name_ option is important because it is the key you use to pause, resume, or destroy the job at runtime through the _ScheduleService_. If you omit the name, the framework generates one automatically from the class and method name, but explicit names are easier to reference in admin commands and dashboards. The _disabled_ option lets you define a job in code but prevent it from starting, which is useful for feature flags.

The _@Cron_ decorator also accepts a _timezone_ option that overrides the global timezone from the schedule config. This is useful when a single bot serves communities in different time zones and you need a daily summary to go out at 9 AM local time in each region.

<CodeTabs>
  <Tab label="Cron job" language="typescript">
```typescript filename="src/modules/tasks/services/cleanup.service.ts"
import { Injectable } from '@spraxium/common';
import { Cron, CronExpression } from '@spraxium/schedule';

@Injectable()
export class CleanupService {
@Cron(CronExpression.EVERY_DAY_AT_MIDNIGHT, {
name: 'cleanup-expired-warnings', timezone: 'UTC', })
async cleanExpiredWarnings(): Promise<void> {
// Remove warnings older than 30 days
}
}

````
  </Tab>
  <Tab label="Interval job" language="typescript">
```typescript filename="src/modules/tasks/services/sync.service.ts"
import { Injectable } from '@spraxium/common';
import { Interval, IntervalExpression } from '@spraxium/schedule';

@Injectable()
export class SyncService {
  @Interval(IntervalExpression.EVERY_5_MINUTES, {
    name: 'sync-member-count', })
  async syncMemberCount(): Promise<void> {
    // Update cached member counts from the API
  }
}
````

## @Timeout and @AfterOnline

The two one-shot job types handle deferred initialization patterns. _@Timeout_ fires once after a fixed delay from application boot. _@AfterOnline_ fires once after the Discord client emits its _ready_ event plus an optional additional delay. Neither job repeats, they execute exactly once and then the framework removes them from the registry.

The key difference between _@Timeout_ and _@AfterOnline_ is the starting point of the delay. A timeout starts counting from the moment the application boots, which means the Discord client might not be connected yet. An after-online job starts counting from the _ready_ event, guaranteeing that the client, guilds, and channels are available when the method runs. Use _@AfterOnline_ whenever your job needs to call Discord API methods.

Both decorators accept the same _name_ and _disabled_ options. The _@Timeout_ decorator does not support _runOnInit_ because it is inherently a one-shot timer that always runs once. The delay value is in milliseconds for both decorators.

<CodeTabs>
  <Tab label="Timeout" language="typescript">
```typescript filename="src/modules/tasks/services/init.service.ts"
import { Injectable } from '@spraxium/common';
import { Timeout, TimeoutExpression } from '@spraxium/schedule';

@Injectable()
export class InitService {
@Timeout(TimeoutExpression.AFTER_30_SECONDS, {
name: 'deferred-cache-warmup', })
async warmCache(): Promise<void> {
// Populate caches 30 seconds after boot
}
}

````
  </Tab>
  <Tab label="After online" language="typescript">
```typescript filename="src/modules/tasks/services/welcome.service.ts"
import { Injectable } from '@spraxium/common';
import { AfterOnline } from '@spraxium/schedule';

@Injectable()
export class WelcomeService {
  @AfterOnline(5000, {
    name: 'post-online-announcement', })
  async announceOnline(): Promise<void> {
    // Send "bot is online" message 5 seconds after ready
  }
}
````

</CodeTabs>

## Enum helpers

The schedule module provides three enum-like objects with named constants for common timing values. Using these constants makes your code more readable than raw numbers and cron strings. Each helper covers the most frequent values you would use in a Discord bot context.

_CronExpression_ provides over 45 presets like `EVERY_DAY_AT_MIDNIGHT`, `EVERY_HOUR`, `EVERY_MONDAY_AT_MIDNIGHT`, and `EVERY_5_MINUTES`. _IntervalExpression_ provides over 25 presets in milliseconds like `EVERY_MINUTE`, `EVERY_5_MINUTES`, and `EVERY_HOUR`. _TimeoutExpression_ provides over 20 presets like `AFTER_10_SECONDS`, `AFTER_5_MINUTES`, and `AFTER_1_HOUR`.

You can always pass raw values instead of enum constants. The enum helpers are just convenience re-exports that improve readability and prevent typos in cron expressions.

```typescript filename="src/modules/tasks/services/example.service.ts"
import { Injectable } from "@spraxium/common";
import {
  Cron,
  CronExpression,
  Interval,
  IntervalExpression,
} from "@spraxium/schedule";

@Injectable()
export class ExampleService {
  @Cron(CronExpression.EVERY_WEEKDAY)
  async weekdayReport(): Promise<void> {}

  @Cron("0 */6 * * *")
  async customCron(): Promise<void> {}

  @Interval(IntervalExpression.EVERY_30_SECONDS)
  async frequentPoll(): Promise<void> {}
}
```

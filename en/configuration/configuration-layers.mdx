## The config file

Every Spraxium project has a `spraxium.config.ts` file at its root. This file is the single source of truth for all runtime behavior: which plugins are active, how commands match, how errors are presented, and how the local dev watcher behaves. The framework reads it once at startup and stores the result as an immutable snapshot before any module begins running.

You write the file using the _defineConfig_ helper exported from `@spraxium/core`. It accepts either a plain object or a callback that receives a _ConfigEnv_ argument. Both forms produce a valid config, but the callback form is recommended whenever options need to differ between environments, as it makes all branching explicit in one place.

```typescript filename="spraxium.config.ts"
import { defineConfig } from "@spraxium/core";
import { defineSchedule } from "@spraxium/schedule";

export default defineConfig((env) => ({
  debug: env.isNeutral,
  plugins: [
    defineSchedule({ timezone: "America/Sao_Paulo", disableAll: false }),
  ],
  dev: {
    entrypoint: "src/main.ts",
    debounce: 500,
  },
}));
```

## Callback vs object form

The callback form receives a _ConfigEnv_ object built from a single read of `NODE_ENV` at startup. It exposes _isDev_, _isNeutral_, and _isProd_ so you can branch on environment without checking raw strings. The object form is suitable for projects where configuration is identical in all environments.

<CodeTabs>
  <Tab label="Callback" language="typescript">
```typescript filename="spraxium.config.ts"
import { defineConfig } from '@spraxium/core';

export default defineConfig((env) => ({
  debug: env.isNeutral, logger: { maskTokens: true, commandLogging: env.isDev }, }));
```
  </Tab>
  <Tab label="Object" language="typescript">
```typescript filename="spraxium.config.ts"
import { defineConfig } from '@spraxium/core';

export default defineConfig({
  debug: false, logger: { maskTokens: true }, });
```
  </Tab>
</CodeTabs>

## Startup sequence

The framework completes these steps before any application module runs.

<Steps>
  <Step title="Config file discovery">
    The framework scans the project root for `spraxium.config.ts`, then
    `spraxium.config.js`, then `spraxium.config.mjs`. In production mode,
    compiled artifacts in `dist/` and `.spraxium/dist/` are checked first. The
    first file that exists on disk wins, and the search stops immediately.
  </Step>
  <Step title="Environment resolution and evaluation">
    `NODE_ENV` is read once and mapped to a *ConfigEnv* object. If your export
    is a function, it is called with this object and the return value is used.
    If it is a plain object, it is used directly. The result is stored as an
    immutable singleton for the rest of the process lifetime.
  </Step>
  <Step title="Module and plugin initialization">
    The framework resolves modules, providers, and lifecycle hooks. Each plugin
    reads its own config slice from the singleton using its factory as a lookup
    key. No further reads of the config file occur after this point.
  </Step>
</Steps>

<Callout tone="info" title="No config file is not an error">
  If no config file is found at startup, the framework continues with all
  defaults applied and logs a debug message. This is the expected state during
  initial project setup, before the config file is created.
</Callout>

## One concern per block

Each top-level key in the config object is owned by one subsystem. Plugin options live inside their respective factory calls. Command matching policy lives in the _prefix_ block. Exception rendering lives in the _exceptions_ block. Keeping each block focused means that modifying one area never creates side effects in another, and narrowing down an incident during a production outage stays fast.

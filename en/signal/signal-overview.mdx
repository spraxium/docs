## Why Signal exists in bot architectures

_Signal_ was designed for teams that run bots in real operations where multiple external services need to inform the bot about domain changes without creating hard runtime dependencies. Instead of wiring each API, worker, or admin panel directly into your bot internals, you publish standardized events through Discord webhooks and let the bot consume them through one secure path. This keeps ownership boundaries clean because external systems focus on producing business events while the bot focuses on reacting to them.

The protocol uses Discord's own webhook infrastructure as the transport layer. External services send signed JSON envelopes to a private Discord channel, and the bot's Signal module listens for new messages on that channel. This design avoids exposing internal service APIs, avoids setting up custom transport servers, and leverages Discord's existing reliability and delivery guarantees. The separation between producer and consumer is the core reason Signal scales better than ad-hoc webhook handlers scattered across feature modules.

  is the bot-side receiver that validates and dispatches events.{" "}
  <code>@spraxium/signal-client</code>

</Callout>

## Architecture overview

The Signal ecosystem splits into a receiver (the bot) and one or more emitters (external services). The emitter builds a signed envelope, posts it as a webhook message to a private Discord channel, and the bot's listener picks it up. The receiver validates the envelope through a strict ten-rule pipeline before dispatching the event to a registered handler.

This architecture keeps the bot process clean. It never exposes incoming HTTP ports for event ingestion. Instead, it leverages Discord's existing _messageCreate_ event, which the bot already subscribes to. The Signal module simply adds a filter that identifies webhook messages from approved sources and routes them through the validation pipeline.

## The envelope contract

Every Signal message follows a single envelope shape. This standardized structure ensures that transport validation happens before any business code runs. The envelope is a JSON object with seven fields, each serving a specific role in the validation and dispatch pipeline.

## HMAC security model

The HMAC layer is the security boundary that tells the bot whether an envelope was produced by an authorized emitter and whether the content stayed intact in transit. The signature is computed by concatenating the _nonce_, _sentAt_, and _payload_ fields into a single string (`${nonce}.${sentAt}.${JSON.stringify(payload)}`), then hashing it with HMAC-SHA256 using a shared secret key.

On the bot side, the same computation is performed and the result is compared to the received signature using a timing-safe comparison function. If the values do not match, the envelope is rejected before any business logic executes. This protects against three categories of attacks: payload tampering in transit, replay with modified data, and envelopes forged by unauthorized services.

In practical terms, HMAC allows your team to trust automation coming from external dashboards, billing systems, or moderation backoffices without exposing internal service APIs to the bot process. The shared secret must be the same on both the emitter and receiver side, and it should never be committed to version control.

<Callout tone="warning" title="Shared secret must match exactly">
  The <code>hmacSecret</code> value in <code>defineSignal</code> on the bot side
  must be identical to the <code>hmacSecret</code> value in the{" "}
  <code>SignalClient</code> constructor. If they diverge even by one character,
  every envelope will be rejected as untrusted.
</Callout>

## Ten-rule validation pipeline

Before any domain handler executes, every incoming message passes through a strict ten-rule validation pipeline implemented in _SignalValidator_. Each rule is evaluated sequentially, and the first failure stops the pipeline immediately. This fail-fast design ensures that invalid messages consume minimal processing time.

## Full processing pipeline

After the validator approves an envelope, the _SignalProcessor_ continues with three additional steps before the domain handler executes. Understanding the complete pipeline helps you debug delivery issues and reason about the order of operations during incident analysis.

<Steps>
  <Step title="Ten-rule validation">
    The *SignalValidator* checks every incoming message against ten sequential
    rules. The first failure rejects the message and logs a diagnostic entry
    when debug mode is enabled. Only messages that pass all ten rules proceed to
    the next stage.
  </Step>
  <Step title="Nonce deduplication">
    The *NonceCache* checks whether this nonce was already seen within the last
    60 seconds. If it was, the message is treated as a replay and silently
    dropped. If it is new, the nonce is recorded with a timestamp. Expired
    entries are evicted automatically on the next check.
  </Step>
  <Step title="Handler resolution and routing">
    The *SignalRouter* looks up the handler registered for this event name. If
    no handler exists, the message is logged as unroutable and dropped. This
    should not happen because rule R6 already verified the event was registered,
    but the router check adds defense in depth.
  </Step>
  <Step title="Zod payload validation">
    If the resolved handler has a Zod schema attached via the *@OnSignal*
    options, the processor runs *safeParse* on the payload. If validation fails,
    the message is rejected with a diagnostic log. If no schema is configured,
    the raw payload is passed through unchanged.
  </Step>
  <Step title="Optional message deletion">
    If *deleteAfterProcessing* is enabled in the Signal config, the processor
    attempts to delete the webhook message from the Discord channel. Deletion
    failures are logged as warnings but do not block handler execution.
  </Step>
  <Step title="Handler execution">
    The processor invokes the resolved handler with the validated payload and
    the full envelope. Errors thrown by the handler are caught, logged, and do
    not crash the Signal pipeline. Other events continue to be processed
    normally.
  </Step>
</Steps>

## When to avoid Signal

If the producer and consumer live in the same module graph and you do not need asynchronous boundaries, direct service calls are simpler to reason about and easier to trace. Signal should be adopted when its operational guarantees, security model, and decoupling benefits are genuinely useful, not as a default for every internal interaction. For synchronous user-facing feedback loops where the caller needs an immediate response, HTTP endpoints or direct Discord API calls remain the better choice.

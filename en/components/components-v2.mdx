## What V2 containers are

Discord's Components V2 API replaces the traditional message layout model with a composable block
system. Instead of a message being a flat combination of text content, embeds, and component rows, V2 lets you combine text blocks, sections, media galleries, separators, and action rows inside a
single container element. The container becomes the top-level layout unit for the message.

Spraxium's V2 system wraps this API with the same decorator-handler architecture used across the
rest of _@spraxium/components_. You declare a container schema as a decorated class, annotate each
structural element with a property decorator, and render the whole container through _V2Service_ by
passing runtime data. The result is a strongly-typed, composable layout that separates structure
from the data flowing through it at render time.

The package provides two authoring styles for V2. The class-based schema pattern uses
_@V2Container_ plus property decorators and is best for reusable, data-driven layouts. The fluent
API is an imperative builder that chains calls on a _V2ContainerFluentBuilder_ instance and is best
for one-off or dynamically assembled messages. Both styles produce the same wire-format and can be
mixed within the same bot.

## When to use V2 containers

V2 containers are appropriate whenever the message needs structured layout that a plain embed cannot
express. Common use cases include leaderboards with variable-length entry lists, server dashboard
panels with an icon thumbnail and stat grid, profiles with an avatar alongside multiple fields, and
image galleries embedded in command responses. Any message that benefits from visual separation
between sections, or that would require multiple embeds to approximate, is a good candidate for a
V2 container.

<Callout type="warning">
  V2 containers require the MessageFlags.IsComponentsV2 flag when sending
  messages. Setting this flag means the message cannot simultaneously carry
  traditional embeds or a plain content string alongside the containers, design
  V2 messages as self-contained layouts, not as enhancements to existing
  content.
</Callout>

## Setting up V2Service

_V2Service_ is the rendering and construction service for all V2 work. It is automatically
available via injection once _ComponentsModule_ is registered in your module graph. No additional
configuration is needed beyond the standard _defineComponents_ plugin setup.

Inject _V2Service_ into any handler or service that needs to render V2 containers.

```typescript filename="src/handlers/my.handler.ts"
import { V2Service } from "@spraxium/components";

@SlashCommandHandler(MyCommand)
export class MyHandler {
  constructor(private readonly v2: V2Service) {}

  async handle(@Ctx() interaction: ChatInputCommandInteraction): Promise<void> {
    // ...
  }
}
```

## Class-based container schemas

The class-based pattern declares a container as an annotated class. Apply _@V2Container_ to the
class to mark it as a container and configure the accent color and spoiler options. Apply property
decorators to each field to register child components in declaration order. At render time, _V2Service.build_ reads the metadata, evaluates all factory functions with your runtime data, and
assembles the container.

The property names in the schema class do not matter at runtime. Only the metadata registered by
the decorators is read. Declare the property type as `never` to signal that the field only exists
as a metadata slot.

<CodeTabs>
  <Tab label="Schema" language="typescript">
```typescript filename="src/schemas/user-profile.container.ts"
import {
  V2Container, V2Separator, V2Section, V2Text, desc, } from '@spraxium/components';

export interface UserProfileData {
  username: string;
  userId: string;
  joinedAt: number;
  avatarUrl: string;
  bio: string;
}

@V2Container({ accentColor: 0x5865f2 })
export class UserProfileContainer {
@V2Text((data: UserProfileData) => desc().h2(`👤 ${data.username}`))
title!: never;

@V2Separator()
divider!: never;

@V2Section({
text: (data: UserProfileData) =>
[
`**ID:** \`${data.userId}\``, `**Joined:** <t:${Math.floor(data.joinedAt / 1000)}:R>`, `**Bio:** ${data.bio}`, ].join('\n'), thumbnail: {
url: (data: UserProfileData) => data.avatarUrl, description: 'User avatar', }, })
stats!: never;

@V2Separator({ divider: false })
spacer!: never;

@V2Text((data: UserProfileData) => desc().subtext(`Profile for ${data.username}`))
footer!: never;
}

````

```typescript filename="src/handlers/profile.handler.ts"
import { Ctx, SlashCommandHandler } from '@spraxium/common';
import { V2Service } from '@spraxium/components';
import { MessageFlags, type ChatInputCommandInteraction } from 'discord.js';
import { ProfileCommand } from '../commands/profile.command';
import { UserProfileContainer, type UserProfileData } from '../schemas/user-profile.container';

@SlashCommandHandler(ProfileCommand)
export class ProfileHandler {
  constructor(private readonly v2: V2Service) {}

  async handle(@Ctx() interaction: ChatInputCommandInteraction): Promise<void> {
    const user = interaction.user;

    const data: UserProfileData = {
      username: user.username, userId: user.id, joinedAt: user.createdTimestamp, avatarUrl: user.displayAvatarURL({ size: 128 }), bio: 'No bio set.', };

    const payload = this.v2.buildReply(UserProfileContainer, data);

    await interaction.reply({
      ...payload, flags: payload.flags | MessageFlags.Ephemeral, });
  }
}
````

## The fluent API

The fluent API is a chainable builder attached to _V2Service_. Call _v2.container()_ to open a
builder, chain child component calls, and call _.toReply()_ at the end to produce the payload
ready for _interaction.reply()_. Each chain method returns the same builder, so every operation
can be inlined.

The fluent API does not support property-level decorators or _@V2When_ conditions because there is
no class-level metadata to annotate. Use the class-based pattern when you need conditional
rendering or when you want to reuse the same layout in multiple handlers.

```typescript filename="src/handlers/greet.handler.ts"
import { Ctx, SlashCommandHandler } from "@spraxium/common";
import { V2Service, desc } from "@spraxium/components";
import { MessageFlags, type ChatInputCommandInteraction } from "discord.js";
import { GreetCommand } from "../commands/greet.command";

@SlashCommandHandler(GreetCommand)
export class GreetHandler {
  constructor(private readonly v2: V2Service) {}

  async handle(@Ctx() interaction: ChatInputCommandInteraction): Promise<void> {
    const user = interaction.user;

    const payload = this.v2
      .container({ accentColor: 0x57f287 })
      .text(desc().h2(`👋 Hello, ${user.username}!`))
      .sep()
      .section(
        [
          `**ID:** \`${user.id}\``,
          `**Account age:** <t:${Math.floor(user.createdTimestamp / 1000)}:R>`,
        ].join("\n"),
        this.v2.thumbnail(user.displayAvatarURL({ size: 128 }), {
          description: `${user.username} avatar`,
        }),
      )
      .sep({ divider: false })
      .text(desc().subtext("Powered by Spraxium V2"))
      .toReply();

    await interaction.reply({
      ...payload,
      flags: payload.flags | MessageFlags.Ephemeral,
    });
  }
}
```

## Dynamic children with @V2Dynamic

_@V2Dynamic_ is the solution when the number of child components depends on runtime data. Instead
of a static configuration, you provide a factory function that receives the data object and returns
an array of child specs created with the helper functions _v2text_, _v2sep_, _v2section_, _v2gallery_, and _v2row_. The engine walks this array and renders each spec in sequence.

A common pattern is to generate one or more text lines per data entry and insert separators between
them, stopping one separator short so the last entry isn't followed by a divider.

```typescript filename="src/schemas/leaderboard.container.ts"
import {
  V2Container,
  V2Dynamic,
  V2Separator,
  V2Text,
  desc,
  v2sep,
  v2text,
} from "@spraxium/components";

export interface LeaderboardEntry {
  username: string;
  score: number;
  rank: number;
}

export interface LeaderboardData {
  title: string;
  entries: Array<LeaderboardEntry>;
}

const MEDALS = ["🥇", "🥈", "🥉"];

@V2Container({ accentColor: 0xf1c40f })
export class LeaderboardContainer {
  @V2Text((data: LeaderboardData) => desc().h2(`🏆 ${data.title}`))
  title!: never;

  @V2Separator()
  divider!: never;

  @V2Dynamic((data: LeaderboardData) => {
    if (data.entries.length === 0) {
      return [v2text(desc().subtext("No entries yet."))];
    }

    return data.entries.flatMap((entry, index) => {
      const medal = MEDALS[index] ?? `**${entry.rank}.**`;
      return [
        v2text(
          `${medal} **${entry.username}**, ${entry.score.toLocaleString()} pts`,
        ),
        ...(index < data.entries.length - 1 ? [v2sep({ divider: false })] : []),
      ];
    });
  })
  entries!: never;

  @V2Separator({ divider: false })
  footer!: never;

  @V2Text((data: LeaderboardData) =>
    desc().subtext(`${data.entries.length} players ranked`),
  )
  summary!: never;
}
```

The handler passes the entry list as data and calls _buildReply_ without knowing how many rows will
be generated.

```typescript filename="src/handlers/leaderboard.handler.ts"
@SlashCommandHandler(LeaderboardCommand)
export class LeaderboardHandler {
  constructor(private readonly v2: V2Service) {}

  async handle(@Ctx() interaction: ChatInputCommandInteraction): Promise<void> {
    const entries: Array<LeaderboardEntry> = await fetchTopPlayers(10);

    const payload = this.v2.buildReply(LeaderboardContainer, {
      title: "Weekly Top Players",
      entries,
    });

    await interaction.reply(payload);
  }
}
```

## Conditional rendering with @V2When

_@V2When_ conditionally includes a child based on runtime data. It is applied as an additional
decorator _below_ the child decorator, which means it runs last and patches the existing child
definition with a guard predicate. If the predicate returns `false` for the given data, the child
is excluded from the container entirely, not rendered as empty.

_@V2When_ is stackable: multiple children in the same container can each carry independent
conditions. The order property assigned at registration time is preserved regardless of which
children are filtered out.

```typescript filename="src/schemas/member-card.container.ts"
import {
  V2Container,
  V2Separator,
  V2Text,
  V2When,
  desc,
} from "@spraxium/components";

export interface MemberCardData {
  username: string;
  boostLevel: number;
  isAdmin: boolean;
  bio: string;
}

@V2Container({ accentColor: 0x5865f2 })
export class MemberCardContainer {
  @V2Text((data: MemberCardData) => desc().h2(data.username))
  title!: never;

  @V2Separator()
  divider!: never;

  @V2When((data: MemberCardData) => data.isAdmin)
  @V2Text("⚙️ **Server Administrator**")
  adminBadge!: never;

  @V2When((data: MemberCardData) => data.boostLevel > 0)
  @V2Text(
    (data: MemberCardData) => `⭐ **Server Booster**, Level ${data.boostLevel}`,
  )
  boostBadge!: never;

  @V2Text((data: MemberCardData) => data.bio)
  bio!: never;
}
```

When `isAdmin` is `false`, the admin badge row is not rendered at all. The remaining children close
up naturally without any gap.

## Embedding action rows with @V2Row

_@V2Row_ places a button row or select menu row directly inside a V2 container. Pass your component
classes in the `components` array the same way you would pass them to _ButtonService_ or
_SelectService_. The service auto-detects whether the first class is a select menu and routes to
the correct builder automatically.

<CodeTabs>
  <Tab label="Button row" language="typescript">
```typescript filename="src/schemas/confirm-panel.container.ts"
import {
  Button, LinkButton, V2Container, V2Row, V2Separator, V2Text, desc, } from '@spraxium/components';

@Button({ customId: 'panel_confirm', label: 'Confirm', style: 'success', emoji: '✅' })
export class ConfirmButton {}

@Button({ customId: 'panel_cancel', label: 'Cancel', style: 'danger', emoji: '✖' })
export class CancelButton {}

@V2Container({ accentColor: 0xed4245 })
export class ConfirmPanelContainer {
@V2Text((data: { action: string }) => desc().h2(`Confirm: ${data.action}`))
title!: never;

@V2Text('This action cannot be undone. Are you sure?')
body!: never;

@V2Separator({ divider: false })
spacer!: never;

@V2Row({ components: [ConfirmButton, CancelButton] })
actions!: never;
}

````

## The DescriptionBuilder

The _desc()_ helper function returns a _DescriptionBuilder_ that chains Discord-flavoured Markdown
helpers. It is available in both class-based and fluent patterns anywhere a text content value is
accepted. The builder is a thin string accumulator; calling _.toString()_ returns the final string, which the V2 engine calls automatically before passing content to the discord.js builders.

## Building the reply payload

Both _buildReply_ and the fluent _.toReply()_ method return a `{ components, flags }` object.
Spreading this into _interaction.reply()_ sets `IsComponentsV2` automatically. You can merge
additional flags using bitwise OR.

```typescript filename="sending a V2 reply"
const payload = this.v2.buildReply(MyContainer, data);

await interaction.reply({
  ...payload,
  flags: payload.flags | MessageFlags.Ephemeral,
});
```

If you need to update an existing V2 message, use _interaction.editReply()_ or pass the payload
to _message.edit()_ directly. The flag is required on both the initial send and subsequent edits, because Discord identifies a message as a V2 message by the presence of that flag.

## What are components

The _@spraxium/components_ package provides a decorator-driven system for building interactive Discord UI elements. Buttons, select menus, modals, and embeds are all defined as decorated classes, wired through metadata, and dispatched automatically when users interact with your bot. The package also includes a flow context system that threads stateful data across multi-step interactions without manual state management.

Unlike raw discord.js builders where you construct every object imperatively, Spraxium components use class decorators to declare what a component looks like, separate handler decorators to define what happens when a user interacts, and injectable services to render components at runtime. This separation keeps component definitions reusable, handlers testable, and rendering decoupled from business logic.

The package ships five component families: buttons (static, link, and dynamic), select menus (string, user, role, mentionable, channel), modals (with 11 field types, validation, and caching), embeds (declarative schema-driven), and v2 containers (composable layout blocks). Each family follows the same decorator-handler-service pattern, so learning one transfers directly to the others.

## Plugin configuration

The module reads its configuration from a plugin entry produced by _defineComponents_. This factory accepts a _ComponentsConfig_ object that controls context storage, error messages, and per-component behavior. Registration follows the same pattern as every Spraxium plugin: call the factory, add the result to the _plugins_ array in `spraxium.config.ts`.

```typescript filename="config/components.config.ts"
import { defineComponents } from "@spraxium/components";

export const componentsPlugin = defineComponents({
  context: {
    storage: "memory",
    defaultTtl: 300,
  },
  errorMessages: {
    expired: "This interaction has expired.",
    restricted: "You are not allowed to use this component.",
  },
});
```

The _context.storage_ field selects the persistence backend for flow contexts. Four adapters are available: `'memory'` for development, `'filesystem'` for single-instance bots, `'redis'` for multi-instance deployments, and `'sqlite'` for lightweight persistence. The _defaultTtl_ sets the expiration in seconds for contexts that do not specify their own TTL.

## The decorator-handler pattern

Every interactive component in Spraxium follows a two-class architecture: one class defines what the component looks like, and a separate class defines what happens when the user interacts with it. This separation exists because Discord components are rendered once but can be clicked many times by different users. Keeping the visual definition apart from the interaction logic makes components reusable across multiple messages and handlers independently testable.

The component class carries a decorator like _@Button_, _@StringSelect_, or _@ModalComponent_ that stores metadata about appearance and identity. The handler class carries a corresponding decorator like _@ButtonHandler_, _@StringSelectHandler_, or _@ModalHandler_ that references the component class. At runtime, the dispatcher reads both sets of metadata, matches incoming interactions to the correct handler, and invokes the _handle_ method with injected parameters.

<CodeTabs>
  <Tab label="Component" language="typescript">
```typescript filename="src/components/confirm-btn.component.ts"
import { Button } from '@spraxium/components';

@Button({
customId: 'confirm_action', label: 'Confirm', style: 'success', })
export class ConfirmButton {}

````
  </Tab>
  <Tab label="Handler" language="typescript">
```typescript filename="src/components/confirm-btn.handler.ts"
import { ButtonHandler, Ctx } from '@spraxium/components';
import type { ButtonInteraction } from 'discord.js';
import { ConfirmButton } from './confirm-btn.component';

@ButtonHandler(ConfirmButton)
export class ConfirmButtonHandler {
  async handle(@Ctx() interaction: ButtonInteraction): Promise<void> {
    await interaction.reply({ content: 'Action confirmed.', ephemeral: true });
  }
}
````

## Module wiring

The _ComponentsModule_ is a global module that provides all component services (_ButtonService_, _SelectService_, _ModalService_, _EmbedService_, _V2Service_, _ContextService_). You do not import it directly. Add your handler classes to the _handlers_ array of your feature module and the services become injectable automatically. Component classes decorated with _@Button_, _@StringSelect_, _@ModalComponent_, or _@Embed_ are metadata-only and are never registered in any module.

```typescript filename="src/modules/settings/settings.module.ts"
import { Module } from "@spraxium/common";
import { ConfirmButtonHandler } from "./components/confirm-btn.handler";

@Module({
  handlers: [ConfirmButtonHandler],
})
export class SettingsModule {}
```

Only the handler class goes in the _handlers_ array. Component classes store their configuration as constructor-level metadata via `Reflect.defineMetadata`. The dispatcher reads that metadata directly from the class reference stored inside each handler's decorator, so no DI instantiation of component classes is ever needed.

## Bootstrap lifecycle

<Steps>
  <Step title="Instance scanning">
    When the module loader instantiates each provider and handler, an instance
    scanner registered by *ComponentsModule* pushes every instance into a
    *pendingInstances* buffer. This happens during the standard dependency
    injection phase, before any lifecycle hooks fire. The scanner collects all
    classes, not just component-related ones, because filtering happens in the
    next phase.
  </Step>
  <Step title="Configuration loading">
    The *ComponentLifecycle.onBoot* hook reads the plugin configuration from
    *ConfigStore* using the *defineComponents* factory. If configuration is
    present, it sets the dispatcher config and initializes the selected context
    storage adapter with the configured TTL.
  </Step>
  <Step title="Handler registration">
    The dispatcher drains the *pendingInstances* buffer and checks each class
    for handler metadata. Classes decorated with *@ButtonHandler*,
    *@SelectHandler*, or *@ModalHandler* are registered with their respective
    sub-dispatchers. The sub-dispatcher reads the linked component's metadata to
    extract the *customId* or *prefix* used for routing.
  </Step>
  <Step title="Event binding">
    If at least one handler was registered, the dispatcher attaches a listener
    to the Discord client's *interactionCreate* event. When an interaction
    arrives, the dispatcher parses the *customId*, identifies the correct
    sub-dispatcher, resolves the flow context if present, and invokes the
    handler's *handle* method with injected parameters.
  </Step>
</Steps>

## Custom ID anatomy

Every interactive Discord component carries a _customId_ string that identifies it when the user interacts. Spraxium uses a structured format to encode both the component identity and optional flow context into this string. The tilde character (`~`) separates the base ID from the context ID.

## Parameter injection

Handler _handle_ methods receive their arguments through parameter decorators. The dispatcher resolves each parameter at invocation time based on the decorator metadata attached to the method signature. This system is consistent across all component types, so the same decorators work in button handlers, select handlers, and modal handlers.

## Rendering components

Each component family has an injectable service that reads decorator metadata and produces discord.js builder objects. You inject the service into your command handlers, listeners, or other services and call its _build_ method to produce action rows ready for message replies.

```typescript filename="src/commands/settings.command-handler.ts"
import { Ctx, SlashCommandHandler } from "@spraxium/common";
import { ButtonService, ContextService } from "@spraxium/components";
import type { ChatInputCommandInteraction } from "discord.js";
import { ConfirmButton } from "../components/confirm-btn.component";
import { CancelButton } from "../components/cancel-btn.component";
import { SettingsCommand } from "./settings.command";

@SlashCommandHandler(SettingsCommand)
export class SettingsCommandHandler {
  constructor(
    private readonly buttons: ButtonService,
    private readonly context: ContextService,
  ) {}

  async handle(@Ctx() interaction: ChatInputCommandInteraction): Promise<void> {
    const ctx = await this.context.create({ guildId: interaction.guildId });
    const row = this.buttons.rowWithContext(ctx, ConfirmButton, CancelButton);

    await interaction.reply({
      content: "Are you sure you want to reset settings?",
      components: [row],
    });
  }
}
```

The _build_ method accepts one or more component classes and returns an _ActionRowBuilder_. The _rowWithContext_ variant attaches a context ID to every button in the row. For select menus, _SelectService.build_ handles both static and dynamic options. For modals, _ModalService.build_ constructs the full modal from schema metadata.

## Shared concepts across families

Several architectural patterns recur across buttons, selects, modals, and embeds. Understanding these shared concepts once means you can apply them to any component type without relearning the fundamentals. The decorator-handler separation, metadata-driven rendering, context threading, and parameter injection all work identically regardless of the component family.

The documentation in this category covers buttons, selects, embeds, and the context system in detail. Modals have their own dedicated category because their field system, validation pipeline, caching mechanism, and dynamic schema building introduce concepts that are unique to form-like interactions and warrant deeper coverage.

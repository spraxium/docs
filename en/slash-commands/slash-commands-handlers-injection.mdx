## The handler class

A handler class is a standalone class decorated with *@SlashCommandHandler* that contains the execution logic for a slash command. The decorator receives the command definition class as its first argument, creating a metadata link between the schema and the logic. The handler class is instantiated through the DI container during module loading, so constructor parameters are resolved automatically from the module providers.

The handler must expose a *handle* method. When an interaction arrives that matches the command name (and optionally the subcommand and group), the *SlashInvoker* calls this method after the guard pipeline passes. The parameter list of *handle* is built dynamically from metadata: *@Ctx()* injects the raw interaction, *@SlashOpt(name)* injects a resolved option value, and any parameter without a decorator receives `undefined`.

```typescript filename="handlers/ping.handler.ts"
import { Ctx, SlashCommandHandler, SlashOpt } from '@spraxium/common';
import type { ChatInputCommandInteraction } from 'discord.js';
import { PingCommand } from '../commands/ping.command';

@SlashCommandHandler(PingCommand)
export class PingHandler {
  async handle(
    @Ctx() interaction: ChatInputCommandInteraction,
    @SlashOpt('service') service: string | null,
  ): Promise<void> {
    const ws = interaction.client.ws.ping;
    if (service) {
      await interaction.reply(`Service: ${service}, Latency: ${ws}ms`);
    } else {
      await interaction.reply(`Pong! ${ws}ms`);
    }
  }
}
```

Spraxium exports a `SlashContext` type from `@spraxium/common` that is a direct alias for `ChatInputCommandInteraction`. Importing from `@spraxium/common` reduces the direct coupling to discord.js in your handler files. Both types are interchangeable; use whichever path fits your team's import organization.

## The @SlashCommandHandler decorator

The *@SlashCommandHandler* decorator accepts one required argument and one optional argument. The first argument is the *@SlashCommand*-decorated class that this handler belongs to. The second argument is an options object with *sub* and *group* properties for routing to specific subcommands.

The routing resolution follows a strict matching rule. When the binder receives an interaction, it extracts the command name, optional subcommand, and optional subcommand group from the Discord interaction object. The registry then finds the handler where all three values match exactly. This means a root handler (no *sub* or *group*) will never receive subcommand interactions, and a subcommand handler will never receive root interactions.

## Parameter injection with @Ctx and @SlashOpt

The *handle* method parameters are populated by the *SlashInvoker* at runtime. Two parameter decorators control what each position receives: *@Ctx()* injects the raw Discord interaction, and *@SlashOpt(name)* injects the resolved value for the named option. Parameter positions are independent, so you can place *@Ctx* at any index and mix it with *@SlashOpt* parameters in any order.

*@SlashOpt* resolves the option value by calling the appropriate discord.js getter based on the option type declared in the command definition. For a *STRING* option, it calls `interaction.options.getString(name)`. For a *USER* option, it calls `interaction.options.getUser(name)`. The mapping covers all nine option types.

<Callout tone="warning" title="@SlashOpt names must match the option definition exactly">
  The string passed to *@SlashOpt('name')* must be identical to the option name declared in the *@SlashOption* decorator on the command definition. If the names do not match, the framework cannot look up the option metadata and the parameter receives `null` at runtime.
  Double-check option names when renaming command options, or import the command class as a type reference to catch mismatches during code review.
</Callout>

## Dependency injection in handlers

Handler classes are instantiated through the *SpraxiumContainer*, so they participate fully in the framework DI system. You can declare constructor parameters for any provider registered in the module tree. The DI container resolves dependencies recursively, so a handler can depend on services that themselves depend on other services.

```typescript filename="handlers/locale.handler.ts"
import { Ctx, SlashCommandHandler, SlashOpt } from '@spraxium/common';
import { I18nService } from '@spraxium/i18n';
import type { ChatInputCommandInteraction } from 'discord.js';
import { LocaleCommand } from '../commands/locale.command';

@SlashCommandHandler(LocaleCommand)
export class LocaleHandler {
  constructor(private readonly i18n: I18nService) {}

  async handle(
    @Ctx() interaction: ChatInputCommandInteraction,
    @SlashOpt('language') language: string,
  ): Promise<void> {
    const userId = interaction.user.id;
    if (!this.i18n.has(language)) {
      const msg = await this.i18n.tUser(userId, 'commands.locale.invalid', {
        locale: language,
        available: this.i18n.locales().join(', '),
      });
      await interaction.reply({ content: msg, flags: 'Ephemeral' });
      return;
    }

    await this.i18n.setUserLocale(userId, language);
    const msg = this.i18n.t('commands.locale.set', language, { locale: language });
    await interaction.reply({ content: msg, flags: 'Ephemeral' });
  }
}
```

The *I18nService* is injected by the container because it is registered as a provider in the module tree. The handler does not need to import the service from the container manually; the framework reads the `design:paramtypes` metadata emitted by TypeScript and resolves the constructor parameters automatically.

## Design checklist for handlers

<Steps>
  <Step title="Match @SlashOpt names to command options exactly">
    The string passed to *@SlashOpt('name')* must be identical to the option name in the *@SlashOption* decorator on the command definition. A mismatch causes the invoker to fail the metadata lookup and inject `null` instead of the actual value. Use the same constant or import the command class to verify names match during code review.
  </Step>
  <Step title="Handle null for optional parameters">
    When a command option has `required: false`, the handler parameter will be `null` when the user omits the option. Always type optional parameters as `Type | null` and check for `null` before using the value. Required options are guaranteed by Discord to have a value, so they can be typed without `null`.
  </Step>
  <Step title="Respond within three seconds">
    Discord requires a response within three seconds of receiving the interaction. If your handler needs more time, call `interaction.deferReply()` immediately and then use `interaction.editReply()` when the work is done. Deferred replies show a "Bot is thinking..." indicator to the user.
  </Step>
  <Step title="Keep handler classes focused">
    Each handler should implement exactly one logical operation. When a command has subcommands, use separate handler classes for each subcommand instead of branching inside a single handler. This keeps files small, testable, and easy to navigate in the file tree.
  </Step>
  <Step title="Use constructor injection for shared logic">
    When multiple handlers share logic like validation, formatting, or database access, extract that logic into a service class and inject it through the constructor. The DI container handles the lifecycle, and the handlers stay thin.
  </Step>
</Steps>

## Why the exception layer matters

In Spraxium, exceptions are not just about interrupting control flow when something goes wrong. They are the mechanism that transforms both technical and domain failures into predictable user-facing behavior and operationally useful signals that help your team understand what happened and why. Without a structured exception layer, each handler ends up inventing its own error response style, which leads to inconsistent UX, noisy logs, and slower incident resolution across the board.

As your bot grows, failures stop being edge cases and become part of the normal product experience that users encounter regularly. Users will trigger permission denials, cooldown limits, invalid arguments, and unavailable resources as part of their normal interaction with the bot. A dedicated exception layer ensures that all of these scenarios produce responses that are consistent, meaningful, and easy to maintain as the bot evolves over time.

## Error as contract, not accident

Treating an exception as a contract means that each relevant failure has explicit intent behind it rather than being an afterthought or a generic catch-all response. Instead of returning vague error messages, you define exception classes and layouts that represent concrete scenarios such as missing permissions, cooldown violations, invalid arguments, or unavailable resources. This approach creates a stable vocabulary across the team where engineers can read code and immediately understand what failed, why it failed, and how the user should be informed.

The practical goal is predictability across your entire bot. The same failure type should always produce the same exception class, the same response format, and the same operational signal regardless of where in the codebase the error originates. When that contract holds consistently, debugging becomes faster, user experience stays coherent, and the team can reason about failures without reading every handler individually to understand error behavior.

## The SpraxiumException base class

Every exception in the Spraxium ecosystem extends `SpraxiumException`, which itself extends the native JavaScript *Error*. The base class adds structured fields that the pipeline uses for layout resolution, logging decisions, and response rendering. Understanding these fields is essential before creating custom exceptions or working with built-in ones, because each field controls a specific aspect of how the error is processed at runtime.

```typescript filename="shop.command.ts"
import { SpraxiumException } from '@spraxium/core';

throw new SpraxiumException({
  code: 'ORDER_LIMIT_EXCEEDED', message: 'You have exceeded the {{limit}} order limit for today.', props: { limit: 5, userId: interaction.user.id }, shouldReply: true, shouldLog: false,
});
```

For anything beyond a one-off throw, extend `SpraxiumException` into a named class so the exception carries a stable identity that layouts and monitoring rules can target. The constructor accepts the same options object and forwards it to `super`, letting each subclass own its code, default message, and typed props interface.

```typescript filename="order-limit.exception.ts"
import { SpraxiumException } from '@spraxium/core';

interface OrderLimitProps {
  limit: number;
  userId: string;
}

export class OrderLimitExceededException extends SpraxiumException {
  constructor(props: OrderLimitProps) {
    super({
      code: 'ORDER_LIMIT_EXCEEDED', message: 'You have exceeded the {{limit}} order limit for today.', props, });
  }
}

// Usage anywhere in domain code
throw new OrderLimitExceededException({ limit: 5, userId: interaction.user.id });
```

## Built-in exceptions

Spraxium ships with fifteen exception classes that cover the most common failure scenarios in Discord bots. Each class extends `SpraxiumException` with a predefined *code*, a default *message* with placeholder support, and typed *props* for layout rendering. You can throw any of these directly without creating custom classes, and they are all imported from *@spraxium/core*.

<DropdownGroup>
<Dropdown title="ArgumentException: INVALID_ARGUMENT">
  Reports an invalid argument with the expected versus received values. This exception is thrown by the argument parsing pipeline when a command argument fails type coercion or constraint validation. The *props* object accepts *expected* and *received* fields for placeholder interpolation.
</Dropdown>

<Dropdown title="CommandNotFoundException: COMMAND_NOT_FOUND">
  Signals that the requested command does not exist in the command registry. This exception is thrown internally when prefix or slash command lookup fails to find a matching handler. The *props* object accepts a *command* field with the unrecognized name.
</Dropdown>

<Dropdown title="ConfigurationException: CONFIGURATION_ERROR">
  Reports a configuration key error with a human-readable reason. This exception surfaces during bootstrap when a required config field is missing, has the wrong type, or conflicts with another field. The *props* object accepts *key* and *reason* fields.
</Dropdown>

<Dropdown title="CooldownException: COOLDOWN">
  Tells the user how many seconds to wait before retrying the command. This exception is thrown by the *CooldownGuard* or manually from handler code when rate-limiting logic determines the user must wait. The *props* object accepts a *seconds* field.
</Dropdown>

<Dropdown title="DMOnlyException: DM_ONLY">
  Restricts the command to direct message channels only. This exception is thrown by the *DMOnly* guard when a user attempts to run a DM-exclusive command inside a guild channel. No additional props are required beyond the default message.
</Dropdown>

<Dropdown title="GuardDeniedException: GUARD_DENIED">
  Signals that a guard blocked command execution with an optional reason. This is the generic denial exception used when a guard returns `false` and no more specific exception applies. The *props* object accepts a *reason* field describing why access was denied.
</Dropdown>

<Dropdown title="GuildOnlyException: GUILD_ONLY">
  Restricts the command to guild (server) channels only. This exception is thrown by the *GuildOnly* guard when a user attempts to run a server-only command from a direct message. No additional props are required beyond the default message.
</Dropdown>

<Dropdown title="InternalException: INTERNAL_ERROR">
  Provides a generic internal error with a safe user-facing message. Use this exception when an unexpected failure occurs and you want to hide technical details from the user while still logging the root cause. The *props* object accepts a *detail* field for internal diagnostics.
</Dropdown>

<Dropdown title="MaintenanceException: MAINTENANCE">
  Signals that a feature is temporarily under maintenance with an optional estimated time of arrival. Use this exception to gracefully disable features during deployments or planned downtime. The *props* object accepts *feature* and *eta* fields.
</Dropdown>

<Dropdown title="NsfwOnlyException: NSFW_ONLY">
  Restricts the command to channels marked as NSFW in the guild settings. This exception is thrown by the *NSFWGuard* when a user attempts to run an age-restricted command in a regular channel. No additional props are required beyond the default message.
</Dropdown>

<Dropdown title="OwnerOnlyException: OWNER_ONLY">
  Restricts the command to the bot owner or configured owner IDs. This exception is thrown by the *OwnerOnly* guard when a non-owner user attempts to run a privileged command. No additional props are required beyond the default message.
</Dropdown>

<Dropdown title="PermissionDeniedException: PERMISSION_DENIED">
  Reports that the user is missing a required Discord permission. This exception is thrown by the *PermissionGuard* or manually when access control checks fail. The *props* object accepts a *permission* field naming the missing permission.
</Dropdown>

<Dropdown title="PremiumOnlyException: PREMIUM_ONLY">
  Restricts the feature to premium subscribers with optional tier information. Use this exception when monetized features need a clear upgrade prompt. The *props* object accepts *tier* and *upgradeUrl* fields for guiding users to subscription options.
</Dropdown>

<Dropdown title="RateLimitException: RATE_LIMITED">
  Reports API rate limiting with retry timing and the affected route. This exception surfaces when Discord or an external API returns a 429 response. The *props* object accepts *retryAfter* (in seconds) and *route* fields for client-side retry logic.
</Dropdown>

<Dropdown title="ValidationException: VALIDATION">
  Reports an invalid field value with the field name and a reason message. Use this exception for domain-level validation failures that occur outside the argument parsing pipeline. The *props* object accepts *field* and *reason* fields for precise error reporting.
</Dropdown>
</DropdownGroup>

### Using built-in exceptions

The built-in exceptions are ready to use with zero configuration. Every class accepts an optional *props* object that populates placeholders in the default message and provides context for layout rendering. The example below shows three common scenarios you will encounter in most bots.

```typescript filename="shop.command.ts"
import {
  CooldownException, PermissionDeniedException, MaintenanceException,
} from '@spraxium/core';

// User must wait before retrying the command
throw new CooldownException({ seconds: 10 });

// User lacks the required permission
throw new PermissionDeniedException({ permission: 'MANAGE_MESSAGES' });

// Feature is temporarily disabled
throw new MaintenanceException({ feature: 'Leaderboard', eta: '2 hours' });
```

## Where exceptions enter runtime flow

In practice, an exception is raised inside domain code such as a service, guard, handler, or listener, and then resolved by the core pipeline that decides the rendering strategy and logging behavior. Understanding this flow is essential because it determines where your exception classes originate, how layouts are selected to present errors to users, and what kind of operational data gets recorded for diagnostics and monitoring.

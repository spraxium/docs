## Separando estrutura de execução

No Spraxium, prefix commands utilizam um padrão de duas classes. O decorator _@PrefixCommand_ marca uma classe como definição de comando que descreve o nome, aliases, schema de argumentos e metadados. O decorator _@PrefixCommandHandler_ marca uma classe separada que contém o método _handle_ com a lógica de execução real. Essa separação mantém o contrato do comando estável enquanto permite que você evolua o comportamento do handler independentemente.

A classe de comando atua como um container de metadados. Ela declara como o comando se chama, quais argumentos espera e como aparece na saída de help. A classe de handler é onde você injeta serviços, interage com o Discord e produz respostas. Essa divisão também habilita múltiplos handlers para o mesmo comando através do mecanismo de roteamento de subcomandos, onde cada subcomando recebe seu próprio handler dedicado.

Ambas as classes devem ser registradas no mesmo módulo. Classes de comando vão no array _commands_ e classes de handler vão no array _handlers_. Se apenas comandos forem registrados, nada executa porque não há handler para chamar. Se apenas handlers forem registrados, não há schema de comando para o registry indexar, então o parser não consegue resolver o nome do comando.

## Um comando simples

O prefix command mais simples precisa de uma classe de comando com um schema de argumentos e uma classe de handler com um método _handle_. A classe de comando usa decorators de método _@PrefixArg_ em um método _build_ para declarar argumentos posicionais. A classe de handler usa decorators de parâmetro _@PrefixArg_ no método _handle_ para receber os valores parseados.

<CodeTabs>
  <Tab label="Comando" language="typescript">
```typescript filename="src/modules/fun/commands/roll.command.ts"
import { PrefixArg, PrefixCommand } from '@spraxium/common';

@PrefixCommand({
  name: 'roll',
  aliases: ['r', 'dice'],
  description: 'Rola um ou mais dados.',
  category: 'Diversão',
  usage: '!roll [lados] [quantidade]',
  cooldown: 3,
})
export class RollCommand {
  @PrefixArg.Integer('sides', { required: false, min: 2, max: 1000 })
  @PrefixArg.Integer('count', { required: false, min: 1, max: 25 })
  build() {}
}

````
  </Tab>
  <Tab label="Handler" language="typescript">
```typescript filename="src/modules/fun/handlers/roll-command.handler.ts"
import { Ctx, PrefixArg, PrefixCommandHandler } from '@spraxium/common';
import type { Message } from 'discord.js';
import { RollCommand } from '../commands/roll.command';

@PrefixCommandHandler(RollCommand)
export class RollHandler {
  async handle(
    @Ctx() message: Message,
    @PrefixArg('sides') sides: number | undefined,
    @PrefixArg('count') count: number | undefined,
  ): Promise<void> {
    const faces = sides ?? 6;
    const rolls = count ?? 1;
    const results = Array.from({ length: rolls }, () =>
      Math.floor(Math.random() * faces) + 1, );
    const total = results.reduce((a, b) => a + b, 0);
    await message.reply(`Rolled ${rolls}d${faces}: [${results.join(', ')}] = ${total}`);
  }
}
````

  </Tab>
</CodeTabs>

Os decorators de método _@PrefixArg_ na classe de comando definem o schema de argumentos. Eles são empilhados de cima para baixo para determinar a ordem posicional: o primeiro decorator se torna a posição 0, o segundo a posição 1, e assim por diante. No handler, decorators de parâmetro _@PrefixArg('nome')_ injetam o valor parseado pelo nome. O decorator _@Ctx_ injeta o objeto _Message_ bruto.

## Roteamento de subcomandos

Para famílias de comandos como `!mod warn`, `!mod kick` e `!mod ban`, use _@PrefixSubcommand_ para definir cada rota na classe de comando e conectar cada rota a um handler dedicado via _@PrefixCommandHandler_ com a opção _subcommand_. O primeiro token argv após o nome do comando é comparado com os nomes de subcomandos registrados, e o framework roteia automaticamente.

<CodeTabs>
  <Tab label="Comando" language="typescript">
```typescript filename="src/modules/moderation/commands/mod.command.ts"
import { PrefixArg, PrefixCommand, PrefixSubcommand } from '@spraxium/common';

@PrefixCommand({
  name: 'mod',
  aliases: ['m'],
  description: 'Comandos de moderação.',
  category: 'Moderação',
  usage: '!mod <warn|kick|ban> <usuario> [motivo]',
})
export class ModCommand {
  @PrefixSubcommand({ name: 'warn', description: 'Avisar um usuário' })
  @PrefixArg.User('target')
  @PrefixArg.Rest('reason', { required: false })
  warn() {}

  @PrefixSubcommand({ name: 'kick', description: 'Expulsar um usuário' })
  @PrefixArg.User('target')
  @PrefixArg.Rest('reason', { required: false })
  kick() {}

  @PrefixSubcommand({ name: 'ban', description: 'Banir um usuário' })
  @PrefixArg.User('target')
  @PrefixArg.Rest('reason', { required: false })
  ban() {}
}

````
  </Tab>
  <Tab label="Handler Warn" language="typescript">
```typescript filename="src/modules/moderation/handlers/mod-warn-command.handler.ts"
import { Ctx, PrefixArg, PrefixCommandHandler, UseGuards } from '@spraxium/common';
import type { Message } from 'discord.js';
import { ModCommand } from '../commands/mod.command';
import { ModeratorGuard } from '../guards/moderator.guard';

@UseGuards(ModeratorGuard)
@PrefixCommandHandler(ModCommand, { subcommand: 'warn' })
export class ModWarnHandler {
  async handle(
    @Ctx() message: Message,
    @PrefixArg('target') userId: string,
    @PrefixArg('reason') reason: string | undefined,
  ): Promise<void> {
    const text = reason ?? 'Nenhum motivo informado';
    await message.reply(`Aviso para <@${userId}>. Motivo: ${text}`);
  }
}
````

  </Tab>
</CodeTabs>

Cada método de subcomando na classe de comando define seu próprio schema de argumentos. O subcomando _warn_ espera uma menção de usuário seguida de texto rest opcional, enquanto _kick_ e _ban_ seguem o mesmo formato. Cada subcomando recebe sua própria classe de handler, o que significa que você pode aplicar guards diferentes, injetar serviços distintos e tratar erros independentemente por subcomando.

## Configuração do @PrefixCommand

O decorator _@PrefixCommand_ aceita um objeto de configuração que controla como o comando aparece na saída de help, como é acionado e quais limites operacionais possui. Todos os campos exceto _name_ são opcionais, mas fornecer _description_, _usage_ e _category_ melhora significativamente a saída de qualquer implementação de comando help.

<DropdownGroup>
  <Dropdown title="name (string, obrigatório)">
    O acionador principal que os usuários digitam após o prefixo. Quando um usuário envia `!ping`, o parser remove o prefixo, lê `ping` e busca um comando com esse nome no registry. Todo comando precisa de um nome único; se dois comandos compartilharem o mesmo nome, o último registrado vai sobrescrever o primeiro.
  </Dropdown>
  <Dropdown title="aliases (string[])">
    Nomes alternativos de acionamento que resolvem para o mesmo comando e handler. Definir `aliases: ['p', 'latency']` faz com que `!p` e `!latency` funcionem de forma idêntica ao `!ping`. Aliases compartilham o mesmo pool de cooldown que o nome primário, então um usuário em cooldown por `!ping` também fica bloqueado para `!p`.
  </Dropdown>
  <Dropdown title="description (string)">
    Uma linha curta de texto descrevendo o que o comando faz, usada nos embeds de help. O framework não exibe esse texto automaticamente; ele está disponível no método *getCommandInfo* do dispatcher para que seu handler de help o formate. Manter as descrições concisas e consistentes entre os comandos melhora a qualidade dos embeds de ajuda do bot.
  </Dropdown>
  <Dropdown title="usage (string)">
    Uma string legível mostrando o formato de invocação esperado, como `!ban <usuario> [motivo]`. O framework armazena isso nos metadados do comando e o retorna como parte de *getCommandInfo*, mas não faz parse nem valida automaticamente. Use em embeds de help e nas mensagens de *ArgumentException* para que os usuários saibam a sintaxe correta quando o parsing falhar.
  </Dropdown>
  <Dropdown title="category (string)">
    Um label de agrupamento para organizar a saída de help. Comandos na mesma categoria são tipicamente exibidos juntos nos embeds de help. O framework não impõe categorias nem atribui comportamento especial a elas; são metadados puros que seu handler de help lê de *getCommandInfo*.
  </Dropdown>
  <Dropdown title="cooldown (number)">
    Um cooldown por comando em segundos que sobrescreve o *defaultCooldown* global do *PrefixConfig*. Quando um usuário executa o comando, o framework registra o timestamp e bloqueia execuções subsequentes até o cooldown expirar. Definir `cooldown: 0` desabilita o cooldown para esse comando independente do padrão global.
  </Dropdown>
</DropdownGroup>

## Registro no módulo

Tanto classes de comando quanto de handler devem ser registradas em um módulo para que o framework as descubra durante o bootstrap. A classe de comando vai no array _commands_ e a classe de handler vai no array _handlers_. O framework processa ambos os arrays durante o carregamento do módulo, registra os metadados do comando no _PrefixRegistry_ e conecta as instâncias de handler através do container de DI.

```typescript filename="src/modules/moderation/moderation.module.ts"
import { Module } from '@spraxium/common';
import { ModCommand } from './commands/mod.command';
import { RollCommand } from '../fun/commands/roll.command';
import { ModWarnHandler } from './handlers/mod-warn.handler';
import { ModKickHandler } from './handlers/mod-kick.handler';
import { ModBanHandler } from './handlers/mod-ban.handler';
import { RollHandler } from '../fun/handlers/roll.handler';

@Module({
  commands: [ModCommand, RollCommand],
  handlers: [ModWarnHandler, ModKickHandler, ModBanHandler, RollHandler],
})
export class ModerationModule {}
```

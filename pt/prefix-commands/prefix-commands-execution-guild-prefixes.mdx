## Precedência na resolução de prefixo

Quando o framework recebe uma mensagem, o _PrefixParser_ verifica se o texto começa com um prefixo reconhecido. A resolução segue uma ordem estrita de precedência: prefixos específicos de servidor são verificados primeiro, depois o prefixo global e, por fim, a menção do bot se habilitada. A primeira correspondência vence e as opções restantes são ignoradas.

Esse modelo de precedência significa que uma sobrescrita de servidor sempre tem prioridade sobre o padrão global. Se um servidor registrou `?` como seu prefixo e o config global usa `!`, mensagens começando com `?` são capturadas naquele servidor enquanto `!` é ignorado. Se não existir sobrescrita de servidor, o parser recorre ao prefixo global. O prefixo de menção (ex: `@Bot roll 6`) é sempre verificado por último como um fallback universal.

A ordem de resolução é importante para bots que atendem múltiplas comunidades com convenções diferentes. Um servidor focado em moderação pode preferir `.` como prefixo porque é discreto e silencioso, enquanto um servidor de jogos pode escolher `>>` pela visibilidade. O sistema de prefixos por servidor permite que cada comunidade defina sua própria convenção sem afetar as demais.

<Steps>
  <Step title="Verifica sobrescritas de servidor">
    O parser chama *PrefixGuildManager.getGuildPrefix* com o ID do servidor da mensagem recebida. Se o servidor tem um ou mais prefixos registrados, o parser testa a mensagem contra cada um em ordem. O primeiro prefixo correspondente é usado e o parser prossegue diretamente para a tokenização. Prefixos de servidor podem ser uma string única ou um array de strings.
  </Step>
  <Step title="Verifica prefixo global">
    Se nenhuma sobrescrita de servidor combinar (ou a mensagem for de DM), o parser testa a mensagem contra o prefixo global definido no *PrefixConfig*. O prefixo global também suporta arrays, então você pode definir múltiplos prefixos globais como `['!', '?']`. Se a mensagem começar com qualquer um dos prefixos globais, a primeira correspondência é usada.
  </Step>
  <Step title="Verifica prefixo de menção">
    Se *mentionPrefix* estiver habilitado (o padrão), o parser verifica se a mensagem começa com uma menção do bot como `<@BOT_ID>`. Isso funciona como um prefixo universal que sempre opera independente da configuração de servidor ou global. O prefixo de menção é útil quando usuários esquecem o prefixo configurado e precisam de um fallback confiável.
  </Step>
</Steps>

## Referência do PrefixConfig

A interface _PrefixConfig_ controla o comportamento global do subsistema de prefix. Você a passa para _definePrefix_ no arquivo de configuração. Todo campo exceto _prefix_ é opcional e possui um valor padrão adequado.

<DropdownGroup>
  <Dropdown title="prefix (string | string[], obrigatório)">
    A string de prefixo global ou um array de prefixos que o parser reconhecerá
    para todos os servidores sem sobrescrita customizada. Definir um array como
    `['!', '?']` permite que os usuários escolham entre múltiplos prefixos. Não
    há valor padrão; esse campo é obrigatório e deve ser fornecido em toda
    configuração.
  </Dropdown>
  <Dropdown title="guildPrefixProvider (PrefixGuildProvider)">
    Uma função assíncrona chamada na inicialização que retorna um array de
    objetos *PrefixGuildEntry* para pré-popular o manager de prefixos por
    servidor. Esta é a forma recomendada de hidratar sobrescritas por servidor a
    partir de um banco de dados no boot. Se omitida, o store em memória inicia
    vazio e precisa ser populado em runtime via chamadas diretas à API.
  </Dropdown>
  <Dropdown title="caseSensitive (boolean, padrão false)">
    Controla se a correspondência de string de prefixo diferencia maiúsculas e
    minúsculas. Quando `false` (o padrão), `!Ping` e `!ping` são tratados de
    forma idêntica. Ative isso apenas se houver uma razão deliberada para
    distinguir caracteres de acionamento em maiúsculas e minúsculas.
  </Dropdown>
  <Dropdown title="mentionPrefix (boolean, padrão true)">
    Quando `true`, uma menção ao bot no início de uma mensagem é tratada como um
    prefixo válido. Isso permite que usuários invoquem comandos com `@NomeBot
    roll 6` independentemente do prefixo configurado, fornecendo um fallback
    confiável quando os usuários esquecem o prefixo ativo. Desative apenas se
    comandos acionados por menção conflitarem com outro comportamento do bot.
  </Dropdown>
  <Dropdown title="defaultCooldown (number, padrão 0)">
    Um cooldown global em segundos aplicado a todo comando que não define seu
    próprio campo *cooldown*. Definir esse valor como `3` significa que todos os
    comandos têm um cooldown de 3 segundos, salvo sobrescrita específica. Um
    valor de `0` desabilita o cooldown padrão completamente.
  </Dropdown>
  <Dropdown title="replyOnNotFound (boolean, padrão false)">
    Quando `true`, o framework envia automaticamente uma resposta quando um
    usuário digita um prefixo reconhecido seguido de um nome de comando
    desconhecido. Isso reduz o atrito para usuários que erram a digitação dos
    comandos. Quando `false`, comandos não reconhecidos são ignorados
    silenciosamente.
  </Dropdown>
</DropdownGroup>

```typescript filename="config/prefix.config.ts"
import { definePrefix } from '@spraxium/core';
import { loadGuildPrefixes } from './prefix/guild-prefix.provider';

export default definePrefix({
  prefix: '!',
  caseSensitive: false,
  mentionPrefix: true,
  defaultCooldown: 3,
  replyOnNotFound: true,
  guildPrefixProvider: loadGuildPrefixes,
});
```

<Callout tone="info">
  O arquivo de config deve declarar somente política global e hidratação inicial
  do subsistema de prefix. Lógica de leitura, escrita, auditoria e sincronização
  de prefixos por servidor deve ficar em services que injetam
  *PrefixDispatcher*. A página *API de Prefixos via Services* mostra esse padrão
  completo em cenários de produção.
</Callout>

## Gerenciando prefixos de servidor em runtime

O _PrefixGuildManager_ é acessível através do _PrefixDispatcher_ e fornece uma API completa para leitura, escrita e remoção de sobrescritas de prefixo por servidor. Alterações entram em vigor imediatamente para a próxima mensagem recebida sem necessidade de reiniciar. Isso permite construir comandos de administração ou integrações de dashboard que atualizam prefixos dinamicamente.

Cada entrada de servidor armazena o ID do servidor, um ou mais prefixos e um registro de metadados opcional. O campo de metadados não é utilizado pelo framework em si, mas oferece um lugar para armazenar contexto como quem alterou o prefixo e quando. Isso é útil para logs de auditoria em bots de moderação.

O método _loadFromProvider_ é chamado automaticamente na inicialização se você definir um _guildPrefixProvider_ no config. Você também pode chamá-lo manualmente para recarregar prefixos de servidor do banco de dados a qualquer momento, por exemplo após uma importação em massa ou migração.

<DropdownGroup>
  <Dropdown title="getGuildPrefix(guildId)">
    Retorna o array de strings de prefixo ativas para o servidor, ou `undefined`
    se não houver sobrescrita. É o método de leitura principal para verificar o
    prefixo atual antes de exibi-lo ao usuário ou construir conteúdo de help. A
    chamada é síncrona e lê diretamente do store em memória.
  </Dropdown>
  <Dropdown title="getGuildEntry(guildId)">
    Retorna o objeto *PrefixGuildEntry* completo para o servidor, incluindo
    `guildId`, `prefix` e qualquer `metadata` armazenado. Use quando você
    precisa dos metadados além do valor do prefixo, por exemplo para exibir quem
    o alterou por último. Retorna `undefined` se não houver sobrescrita.
  </Dropdown>
  <Dropdown title="hasGuildPrefix(guildId)">
    Retorna `true` se o servidor tem uma sobrescrita de prefixo ativa, `false`
    caso contrário. Use para verificações de existência quando você só precisa
    bifurcar na presença sem ler o valor do prefixo. É mais expressivo do que
    comparar *getGuildPrefix* contra `undefined`.
  </Dropdown>
  <Dropdown title="getAllGuildIds()">
    Retorna um array de todos os IDs de servidores que atualmente possuem
    sobrescritas de prefixo registradas. Use em dashboards administrativos,
    rotinas de sincronização ou ferramentas de diagnóstico que precisam enumerar
    todos os servidores configurados. O resultado reflete o estado ativo em
    memória.
  </Dropdown>
  <Dropdown title="setGuildPrefix(guildId, prefix, metadata?)">
    Cria uma nova entrada de prefixo para o servidor ou substitui inteiramente
    uma existente. O objeto de metadados opcional armazena campos de auditoria
    como ID do ator e timestamp. Sempre persista a alteração no banco de dados
    antes de chamar esse método para evitar divergência em caso de
    reinicialização do processo.
  </Dropdown>
  <Dropdown title="updateGuildPrefix(guildId, prefix, metadata?)">
    Mescla parcialmente os valores fornecidos na entrada existente em vez de
    substituí-la. Se o servidor não tiver entrada existente, comporta-se como
    *setGuildPrefix*. Use quando quiser atualizar apenas o prefixo ou apenas os
    metadados sem sobrescrever o outro campo.
  </Dropdown>
  <Dropdown title="removeGuildPrefix(guildId)">
    Exclui a sobrescrita em memória para o servidor e retorna `true` se existia
    uma entrada ou `false` caso contrário. Após a remoção, o parser usa o
    prefixo global para esse servidor. Siga o padrão persist-first e remova do
    banco de dados antes de chamar esse método.
  </Dropdown>
  <Dropdown title="clear()">
    Remove todas as sobrescritas de prefixo de servidor do store em memória de
    uma vez. É útil principalmente durante a limpeza de testes ou resets
    controlados. Em produção, coordene um reload completo com *loadFromProvider*
    para restaurar o estado conhecido.
  </Dropdown>
  <Dropdown title="loadFromProvider(provider)">
    Aceita uma função provider assíncrona, a aguarda e substitui todas as
    entradas de prefixo em memória pelo array retornado. É chamado
    automaticamente no boot se você definir um *guildPrefixProvider* no config,
    mas também pode ser chamado manualmente para recarregar do banco após uma
    importação em lote.
  </Dropdown>
  <Dropdown title="size (propriedade)">
    Propriedade numérica que reflete a contagem total de entradas de prefixo por
    servidor atualmente em memória. Use em health checks ou endpoints de
    diagnóstico para confirmar se os dados de servidor foram hidratados. Um
    valor de `0` após o boot geralmente significa que o provider não retornou
    entradas ou não foi configurado.
  </Dropdown>
</DropdownGroup>

## Exemplo de mutação de prefixo em runtime

Um padrão comum é construir um comando administrativo que permite que donos de servidor alterem seu prefixo através de um comando por mensagem. O handler injeta o _PrefixDispatcher_ para acessar o manager _guildPrefixes_ e persiste a alteração no banco de dados. A atualização em runtime entra em vigor imediatamente para mensagens subsequentes.

<CodeTabs>
  <Tab label="Comando" language="typescript">
```typescript filename="src/modules/admin/commands/set-prefix.command.ts"
import { PrefixArg, PrefixCommand } from '@spraxium/common';

@PrefixCommand({ name: 'set-prefix' })
export class SetPrefixCommand {
@PrefixArg.String('newPrefix', { minLength: 1, maxLength: 5 })
build() {}
}

````
  </Tab>
  <Tab label="Handler" language="typescript">
```typescript filename="src/modules/admin/handlers/set-prefix-command.handler.ts"
import { Ctx, PrefixArg, PrefixCommandHandler } from '@spraxium/common';
import { PrefixDispatcher } from '@spraxium/core';
import type { Message } from 'discord.js';
import { SetPrefixCommand } from '../commands/set-prefix.command';

@PrefixCommandHandler(SetPrefixCommand)
export class SetPrefixHandler {
  constructor(private readonly dispatcher: PrefixDispatcher) {}

  async handle(
    @Ctx() message: Message,
    @PrefixArg('newPrefix') newPrefix: string,
  ): Promise<void> {
    const guildId = message.guildId;
    if (!guildId) {
      await message.reply('Este comando só pode ser usado em um servidor.');
      return;
    }

    this.dispatcher.guildPrefixes.setGuildPrefix(guildId, newPrefix, {
      setBy: message.author.id,
      setAt: new Date().toISOString(),
    });

    await message.reply(`Prefixo atualizado para \`${newPrefix}\` neste servidor.`);
  }
}
````

  </Tab>
</CodeTabs>

## Sistema de cooldown

O _PrefixCooldownManager_ aplica cooldowns por usuário e por comando. Quando um comando define um valor _cooldown_ no decorator _@PrefixCommand_ (ou usa o _defaultCooldown_ global do config), o manager rastreia o timestamp da última execução para cada par usuário-comando. Se um usuário tentar executar um comando antes do cooldown expirar, o framework bloqueia a execução e pode opcionalmente responder com o tempo restante.

Cooldowns são armazenados em memória e resetam quando o bot reinicia. O método _check_ retorna os segundos restantes (zero se o cooldown expirou), o método _set_ registra uma nova entrada de cooldown e o método _clear_ remove todos os cooldowns rastreados. Para a maioria dos bots, o campo _cooldown_ no decorator e o _defaultCooldown_ global lidam com tudo automaticamente sem chamadas diretas à API.

<Callout tone="info">
  O campo *cooldown* por comando no *@PrefixCommand* sobrescreve o
  *defaultCooldown* global. Se um comando define cooldown: 5 e o padrão global é
  3, aquele comando usa um cooldown de 5 segundos.
</Callout>

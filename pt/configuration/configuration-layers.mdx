## O arquivo de configuração

Todo projeto Spraxium tem um arquivo `spraxium.config.ts` na raiz. Esse arquivo é a fonte canônica para todo o comportamento de runtime: quais plugins estão ativos, como os comandos fazem matching, como os erros são apresentados e como o watcher de dev se comporta. O framework o lê uma vez na inicialização e armazena o resultado como um snapshot imutável antes de qualquer módulo começar a executar.

Você escreve o arquivo usando o helper *defineConfig* exportado de `@spraxium/core`. Ele aceita um objeto simples ou um callback que recebe um argumento *ConfigEnv*. Ambas as formas produzem uma configuração válida. O padrão de callback é recomendado sempre que as opções precisam variar entre ambientes, pois centraliza o branching por modo em um único lugar.

```typescript filename="spraxium.config.ts"
import { defineConfig } from '@spraxium/core';
import { defineSchedule } from '@spraxium/schedule';

export default defineConfig((env) => ({
  debug: env.isNeutral, plugins: [
    defineSchedule({ timezone: 'America/Sao_Paulo', disableAll: false }), ], dev: {
    entrypoint: 'src/main.ts', debounce: 500, }, }));
```

## Callback vs objeto

O padrão de callback recebe um objeto *ConfigEnv* construído a partir de uma única leitura do `NODE_ENV` na inicialização. Ele expõe *isDev*, *isNeutral* e *isProd* para que você possa ramificar por ambiente sem verificar strings brutas. O padrão de objeto é adequado para projetos onde a configuração é idêntica em todos os ambientes.

<CodeTabs>
  <Tab label="Callback" language="typescript">
```typescript filename="spraxium.config.ts"
import { defineConfig } from '@spraxium/core';

export default defineConfig((env) => ({
  debug: env.isNeutral, logger: { maskTokens: true, commandLogging: env.isDev }, }));
```
  </Tab>
  <Tab label="Objeto" language="typescript">
```typescript filename="spraxium.config.ts"
import { defineConfig } from '@spraxium/core';

export default defineConfig({
  debug: false, logger: { maskTokens: true }, });
```
  </Tab>
</CodeTabs>

## Sequência de inicialização

O framework conclui estas etapas antes de qualquer módulo da aplicação executar.

<Steps>
  <Step title="Descoberta do arquivo de configuração">
    O framework varre a raiz do projeto em busca de `spraxium.config.ts`, depois `spraxium.config.js`, depois `spraxium.config.mjs`. No modo de produção, artefatos compilados em `dist/` e `.spraxium/dist/` são verificados primeiro. O primeiro arquivo que existir vence e a busca é encerrada imediatamente.
  </Step>
  <Step title="Resolução de ambiente e avaliação">
    O `NODE_ENV` é lido uma vez e mapeado para um objeto *ConfigEnv*. Se o export é uma função, ela é chamada com esse objeto e o valor retornado é usado. Se é um objeto simples, é usado diretamente. O resultado é armazenado como singleton imutável pelo restante do processo.
  </Step>
  <Step title="Inicialização de módulos e plugins">
    O framework resolve módulos, providers e hooks de ciclo de vida. Cada plugin lê sua própria fatia de configuração do singleton usando sua factory como chave de lookup. Nenhuma leitura adicional do arquivo de configuração ocorre após esse ponto.
  </Step>
</Steps>

<Callout tone="info" title="Nenhum arquivo de configuração não é um erro">
  Se nenhum arquivo de configuração for encontrado na inicialização, o framework continua com todos os defaults aplicados e registra uma mensagem de debug.
  Esse é o comportamento esperado durante a configuração inicial do projeto, antes de o arquivo ser criado.
</Callout>

## Uma responsabilidade por bloco

Cada chave de topo no objeto de configuração pertence a um subsistema. As opções de plugin ficam dentro das chamadas de factory correspondentes. A política de matching de comandos fica no bloco *prefix*. A apresentação de exceptions fica no bloco *exceptions*. Manter cada bloco focado significa que modificar uma área nunca cria efeitos colaterais em outra, e isolar um incidente em produção permanece rápido.

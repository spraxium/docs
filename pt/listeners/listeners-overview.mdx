## O que e um listener

Um listener e uma classe que reage a eventos do Discord. Quando algo acontece no Discord, como um usuario enviando uma mensagem, um membro entrando em um servidor, ou o bot terminando seu login, o Discord.js dispara um evento. Uma classe listener no Spraxium captura esse evento e roda seu codigo em resposta. Sem listeners, seu bot nao conseguiria responder a nada que acontece fora de comandos explicitos.

A diferenca chave entre listeners e comandos e o que os aciona. Comandos sao acionados por um usuario digitando um slash command ou um prefix command. Listeners sao acionados por eventos do Discord que acontecem em background, como mensagens sendo enviadas, membros entrando ou saindo, reacoes sendo adicionadas, ou o client ficando pronto. Alguns desses eventos envolvem interacoes de usuario que podem receber respostas, e outros sao eventos de background que acontecem silenciosamente.

Listeners no Spraxium nao sao callbacks crus. Sao classes completas gerenciadas pelo container de injecao de dependencia, registradas em modulos e conectadas ao client do Discord automaticamente durante o bootstrap. Isso significa que suas classes listener podem receber servicos injetados, participar do pipeline de guards pra eventos suportados e se beneficiar do mesmo tratamento de exceptions que comandos usam.

## Um exemplo basico de listener

Aqui estao dois listeners que mostram os dois modos de binding. O primeiro usa *@Once* pra reagir ao *ClientReady* uma unica vez quando o bot conecta. O segundo usa *@On* pra reagir a todo evento *MessageCreate* durante toda a vida do bot. Juntos, eles cobrem os dois padroes mais comuns que voce vai usar.

<CodeTabs>
  <Tab label="Ready (Once)" language="typescript">
```typescript filename="src/events/listeners/ready.listener.ts"
import { Events, Listener, Once } from '@spraxium/common';
import { Logger } from '@spraxium/core';
import type { Client } from 'discord.js';

@Listener(Events.ClientReady)
export class ReadyListener {
  private logger: Logger = new Logger(ReadyListener.name);

  @Once()
  onReady(client: Client<true>): void {
    this.logger.info(`Client is ready as ${client.user.tag}`);
  }
}
```
  </Tab>
  <Tab label="Mensagens (On)" language="typescript">
```typescript filename="src/events/listeners/message-create.listener.ts"
import { Events, Listener, On } from '@spraxium/common';
import type { Message } from 'discord.js';

@Listener(Events.MessageCreate)
export class MessageCreateListener {
  @On()
  async onMessage(message: Message): Promise<void> {
    if (message.author.bot) return;
    if (message.content === '!ping') {
      await message.reply('pong');
    }
  }
}
```
  </Tab>
</CodeTabs>

Note que o parametro do metodo recebe o payload do evento do Discord.js diretamente. Pra *ClientReady*, o payload e o objeto *Client*. Pra *MessageCreate*, o payload e o objeto *Message*. Cada evento do Discord tem seu proprio tipo de payload, e seu metodo recebe o que o Discord.js fornece pra aquele evento especifico.

## Registrando listeners em um modulo

Listeners nao se ativam apenas por existirem no seu codigo fonte. Voce precisa registra-los no array *listeners* de um modulo pra que o framework saiba sobre eles e possa conecta-los ao client do Discord durante o bootstrap. Sem esse registro, sua classe listener sera completamente ignorada.

```typescript filename="src/events/events.module.ts"
import { Module } from '@spraxium/common';
import { ReadyListener } from './listeners/ready.listener';
import { MessageCreateListener } from './listeners/message-create.listener';

@Module({
  listeners: [ReadyListener, MessageCreateListener],
})
export class EventsModule {}
```

O modulo em si deve ser importado pelo seu modulo raiz ou outro modulo na cadeia de imports. O framework processa modulos durante o bootstrap, resolve os arrays de listeners, instancia cada classe listener atraves do container de DI e entao os conecta ao client do Discord. Se um listener depende de um servico, esse servico deve estar disponivel no escopo de providers do modulo ou exportado de um modulo importado, assim como qualquer outra dependencia injetavel.

## Como o bootstrap conecta listeners

O processo de bootstrap trata listeners em uma sequencia especifica que importa pra entender quando seu listener fica ativo. Primeiro, o framework le o array *listeners* de cada modulo. Segundo, instancia cada classe listener pelo container de DI, resolvendo dependencias de construtor. Terceiro, o *ListenerRegistry* interno escaneia cada instancia procurando metodos decorados com *@On* ou *@Once* e os agrupa por nome de evento. Quarto, o *ListenerBinder* pega esses grupos e chama o metodo *on* ou *once* do client Discord.js pra criar os bindings reais. Por fim, quando um evento do Discord dispara, o *ListenerInvoker* chama cada handler registrado pra aquele evento.

Essa estrutura mantem todas as classes listener em uma pasta dedicada e o arquivo de modulo na raiz de events. Conforme seu bot cresce, voce pode adicionar mais arquivos de listener sem mudar a estrutura de pastas. Cada listener trata um evento, e o arquivo de modulo e o unico lugar que declara quais listeners estao ativos.

<Callout tone="info" title="Uma classe por evento">
  Cada classe listener e vinculada a exatamente um evento do Discord atraves do decorator Listener. Se voce precisa ouvir multiplos eventos, crie classes listener separadas pra cada um. Isso mantem cada classe focada e facil de encontrar ao debugar.
</Callout>

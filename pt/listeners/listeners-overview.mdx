## O que é um listener

Um listener é uma classe que reage a eventos do Discord. Quando algo acontece no Discord, como um usuário enviando uma mensagem, um membro entrando em um servidor, ou o bot terminando seu login, o Discord.js dispara um evento. Uma classe listener no Spraxium captura esse evento e roda seu código em resposta. Sem listeners, seu bot não conseguiria responder a nada que acontece fora de comandos explícitos.

A diferença chave entre listeners e comandos é o que os aciona. Comandos sao acionados por um usuário digitando um slash command ou um prefix command. Listeners sao acionados por eventos do Discord que acontecem em background, como mensagens sendo enviadas, membros entrando ou saindo, reações sendo adicionadas, ou o client ficando pronto. Alguns desses eventos envolvem interações de usuário que podem receber respostas, e outros sao eventos de background que acontecem silenciosamente.

Listeners no Spraxium não sao callbacks crus. Sao classes completas gerenciadas pelo container de injeção de dependencia, registradas em módulos e conectadas ao client do Discord automaticamente durante o bootstrap. Isso significa que suas classes listener podem receber serviços injetados, participar do pipeline de guards pra eventos suportados e se beneficiar do mesmo tratamento de exceptions que comandos usam.

## Um exemplo básico de listener

Aqui estao dois listeners que mostram os dois modos de binding. O primeiro usa _@Once_ pra reagir ao _ClientReady_ uma única vez quando o bot conecta. O segundo usa _@On_ pra reagir a todo evento _MessageCreate_ durante toda a vida do bot. Juntos, eles cobrem os dois padrões mais comuns que você vai usar.

<CodeTabs>
  <Tab label="Ready (Once)" language="typescript">
```typescript filename="src/events/listeners/ready.listener.ts"
import { Events, Listener, Once } from '@spraxium/common';
import { Logger } from '@spraxium/core';
import type { Client } from 'discord.js';

@Listener(Events.ClientReady)
export class ReadyListener {
  private logger: Logger = new Logger(ReadyListener.name);

  @Once()
  onReady(client: Client<true>): void {
    this.logger.info(`Client is ready as ${client.user.tag}`);
  }
}

````
  </Tab>
  <Tab label="Mensagens (On)" language="typescript">
```typescript filename="src/events/listeners/message-create.listener.ts"
import { Events, Listener, On } from '@spraxium/common';
import type { Message } from 'discord.js';

@Listener(Events.MessageCreate)
export class MessageCreateListener {
  @On()
  async onMessage(message: Message): Promise<void> {
    if (message.author.bot) return;
    if (message.content === '!ping') {
      await message.reply('pong');
    }
  }
}
````

  </Tab>
</CodeTabs>

Note que o parâmetro do método recebe o payload do evento do Discord.js diretamente. Pra _ClientReady_, o payload e o objeto _Client_. Pra _MessageCreate_, o payload e o objeto _Message_. Cada evento do Discord tem seu próprio tipo de payload, e seu método recebe o que o Discord.js fornece pra aquele evento específico.

## Registrando listeners em um módulo

Listeners não se ativam apenas por existirem no seu código fonte. Você precisa registrá-los no array _listeners_ de um módulo pra que o framework saiba sobre eles e possa conecta-los ao client do Discord durante o bootstrap. Sem esse registro, sua classe listener sera completamente ignorada.

```typescript filename="src/events/events.module.ts"
import { Module } from '@spraxium/common';
import { ReadyListener } from './listeners/ready.listener';
import { MessageCreateListener } from './listeners/message-create.listener';

@Module({
  listeners: [ReadyListener, MessageCreateListener],
})
export class EventsModule {}
```

O módulo em si deve ser importado pelo seu módulo raiz ou outro módulo na cadeia de imports. O framework processa módulos durante o bootstrap, resolve os arrays de listeners, instancia cada classe listener através do container de DI e então os conecta ao client do Discord. Se um listener depende de um serviço, esse serviço deve estar disponível no escopo de providers do módulo ou exportado de um módulo importado, assim como qualquer outra dependencia injetável.

## Como o bootstrap conecta listeners

O processo de bootstrap trata listeners em uma sequencia específica que importa pra entender quando seu listener fica ativo. Primeiro, o framework le o array _listeners_ de cada módulo. Segundo, instancia cada classe listener pelo container de DI, resolvendo dependencias de construtor. Terceiro, o _ListenerRegistry_ interno escaneia cada instancia procurando métodos decorados com _@On_ ou _@Once_ e os agrupa por nome de evento. Quarto, o _ListenerBinder_ pega esses grupos e chama o método _on_ ou _once_ do client Discord.js pra criar os bindings reais. Por fim, quando um evento do Discord dispara, o _ListenerInvoker_ chama cada handler registrado pra aquele evento.

Essa estrutura mantem todas as classes listener em uma pasta dedicada e o arquivo de módulo na raiz de events. Conforme seu bot cresce, você pode adicionar mais arquivos de listener sem mudar a estrutura de pastas. Cada listener trata um evento, e o arquivo de módulo e o único lugar que declara quais listeners estao ativos.

<Callout tone="info" title="Uma classe por evento">
  Cada classe listener é vinculada a exatamente um evento do Discord através do
  decorator Listener. Se você precisa ouvir múltiplos eventos, crie classes
  listener separadas pra cada um. Isso mantem cada classe focada e fácil de
  encontrar ao debugar.
</Callout>

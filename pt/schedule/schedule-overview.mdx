## O que o módulo de schedule resolve

A maioria dos bots Discord precisa executar tarefas em timer. Limpar avisos expirados, sincronizar contagens de membros, postar resumos diários e rotacionar mensagens de status são operações que acontecem independentemente de input do usuário. O módulo de schedule fornece uma API orientada a decorators para definir esses jobs em segundo plano, permitindo declarar lógica de temporização junto aos métodos do serviço ao invés de gerenciar timers brutos.

O módulo suporta quatro tipos de job que cobrem diferentes padrões de temporização. Jobs cron executam em agendamentos baseados em calendário usando expressões cron padrão. Intervalos repetem em offsets fixos de milissegundos. Timeouts disparam uma vez após um atraso. Jobs after-online disparam uma vez após o bot ter se conectado completamente ao Discord. Juntos, esses quatro tipos lidam com todos os cenários comuns de agendamento.

O módulo de schedule é um pacote separado (`@spraxium/schedule`) que se conecta ao framework através do sistema padrão de módulos. Ele escaneia seus providers em busca de métodos decorados no momento do boot, os registra em um registry central e gerencia seu ciclo de vida automaticamente. Você não precisa iniciar ou parar timers manualmente.

## Tipos de job em resumo

Cada tipo de job mapeia para um decorator que você coloca em um método dentro de um serviço injetável. O decorator indica ao scheduler quando e como executar aquele método. Todos os quatro decorators aceitam um objeto de opções para nomeação, desativação e controle de comportamento na inicialização.

## Ciclo de vida do runtime

O módulo de schedule opera em duas fases alinhadas com a sequência de boot do framework. Compreender essas fases ajuda a prever quando cada tipo de job começa a executar e por que _@AfterOnline_ existe como um decorator separado ao invés de ser uma variante de cron.

<Steps>
  <Step title="Fase de boot">
    No startup da aplicação, o framework escaneia todos os providers e registra
    cada método decorado com *@Cron*, *@Interval* ou *@Timeout*. Jobs com
    *runOnInit* habilitado executam imediatamente durante esta fase. Timers de
    *@Interval* e *@Timeout* começam a contar a partir deste momento.
  </Step>
  <Step title="Fase de ready">
    Após o client Discord emitir seu evento *ready*, o framework registra os
    jobs *@AfterOnline* e inicia seus timers de atraso. A separação existe
    porque algumas tarefas precisam de uma conexão ativa com o Discord, como
    buscar dados de servidores ou enviar mensagens de boas-vindas. Executá-las
    durante o boot falharia porque o client não está conectado ainda.
  </Step>
  <Step title="Gerenciamento em runtime">
    Uma vez que todos os jobs estão registrados, o *ScheduleService* fornece uma
    API pública para pausar, retomar, destruir e acionar manualmente jobs por
    nome. Você pode injetar *ScheduleService* em qualquer provider para
    controlar jobs programaticamente a partir de slash commands, prefix commands
    ou endpoints HTTP.
  </Step>
</Steps>

## Configuração mínima

Para usar o módulo de schedule, instale o pacote `@spraxium/schedule`, crie um arquivo de configuração usando _defineSchedule_ e registre o _ScheduleModule_ na sua aplicação. O módulo é global por padrão, então você só precisa registrá-lo uma vez. Qualquer provider em qualquer módulo pode então usar os decorators de schedule.

```typescript filename="config/schedule.config.ts"
import { defineSchedule } from "@spraxium/schedule";

export default defineSchedule({
  timezone: "America/Sao_Paulo",
});
```

<Callout type="info">
  O módulo de schedule usa o *MemoryDriver* por padrão, que armazena estado de
  jobs em memória. Para bots distribuídos rodando múltiplos shards ou processos,
  troque para o *RedisScheduleDriver* para prevenir execuções duplicadas.
</Callout>

## Estrutura do projeto

Um bot típico com tarefas agendadas as organiza como providers de serviço dentro de feature modules. Cada serviço agrupa jobs relacionados, e o arquivo de configuração fica no diretório config do projeto junto com configs de prefix e outros plugins.

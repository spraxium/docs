## Padr\u00e3o split-config

Como outros plugins do Spraxium, o m\u00f3dulo de schedule usa o padr\u00e3o split-config onde a configura\u00e7\u00e3o fica em um arquivo dedicado dentro do diret\u00f3rio `config/`. O helper *defineSchedule* fornece type safety completo e auto-complete para todos os campos do config. Seu `spraxium.config.ts` principal referencia o arquivo de config, mantendo o config raiz limpo e a configura\u00e7\u00e3o de cada plugin isolada.

Esse padr\u00e3o escala bem conforme seu bot cresce. Cada plugin tem seu pr\u00f3prio arquivo de config com suas pr\u00f3prias defini\u00e7\u00f5es de tipo, e o config raiz simplesmente importa e comp\u00f5e todos. Voc\u00ea nunca acaba com um objeto de config monol\u00edtico que mistura configura\u00e7\u00f5es de prefix, schedule e HTTP em um \u00fanico lugar.

O framework automaticamente descobre e carrega o config de schedule durante o bootstrap. Voc\u00ea n\u00e3o precisa chamar nenhuma fun\u00e7\u00e3o de registro manualmente. Basta exportar o config do arquivo e referenci\u00e1-lo no array de imports do config raiz.

<CodeTabs>
  <Tab label="Config de schedule" language="typescript">
```typescript filename="config/schedule.config.ts"
import { defineSchedule } from '@spraxium/schedule';

export default defineSchedule({
  timezone: 'America/Sao_Paulo', defaultLockTtlMs: 30_000, disableAll: false,
});
```
  </Tab>
  <Tab label="Config raiz" language="typescript">
```typescript filename="spraxium.config.ts"
import { defineConfig } from '@spraxium/core';
import schedule from './config/schedule.config';

export default defineConfig({
  plugins: [schedule],
});
```
  </Tab>
</CodeTabs>

## Organiza\u00e7\u00e3o do projeto

Servi\u00e7os de schedule s\u00e3o providers injet\u00e1veis regulares que voc\u00ea registra em qualquer m\u00f3dulo. Um padr\u00e3o comum \u00e9 criar um m\u00f3dulo de tasks dedicado que agrupa todos os jobs em segundo plano, mas voc\u00ea tamb\u00e9m pode colocar m\u00e9todos decorados com schedule em servi\u00e7os espec\u00edficos de dom\u00ednio. A escolha depende de se seus jobs s\u00e3o transversais (como limpeza e sincroniza\u00e7\u00e3o) ou fortemente acoplados a uma feature espec\u00edfica.

Para tarefas transversais, um m\u00f3dulo de tasks centralizado mant\u00e9m as coisas organizadas. Para tarefas espec\u00edficas de feature, colocar o decorator de schedule em um m\u00e9todo no servi\u00e7o da feature evita indire\u00e7\u00e3o desnecess\u00e1ria. Ambas as abordagens s\u00e3o v\u00e1lidas e o framework as trata de forma id\u00eantica.

## Padr\u00f5es comuns

### Combinando m\u00faltiplos tipos de job em um servi\u00e7o

Um \u00fanico servi\u00e7o pode ter m\u00e9todos decorados com diferentes tipos de job. Isso \u00e9 \u00fatil quando tarefas relacionadas t\u00eam requisitos de temporiza\u00e7\u00e3o diferentes mas compartilham as mesmas depend\u00eancias. O scheduler trata cada m\u00e9todo decorado como um job independente com seu pr\u00f3prio nome e ciclo de vida.

```typescript filename="src/modules/tasks/services/maintenance.service.ts"
import { Injectable } from '@spraxium/common';
import { Cron, CronExpression, Interval, IntervalExpression, AfterOnline } from '@spraxium/schedule';

@Injectable()
export class MaintenanceService {
  @Cron(CronExpression.EVERY_DAY_AT_3AM, { name: 'daily-cleanup' })
  async dailyCleanup(): Promise<void> {
    // Arquivar logs antigos, limpar dados tempor\u00e1rios
  }

  @Interval(IntervalExpression.EVERY_10_MINUTES, { name: 'health-ping' })
  async healthPing(): Promise<void> {
    // Reportar sa\u00fade do bot para servi\u00e7o de monitoramento
  }

  @AfterOnline(10_000, { name: 'restore-presences' })
  async restorePresences(): Promise<void> {
    // Restaurar status do bot a partir do banco de dados ap\u00f3s reconex\u00e3o
  }
}
```

### Execu\u00e7\u00e3o condicional de jobs com feature flags

\u00c0s vezes voc\u00ea quer definir um job mas s\u00f3 habilit\u00e1-lo baseado em vari\u00e1veis de ambiente ou condi\u00e7\u00f5es de runtime. A op\u00e7\u00e3o *disabled* nas op\u00e7\u00f5es do decorator permite controlar isso no momento do registro. Para controle em runtime, use os m\u00e9todos *ScheduleService.pause* e *ScheduleService.resume*.

```typescript filename="src/modules/tasks/services/analytics.service.ts"
import { Injectable } from '@spraxium/common';
import { Cron, CronExpression } from '@spraxium/schedule';

@Injectable()
export class AnalyticsService {
  @Cron(CronExpression.EVERY_HOUR, {
    name: 'hourly-analytics', disabled: process.env.ENABLE_ANALYTICS !== 'true', })
  async collectAnalytics(): Promise<void> {
    // Coletar e agregar analytics de uso
  }
}
```

### Registro no m\u00f3dulo

Registre seus servi\u00e7os de tarefa como providers em um m\u00f3dulo. O *ScheduleModule* \u00e9 global e n\u00e3o precisa ser importado no seu feature module. Basta adicionar seus servi\u00e7os ao array de providers e o scheduler descobrir\u00e1 seus m\u00e9todos decorados automaticamente.

```typescript filename="src/modules/tasks/tasks.module.ts"
import { Module } from '@spraxium/common';
import { CleanupService } from './services/cleanup.service';
import { SyncService } from './services/sync.service';
import { MaintenanceService } from './services/maintenance.service';

@Module({
  providers: [CleanupService, SyncService, MaintenanceService],
})
export class TasksModule {}
```

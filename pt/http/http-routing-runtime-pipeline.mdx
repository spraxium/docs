## Superfície de decorators

Os decorators definem intenção de transporte em tempo de carregamento da classe, enquanto os serviços de runtime transformam essa intenção em handlers Hono executáveis durante o startup do servidor. Manter essa separação em mente ajuda a debugar comportamento de rota rapidamente porque o decorator é sempre a fonte de verdade e o runtime apenas lê o que o decorator armazenou em _reflect-metadata_.

Todos os decorators da package escrevem em chaves definidas em _HTTP_METADATA_KEYS_. O _@HttpController_ define o prefixo, decorators de método de rota adicionam entradas ao array _ROUTES_, o _@HttpUseMiddleware_ adiciona ao array _MIDDLEWARE_, o _@HttpGuards_ adiciona ao array _GUARDS_ e decorators de parâmetro adicionam ao array _PARAMS_ indexado pelo nome do handler. Essa metadata é imutável após a inicialização da classe, então o runtime pode lê-la com segurança no startup sem efeitos colaterais.

<DropdownGroup>
<Dropdown title="@HttpController(prefix)">
  Um decorator de classe que define o prefixo de rota compartilhado por todos os métodos do controller. O *RouteBuilder* lê a metadata do prefixo e a prepend no sub-path de cada método ao construir o caminho completo da rota Hono. Controllers sem prefixo usam `'/'` como padrão, registrando suas rotas diretamente no caminho raiz.
</Dropdown>

<Dropdown title="@HttpGet · @HttpPost · @HttpPut · @HttpPatch · @HttpDelete">
  Decorators de método que registram um par de método HTTP e sub-path opcional
  no controller. Cada chamada adiciona uma entrada *RouteDefinition* com método,
  sub-path e nome do handler ao array de metadata `ROUTES` da classe. O
  *RouteBuilder* lê esse array no startup e cria uma rota Hono para cada
  entrada, computando o caminho completo juntando o prefixo do controller com o
  sub-path do método.
</Dropdown>

<Dropdown title="@HttpStatus(code)">
  Um decorator de método que substitui o status padrão de sucesso de 200 por
  qualquer número de status HTTP válido. O *RouteBuilder* lê essa metadata ao
  serializar o valor de retorno do handler via `ctx.json(result, statusCode)`.
  Esse decorator não tem efeito quando o handler retorna um objeto `Response`
  nativo, pois essa resposta ignora o caminho normal de serialização JSON.
</Dropdown>

<Dropdown title="@HttpGuards(...classes)">
  Um decorator de classe ou método que vincula tokens de classe de guard a um
  controller ou rota individual. Guards são resolvidos pelo sistema de DI no
  startup e executados antes do handler pelo *GuardExecutor*, com guards de
  classe executando antes dos de método. Um único retorno `false` encerra a
  request na camada de guard com status 401, antes de qualquer middleware ou
  código de handler.
</Dropdown>

<Dropdown title="@HttpUseMiddleware(...classes)">
  Um decorator de classe ou método que vincula tokens de classe de middleware
  resolvidos via *middlewareProviders*. O *@HttpUseMiddleware* de classe executa
  antes das declarações de método, e ambos executam após middleware global e
  execução de guards. Os tokens de classe precisam estar registrados em
  *@HttpClientModule({middlewareProviders})* ou o binding é silenciosamente
  ignorado.
</Dropdown>

<Dropdown title="@HttpParam(key)">
  Um decorator de parâmetro que extrai um parâmetro nomeado de path da URL via
  `ctx.req.param(key)` e o injeta na posição correspondente do handler. A chave
  deve corresponder ao nome do placeholder definido no path da rota; por
  exemplo, `:guildId` em `'/:guildId'` é extraído com `@HttpParam('guildId')`.
</Dropdown>

<Dropdown title="@HttpQuery(key)">
  Um decorator de parâmetro que lê um parâmetro de query string via
  `ctx.req.query(key)` e o injeta como argumento do handler. O valor é uma
  string quando a chave está presente ou `undefined` quando ausente, então
  handlers com parâmetros opcionais de query devem tratar ambos os casos.
</Dropdown>

<Dropdown title="@HttpBody(DtoClass?)">
  Um decorator de parâmetro que faz parse do body JSON da request e o injeta
  como argumento do handler. Quando uma classe DTO é fornecida, o
  *ValidationPipe* executa `plainToInstance` seguido de `validate` com
  `whitelist: true` e `forbidNonWhitelisted: true`, lançando um
  *ValidationError* com detalhes estruturados se qualquer constraint falhar.
</Dropdown>

<Dropdown title="@HttpHeader(key)">
  Um decorator de parâmetro que lê um header específico da request via
  `ctx.req.header(key)` e injeta seu valor como argumento do handler. Nomes de
  headers são case-insensitive por especificação HTTP e o Hono os normaliza
  antes da busca, então `'x-request-id'` e `'X-Request-Id'` resolvem para o
  mesmo valor.
</Dropdown>

<Dropdown title="@HttpCtx()">
  Um decorator de parâmetro que injeta o objeto *Context* completo do Hono no handler. Isso providência acesso direto a todos os dados brutos da request, métodos de resposta e stores de middleware do Hono para cenários onde os outros decorators são insuficientes. Usar *@HttpCtx()* junto com outros decorators de parâmetro é suportado; cada posição anotada é resolvida de sua própria fonte independentemente.
</Dropdown>
</DropdownGroup>
</Table>

## Resolução de parâmetros em profundidade

O _ParamResolver_ é o serviço que lê metadata de _PARAMS_ e extrai valores do _Context_ do Hono para cada invocação de handler. Ele suporta cinco tipos de fonte, cada um mapeado para uma estratégia específica de extração.

Quando um handler não tem nenhum decorator de parâmetro, o _ParamResolver_ injeta o _Context_ completo como único argumento por padrão. Esse comportamento default significa que você pode escrever handlers simples que aceitam `ctx` diretamente sem decorators, o que é útil para cenários de migração ou handlers que precisam de controle total. Assim que um decorator de parâmetro aparece, apenas parâmetros decorados são resolvidos.

A fonte body merece atenção especial. O _ParamResolver_ faz parse do JSON do body da request apenas uma vez por invocação de handler e mantém o resultado em cache mesmo se múltiplos decorators _@HttpBody_ existirem no mesmo método. Quando uma classe DTO é fornecida, o _ValidationPipe_ executa `plainToInstance` do _class-transformer_ para criar uma instância tipada e depois `validate` do _class-validator_ com `whitelist: true` e `forbidNonWhitelisted: true`. Isso significa que propriedades inesperadas no payload são rejeitadas automaticamente antes da lógica de negócio.

```typescript filename="http/controllers/guild-http.controller.ts"
import {
  HttpBody,
  HttpController,
  HttpCtx,
  HttpGet,
  HttpHeader,
  HttpStatus,
  HttpParam,
  HttpPatch,
  HttpQuery,
} from '@spraxium/http';
import type { Context } from 'hono';
import { NotFoundError } from '@spraxium/http';
import { GuildHttpService } from '../services/guild-http.service';
import { UpdateGuildConfigDto } from '../dto/update-guild-config.dto';

@HttpController('/guilds')
export class GuildHttpController {
constructor(private readonly service: GuildHttpService) {}

@HttpGet('/:guildId')
async getGuild(
  @HttpParam('guildId') guildId: string,
  @HttpQuery('includeMembers') includeMembers?: string,
  @HttpHeader('x-request-id') requestId?: string,
) {
const guild = await this.service.getGuildSummary(guildId);
if (!guild) throw new NotFoundError('Guild not found');
return {
  ok: true,
  ...guild,
  includeMembers: includeMembers === 'true',
  requestId,
};
}

@HttpPatch('/:guildId/config')
@HttpStatus(200)
async updateGuildConfig(
  @HttpParam('guildId') guildId: string,
  @HttpBody(UpdateGuildConfigDto) body: UpdateGuildConfigDto,
) {
return this.service.updateConfig(guildId, body);
}

@HttpGet('/:guildId/raw')
async getRawContext(@HttpCtx() ctx: Context) {
const guildId = ctx.req.param('guildId');
ctx.header('X-Custom-Header', 'spraxium');
return ctx.json({ ok: true, guildId }, 200);
}
}
````

## Pipeline de validação de DTO

Quando *@Body(DtoClass)* é usado, o body JSON bruto passa pelo *ValidationPipe* antes de chegar ao handler. O pipeline executa duas transformações em sequência e rejeita a request se a validação falhar.

Primeiro, `plainToInstance(DtoClass, body)` do *class-transformer* converte o objeto anônimo em uma instância de classe tipada. Esse passo habilita o *class-validator* a ler metadata de decorators da classe DTO. Segundo, `validate(instance, { whitelist: true, forbidNonWhitelisted: true })` executa todos os decorators de constraint na instância. A opção *whitelist* remove propriedades desconhecidas, e *forbidNonWhitelisted* rejeita ativamente requests que as incluem.

Quando erros de validação existem, o *ValidationPipe* lança um *ValidationError* com status 400. Cada entrada de erro no array *details* mapeia um nome de *property* para um record de *constraints*. O handler centralizado *onError* captura isso e retorna `{ ok: false, error: 'Validation failed', details: [...] }`. Esse fluxo automatizado significa que handlers nunca precisam validar payloads de body manualmente, e todo endpoint de mutação ganha shapes de erro consistentes de graça.

## Sequência interna de execução

<Steps>
  <Step title="Resolver grafo de services com ordenação topológica">
    O *RouteRegistry* lê tipos de parâmetros de construtor via *reflect-metadata* e constrói um grafo de dependências. Services são instanciados em ordem para que cada classe receba suas dependências já resolvidas. Se uma dependência circular for detectada, o startup falha com um erro listando os nomes das classes envolvidas.
  </Step>
  <Step title="Instanciar controllers com DI">
    Classes de controller são instanciadas usando o mapa de dependências resolvido mais quaisquer valores fallback do container core. O *RouteRegistry* também lê metadata de *@UseMiddleware* em nível de classe neste estágio para que esteja disponível durante o binding de rotas.
  </Step>
  <Step title="Registrar handlers de rota no Hono">
    O *RouteBuilder* itera as definições de rota de cada controller. Para cada rota, computa o caminho completo juntando o prefixo do controller com o sub-path do método. Mescla referências de middleware de classe e método, resolve-as via *middlewareProviders* e registra a cadeia completa como handler de rota Hono. O binding acontece uma vez no startup, tornando contratos de rota imutáveis durante o runtime.
  </Step>
  <Step title="Resolver parâmetros de método por request">
    Quando uma request corresponde a uma rota, o *ParamResolver* lê metadata *PARAMS* daquele handler. Extrai valores de params de URL, query strings, body parseado, headers ou contexto completo. Se um decorator *@Body* referencia uma classe DTO, o *ValidationPipe* é acionado. O array final de argumentos é passado para a função handler via `apply()`.
  </Step>
  <Step title="Executar handler e formatar resposta">
    O handler executa com argumentos resolvidos. Se retornar um `Response` nativo, o Hono repassa sem alteração. Caso contrário, o *RouteBuilder* serializa o valor de retorno com `ctx.json(result, statusCode)` usando o status do *@HttpStatus* ou 200 por padrão. Qualquer *HttpError* lançado durante a execução propaga para o handler centralizado *onError*.
  </Step>
</Steps>

## Regras de formatação de resposta

Handlers podem retornar dois tipos de valor: objetos simples ou instâncias nativas de *Response*. Entender a diferença previne confusão sobre status codes e content types.

Quando um handler retorna um objeto simples, o *RouteBuilder* encapsula com `ctx.json(result, statusCode)` onde o status code vem do *@HttpStatus* ou default para 200. O content type é definido como `application/json` automaticamente. Esse é o padrão recomendado para a maioria dos endpoints porque mantém handlers focados em dados e delega serialização ao framework.

Quando um handler retorna um `Response` nativo (construído com `ctx.json()`, `ctx.text()` ou `new Response(...)`), o Hono repassa sem modificação. Isso dá controle total sobre headers, status e formato do body, o que é útil para streaming de respostas, downloads binários ou redirects. A contrapartida é que o status code centralizado do *@HttpStatus* é ignorado e a resposta bypassa o caminho normal de serialização JSON.

<Callout tone="warning" title="Trate caminhos de rota como contratos públicos">
  Publique versionamento intencional nos paths quando o comportamento do endpoint mudar. O runtime não detecta colisões de path automaticamente, então renomeações silenciosas quebram clientes. Mantenha prefixos de rota estáveis e explícitos entre deploys.
</Callout>

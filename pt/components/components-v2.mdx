## O que são containers V2

A API Components V2 do Discord substitui o modelo tradicional de layout de mensagens por um sistema de blocos composíveis. Em vez de uma mensagem ser uma combinação plana de conteúdo de texto, embeds e action rows, o V2 permite combinar blocos de texto, seções, galerias de mídia, separadores e action rows dentro de um único elemento container. O container se torna a unidade de layout principal da mensagem.

O sistema V2 do Spraxium envolve essa API com a mesma arquitetura decorator-handler usada em todo o pacote _@spraxium/components_. Você declara um schema de container como uma classe decorada, anota cada elemento estrutural com um decorator de propriedade e renderiza o container inteiro através do _V2Service_ passando dados de runtime. O resultado é um layout fortemente tipado e composível que separa a estrutura dos dados que fluem por ele no momento da renderização.

O pacote oferece dois estilos de criação para V2. O padrão de schema baseado em classes usa _@V2Container_ com decorators de propriedade e é ideal para layouts reutilizáveis e orientados a dados. A API fluente é um builder imperativo que encadeia chamadas em uma instância de _V2ContainerFluentBuilder_ e é ideal para mensagens de uso único ou montadas dinamicamente. Ambos os estilos produzem o mesmo formato de wire e podem ser combinados no mesmo bot.

## Quando usar containers V2

Containers V2 são adequados sempre que a mensagem precisa de um layout estruturado que um embed simples não consegue expressar. Casos de uso comuns incluem leaderboards com listas de entradas de tamanho variável, painéis de dashboard de servidor com thumbnail de ícone e grade de estatísticas, perfis com avatar ao lado de múltiplos campos e galerias de imagens incorporadas em respostas de comandos. Qualquer mensagem que se beneficie de separação visual entre seções, ou que exigiria múltiplos embeds para ser aproximada, é uma boa candidata para um container V2.

<Callout type="warning">
  Containers V2 exigem a flag MessageFlags.IsComponentsV2 ao enviar mensagens.
  Definir essa flag significa que a mensagem não pode carregar embeds
  tradicionais ou uma string de conteúdo simples junto com os containers,
  projete mensagens V2 como layouts independentes, não como melhorias de
  conteúdo já existente.
</Callout>

## Configurando o V2Service

O _V2Service_ é o serviço de renderização e construção para todo trabalho com V2. Ele fica disponível automaticamente via injeção assim que o _ComponentsModule_ é registrado no seu grafo de módulos. Nenhuma configuração adicional é necessária além da configuração padrão do plugin _defineComponents_.

Injete o _V2Service_ em qualquer handler ou serviço que precise renderizar containers V2.

```typescript filename="src/handlers/my.handler.ts"
import { V2Service } from '@spraxium/components';

@SlashCommandHandler(MyCommand)
export class MyHandler {
  constructor(private readonly v2: V2Service) {}

  async handle(@Ctx() interaction: ChatInputCommandInteraction): Promise<void> {
    // ...
  }
}
```

## Schemas de container baseados em classes

O padrão baseado em classes declara um container como uma classe anotada. Aplique _@V2Container_ à classe para marcá-la como container e configurar a cor de destaque e opções de spoiler. Aplique decorators de propriedade a cada campo para registrar componentes filhos na ordem de declaração. No momento da renderização, o _V2Service.build_ lê os metadados, avalia todas as funções fábrica com seus dados de runtime e monta o container.

Os nomes das propriedades na classe de schema não importam em runtime. Somente os metadados registrados pelos decorators são lidos. Declare o tipo da propriedade como `never` para sinalizar que o campo existe apenas como um slot de metadados.

<CodeTabs>
  <Tab label="Schema">
```typescript filename="src/schemas/user-profile.container.ts"
import {
  V2Container,
  V2Separator,
  V2Section,
  V2Text,
  desc,
} from '@spraxium/components';

export interface UserProfileData {
  username: string;
  userId: string;
  joinedAt: number;
  avatarUrl: string;
  bio: string;
}

@V2Container({ accentColor: 0x5865f2 })
export class UserProfileContainer {
  @V2Text((data: UserProfileData) => desc().h2(`👤 ${data.username}`))
  title!: never;

  @V2Separator()
  divider!: never;

  @V2Section({
    text: (data: UserProfileData) =>
      [
        `**ID:** \`${data.userId}\``,
        `**Entrou:** <t:${Math.floor(data.joinedAt / 1000)}:R>`,
        `**Bio:** ${data.bio}`,
      ].join('\n'),
    thumbnail: {
      url: (data: UserProfileData) => data.avatarUrl,
      description: 'Avatar do usu\u00e1rio',
    },
  })
  stats!: never;

  @V2Separator({ divider: false })
  spacer!: never;

  @V2Text((data: UserProfileData) => desc().subtext(`Perfil de ${data.username}`))
  footer!: never;
}

````

```typescript filename="src/handlers/profile.handler.ts"
import { Ctx, SlashCommandHandler } from '@spraxium/common';
import { V2Service } from '@spraxium/components';
import { MessageFlags, type ChatInputCommandInteraction } from 'discord.js';
import { ProfileCommand } from '../commands/profile.command';
import { UserProfileContainer, type UserProfileData } from '../schemas/user-profile.container';

@SlashCommandHandler(ProfileCommand)
export class ProfileHandler {
  constructor(private readonly v2: V2Service) {}

  async handle(@Ctx() interaction: ChatInputCommandInteraction): Promise<void> {
    const user = interaction.user;

    const data: UserProfileData = {
      username: user.username,
      userId: user.id,
      joinedAt: user.createdTimestamp,
      avatarUrl: user.displayAvatarURL({ size: 128 }),
      bio: 'Sem bio definida.',
    };

    const payload = this.v2.buildReply(UserProfileContainer, data);

    await interaction.reply({
      ...payload,
      flags: payload.flags | MessageFlags.Ephemeral,
    });
  }
}
````

## A API fluente

A API fluente é um builder encadeável vinculado ao _V2Service_. Chame _v2.container()_ para abrir um builder, encadeie as chamadas de componentes filhos e chame _.toReply()_ ao final para produzir o payload pronto para _interaction.reply()_. Cada método de encadeamento retorna o mesmo builder, então todas as operações podem ser inline.

A API fluente não suporta decorators de propriedade nem condições _@V2When_, pois não há metadados em nível de classe para anotar. Use o padrão baseado em classes quando precisar de renderização condicional ou quando quiser reutilizar o mesmo layout em múltiplos handlers.

```typescript filename="src/handlers/greet.handler.ts"
import { Ctx, SlashCommandHandler } from '@spraxium/common';
import { V2Service, desc } from '@spraxium/components';
import { MessageFlags, type ChatInputCommandInteraction } from 'discord.js';
import { GreetCommand } from '../commands/greet.command';

@SlashCommandHandler(GreetCommand)
export class GreetHandler {
  constructor(private readonly v2: V2Service) {}

  async handle(@Ctx() interaction: ChatInputCommandInteraction): Promise<void> {
    const user = interaction.user;

    const payload = this.v2
      .container({ accentColor: 0x57f287 })
      .text(desc().h2(`👋 Olá, ${user.username}!`))
      .sep()
      .section(
        [
          `**ID:** \`${user.id}\``,
          `**Idade da conta:** <t:${Math.floor(user.createdTimestamp / 1000)}:R>`,
        ].join("\n"),
        this.v2.thumbnail(user.displayAvatarURL({ size: 128 }), {
          description: `Avatar de ${user.username}`,
        }),
      )
      .sep({ divider: false })
      .text(desc().subtext('Desenvolvido com Spraxium V2'))
      .toReply();

    await interaction.reply({
      ...payload,
      flags: payload.flags | MessageFlags.Ephemeral,
    });
  }
}
```

## Filhos dinâmicos com @V2Dynamic

_@V2Dynamic_ é a solução quando o número de componentes filhos depende de dados de runtime. Em vez de uma configuração estática, você fornece uma função fábrica que recebe o objeto de dados e retorna um array de specs de filhos criados com as funções auxiliares _v2text_, _v2sep_, _v2section_, _v2gallery_ e _v2row_. O motor percorre esse array e renderiza cada spec em sequência.

Um padrão comum é gerar uma ou mais linhas de texto por entrada de dados e inserir separadores entre elas, parando um separador antes para que a última entrada não seja seguida por um divisor.

```typescript filename="src/schemas/leaderboard.container.ts"
import {
  V2Container,
  V2Dynamic,
  V2Separator,
  V2Text,
  desc,
  v2sep,
  v2text,
} from '@spraxium/components';

export interface LeaderboardEntry {
  username: string;
  score: number;
  rank: number;
}

export interface LeaderboardData {
  title: string;
  entries: Array<LeaderboardEntry>;
}

const MEDALS = ['🥇', '🥈', '🥉'];

@V2Container({ accentColor: 0xf1c40f })
export class LeaderboardContainer {
  @V2Text((data: LeaderboardData) => desc().h2(`🏆 ${data.title}`))
  title!: never;

  @V2Separator()
  divider!: never;

  @V2Dynamic((data: LeaderboardData) => {
    if (data.entries.length === 0) {
      return [v2text(desc().subtext('Nenhuma entrada ainda.'))];
    }

    return data.entries.flatMap((entry, index) => {
      const medal = MEDALS[index] ?? `**${entry.rank}.**`;
      return [
        v2text(
          `${medal} **${entry.username}**, ${entry.score.toLocaleString()} pts`,
        ),
        ...(index < data.entries.length - 1 ? [v2sep({ divider: false })] : []),
      ];
    });
  })
  entries!: never;

  @V2Separator({ divider: false })
  footer!: never;

  @V2Text((data: LeaderboardData) =>
    desc().subtext(`${data.entries.length} jogadores classificados`),
  )
  summary!: never;
}
```

O handler passa a lista de entradas como dados e chama _buildReply_ sem precisar saber quantas linhas serão geradas.

```typescript filename="src/handlers/leaderboard.handler.ts"
@SlashCommandHandler(LeaderboardCommand)
export class LeaderboardHandler {
  constructor(private readonly v2: V2Service) {}

  async handle(@Ctx() interaction: ChatInputCommandInteraction): Promise<void> {
    const entries: Array<LeaderboardEntry> = await fetchTopPlayers(10);

    const payload = this.v2.buildReply(LeaderboardContainer, {
      title: 'Top Jogadores da Semana',
      entries,
    });

    await interaction.reply(payload);
  }
}
```

## Renderização condicional com @V2When

_@V2When_ inclui condicionalmente um filho com base nos dados de runtime. É aplicado como um decorator adicional _abaixo_ do decorator do filho, o que significa que executa por último e adiciona um predicado de guarda à definição de filho existente. Se o predicado retornar `false` para os dados fornecidos, o filho é excluído do container, não renderizado como vazio.

_@V2When_ é empilhável: múltiplos filhos no mesmo container podem cada um ter condições independentes. A propriedade de ordem atribuída no momento do registro é preservada independentemente de quais filhos são filtrados.

```typescript filename="src/schemas/member-card.container.ts"
import {
  V2Container,
  V2Separator,
  V2Text,
  V2When,
  desc,
} from '@spraxium/components';

export interface MemberCardData {
  username: string;
  boostLevel: number;
  isAdmin: boolean;
  bio: string;
}

@V2Container({ accentColor: 0x5865f2 })
export class MemberCardContainer {
  @V2Text((data: MemberCardData) => desc().h2(data.username))
  title!: never;

  @V2Separator()
  divider!: never;

  @V2When((data: MemberCardData) => data.isAdmin)
  @V2Text('⚙️ **Administrador do Servidor**')
  adminBadge!: never;

  @V2When((data: MemberCardData) => data.boostLevel > 0)
  @V2Text(
    (data: MemberCardData) => `⭐ **Server Booster**, Nível ${data.boostLevel}`,
  )
  boostBadge!: never;

  @V2Text((data: MemberCardData) => data.bio)
  bio!: never;
}
```

Quando `isAdmin` é `false`, a linha do emblema de administrador não é renderizada. Os filhos restantes se fecham naturalmente sem nenhum espaço vazio.

## Incorporando action rows com @V2Row

_@V2Row_ coloca uma action row de botões ou select menu diretamente dentro de um container V2. Passe as classes dos seus componentes no array `components` da mesma forma que faria com _ButtonService_ ou _SelectService_. O serviço detecta automaticamente se a primeira classe é um select menu e roteia para o builder correto.

## O DescriptionBuilder

A função auxiliar _desc()_ retorna um _DescriptionBuilder_ que encadeia helpers de Markdown com sabor do Discord. O builder é um acumulador de string leve; chamar _.toString()_ retorna a string final, que o motor V2 chama automaticamente antes de passar o conteúdo para os builders do discord.js.

## Construindo o payload de reply

Tanto _buildReply_ quanto o método fluente _.toReply()_ retornam um objeto `{ components, flags }`. Fazer spread desse objeto em _interaction.reply()_ define `IsComponentsV2` automaticamente. Você pode combinar flags adicionais usando OR bitwise.

```typescript filename="enviando um reply V2"
const payload = this.v2.buildReply(MyContainer, data);

await interaction.reply({
  ...payload,
  flags: payload.flags | MessageFlags.Ephemeral,
});
```

Se você precisar atualizar uma mensagem V2 existente, use _interaction.editReply()_ ou passe o payload para _message.edit()_ diretamente. A flag é necessária tanto no envio inicial quanto nas edições subsequentes, pois o Discord identifica uma mensagem como V2 pela presença dessa flag.

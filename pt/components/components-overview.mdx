## O que são componentes

O pacote *@spraxium/components* fornece um sistema baseado em decorators para construir elementos interativos da interface do Discord. Botões, menus de seleção, modais e embeds são todos definidos como classes decoradas, conectados através de metadados e despachados automaticamente quando os usuários interagem com seu bot. O pacote também inclui um sistema de *flow context* que encadeia dados de estado entre interações de múltiplas etapas sem gerenciamento manual de estado.

Diferente dos construtores brutos do discord.js, onde você constrói cada objeto imperativamente, os componentes do Spraxium usam decorators de classe para declarar a aparência de um componente, decorators de handler separados para definir o que acontece quando um usuário interage, e serviços injetáveis para renderizar componentes em tempo de execução. Essa separação mantém as definições de componentes reutilizáveis, os handlers testáveis e a renderização desacoplada da lógica de negócio.

O pacote oferece cinco famílias de componentes: botões (estáticos, link e dinâmicos), menus de seleção (string, user, role, mentionable, channel), modais (com 11 tipos de campos, validação e cache), embeds (declarativos baseados em schema) e containers v2 (blocos de layout composíveis). Cada família segue o mesmo padrão decorator-handler-service, então aprender uma transfere diretamente para as outras.

## Configuração do plugin

O módulo lê sua configuração a partir de uma entrada de plugin produzida por *defineComponents*. Essa fábrica aceita um objeto *ComponentsConfig* que controla o armazenamento de contexto, mensagens de erro e comportamento por componente. O registro segue o mesmo padrão de todo plugin Spraxium: chame a fábrica, adicione o resultado ao array *plugins* em `spraxium.config.ts`.

```typescript filename="config/components.config.ts"
import { defineComponents } from '@spraxium/components';

export const componentsPlugin = defineComponents({
  context: {
    storage: 'memory', defaultTtl: 300, }, errorMessages: {
    expired: 'This interaction has expired.', restricted: 'You are not allowed to use this component.', }, });
```

O campo *context.storage* seleciona o backend de persistência para flow contexts. Quatro adaptadores estão disponíveis: `'memory'` para desenvolvimento, `'filesystem'` para bots de instância única, `'redis'` para implantações multi-instância e `'sqlite'` para persistência leve. O campo *defaultTtl* define a expiração em segundos para contextos que não especificam seu próprio TTL.

## Padrão decorator-handler

Todo componente interativo segue um padrão de duas classes. A primeira classe define a aparência e os metadados do componente. A segunda classe define o que acontece quando o componente é acionado. Um decorator de handler conecta a classe do handler à classe do componente, criando uma ligação tipada que o dispatcher resolve em tempo de execução.

Essa separação significa que a mesma definição de componente pode ter handlers diferentes em módulos diferentes, ou que um handler pode ser trocado sem alterar a aparência do componente. O dispatcher despacha interações para handlers com base nos metadados armazenados pelo decorator, não por referência direta de classe.

<CodeTabs>
<Tab label="Componente">
```typescript filename="src/components/confirm.button.ts"
import { Button } from '@spraxium/components';

@Button({
  customId: 'confirm-action', label: 'Confirm', style: 'success', })
export class ConfirmButton {}
```
</Tab>
<Tab label="Handler">
```typescript filename="src/components/confirm.handler.ts"
import { ButtonHandler, Ctx } from '@spraxium/components';
import type { ButtonInteraction } from 'discord.js';
import { ConfirmButton } from './confirm.button';

@ButtonHandler(ConfirmButton)
export class ConfirmButtonHandler {
  async handle(@Ctx() interaction: ButtonInteraction): Promise<void> {
    await interaction.reply({ content: 'Action confirmed.', ephemeral: true });
  }
}
```
</Tab>
</CodeTabs>

## Registro de módulo

Handlers de componentes devem ser registrados no array *handlers* de um módulo de feature. Classes de componentes decoradas com *@Button*, *@StringSelect*, *@ModalComponent* ou similares são metadados puros e nunca são registradas em nenhum módulo. O dispatcher lê os metadados delas diretamente da referência da classe armazenada dentro do decorator do handler. Você pode agrupar handlers relacionados em um único módulo de feature ou separá-los quando estiverem associados a comandos distintos.

```typescript filename="src/components/components.module.ts"
import { Module } from '@spraxium/core';
import { ConfirmButtonHandler } from './confirm.handler';

@Module({
  handlers: [ConfirmButtonHandler], })
export class ConfirmModule {}
```

## Ciclo de vida do bootstrap

O sistema de componentes inicializa durante a fase de boot do Spraxium, após todos os módulos terem sido carregados e os providers resolvidos. O processo ocorre em quatro etapas que conectam metadados de decorators com o sistema de eventos do discord.js.

<Steps>
<Step title="Escaneamento de instâncias">
O *ComponentsModule* registra um scanner de instâncias que coleta todos os providers e handlers instanciados de todos os módulos. Cada instância é adicionada a um buffer *pendingInstances* que aguarda processamento pelo dispatcher. O scanner executa durante a fase de resolução do módulo.
</Step>

<Step title="Processamento do dispatcher">
O *ComponentLifecycle* chama `dispatcher.processPending()`. O *ComponentDispatcher* percorre as instâncias pendentes, lê os metadados de decorator de cada uma e delega para o sub-despachador apropriado: *ButtonDispatcher*, *SelectDispatcher* ou *ModalDispatcher*. Cada sub-despachador registra os metadados em seus próprios mapas internos de *customId* para handler.
</Step>

<Step title="Inicialização do contexto">
O lifecycle inicializa o adaptador de armazenamento de contexto com base no campo *context.storage* da configuração. O adaptador selecionado é injetado no *ContextService*, que se torna disponível para todos os providers em todo o aplicativo graças ao escopo *@Global()* do módulo.
</Step>

<Step title="Vinculação ao client">
Se algum handler tiver sido registrado, o lifecycle chama `dispatcher.bind(client)`. Isso registra um listener no evento *interactionCreate* que roteia interações de botão, select e modal para o handler correspondente com base na correspondência de *customId*. Se nenhum handler for encontrado, o listener é omitido para evitar overhead.
</Step>
</Steps>

## Injeção de parâmetros

Handlers recebem dados através de decorators de parâmetro no método *handle*. O dispatcher resolve cada decorator antes de invocar o handler, injetando a interação, valores selecionados, dados de contexto ou campos de modal. ## Renderização com serviços

Cada família de componentes tem um serviço injetável que transforma os metadados de decorator em objetos do discord.js prontos para enviar. Os serviços são fornecidos pelo *ComponentsModule* e disponíveis globalmente graças ao decorator *@Global()*. Você os injeta no construtor do seu comando ou handler e chama o método de build apropriado.

Os serviços de botão e select produzem action rows que podem ser anexadas à propriedade *components* de uma resposta de interação. O serviço de modal produz objetos modais que são passados para *interaction.showModal()*. O serviço de embed produz instâncias de *EmbedBuilder* que são passadas para a propriedade *embeds* da resposta.

Todos os serviços seguem uma API consistente: *build* para renderização simples, *buildFor* ou *rowWithContext* quando o flow context está envolvido, e variantes de conveniência como *simple* no serviço de embed.

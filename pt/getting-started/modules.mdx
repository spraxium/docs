## Por que módulos importam

O Spraxium escala através de módulos, e cada feature do bot pertence a exatamente um deles, seja para eventos, moderação, comandos ou integrações externas. Cada módulo é responsável por declarar o que ele provê, o que importa de outros módulos e o que expõe de volta para o resto da aplicação. Esse modelo de fronteiras explícitas significa que você pode crescer a base de código adicionando novos módulos sem aumentar a complexidade dos existentes, porque cada módulo é autocontido, testável independentemente e fácil de entender de forma isolada.

Sem essa estrutura, o código tende a se acumular em arquivos compartilhados, dependências ficam implícitas e adicionar uma nova feature arrisca quebrar algo não relacionado em uma parte completamente diferente da base de código. Módulos previnem isso forçando você a ser explícito sobre relacionamentos desde o início, deixando claro à primeira vista do que cada parte do bot depende e o que ela provê para os demais.

## Arquitetura de módulo recomendada

O layout padrão isola cada feature atrás de uma fronteira de módulo com sua própria pasta contendo no mínimo um arquivo de módulo e um arquivo de service. Módulos evoluem independentemente uns dos outros, e o contêiner de DI cuida de toda a conexão entre eles no boot sem nenhuma configuração manual ou funções factory.

## Declarar o módulo raiz

O arquivo de bootstrap _main.ts_ é o ponto de entrada onde você valida o ambiente, cria a instância da aplicação, carrega o módulo raiz e começa a escutar eventos do Discord. A chamada `app.useModule(AppModule)` é o ponto de entrada de composição, e a partir daí o Spraxium carrega todos os módulos importados recursivamente, resolve providers via injeção de dependência e vincula cada listener registrado ao client do Discord automaticamente.

```typescript filename="src/main.ts"
import { IntentPreset, Logger, SpraxiumFactory } from "@spraxium/core";
import { EnvValidator } from "@spraxium/env";
import { AppEnv } from "./app.env";
import { AppModule } from "./app.module";

async function main() {
  const logger = new Logger("Main");
  const environment = EnvValidator.validate(AppEnv);
  const app = await SpraxiumFactory.create({ token: environment.token });
  app.useModule(AppModule);
  app.intents(IntentPreset.Standard);

  logger.info("Iniciando aplicação...");
  await app.listen();
}

main();
```

## Construir a raiz de composição

O módulo raiz _AppModule_ compõe features importando outros módulos mas nunca deve conter lógica de feature diretamente. Quando você quiser adicionar um novo domínio ao bot, crie um módulo de feature dedicado e importe-o aqui, mantendo a raiz enxuta e legível independente de quão grande ou complexo o bot se torne ao longo do tempo.

```typescript filename="src/app.module.ts"
import { Module } from "@spraxium/common";
import { AnalyticsModule } from "@spraxium/analytics";
import { NomeModuloModule } from "./modules/nome-modulo/nome-modulo.module";

@Module({
  imports: [AnalyticsModule, NomeModuloModule],
})
export class AppModule {}
```

## Referência de propriedades do módulo

<DropdownGroup>
  <Dropdown title="imports, carregar módulos antes do atual">
    Registra os módulos dos quais o módulo atual depende. O Spraxium os carrega
    e inicializa na ordem correta antes de instanciar os providers do módulo que
    os importa. Use no *AppModule* para compor todos os módulos de feature, e em
    qualquer módulo de feature que dependa de providers exportados por outro.
  </Dropdown>
  <Dropdown title="providers, registrar classes injetáveis no contêiner">
    Lista as classes decoradas com *@Injectable()* que o contêiner de DI deve
    instanciar e gerenciar. Cada provider é criado uma vez por módulo e
    compartilhado entre todas as classes que o injetam. Use para services,
    repositórios e qualquer classe de runtime com dependências.
  </Dropdown>
  <Dropdown title="exports, expor providers para outros módulos">
    Torna os providers listados disponíveis para qualquer módulo que importe
    este. Sem exports, os providers são privados ao módulo que os declara. Use
    quando um service precisar ser compartilhado entre fronteiras de módulo sem
    torná-lo global.
  </Dropdown>
  <Dropdown title="listeners, registrar handlers de eventos do Discord">
    Vincula classes listener ao client do Discord para reagir a eventos de
    gateway como *messageCreate* ou *guildMemberAdd*. Os listeners são
    instanciados com suporte completo de DI, podendo injetar qualquer provider
    exportado disponível no módulo.
  </Dropdown>
  <Dropdown title="commands, registrar metadados de comandos">
    Registra as classes de metadados de *SlashCommand* e *PrefixCommand* que
    descrevem o nome, descrição e opções do comando. O Spraxium usa esses
    metadados para sincronizar comandos com o Discord e rotear interações
    recebidas ao handler correto.
  </Dropdown>
  <Dropdown title="handlers, registrar a lógica de execução dos comandos">
    Registra as classes *SlashCommandHandler* e *PrefixCommandHandler* que
    contêm a lógica de execução real. Os handlers recebem o contexto de
    interação com acesso completo ao DI do módulo, o que os mantém focados e
    fáceis de testar de forma isolada.
  </Dropdown>
</DropdownGroup>

## Módulos globais com _@Global()_

Por padrão, os providers de um módulo ficam disponíveis apenas para módulos que o importam explicitamente, que é o comportamento desejado para a maioria dos módulos de feature porque mantém as dependências visíveis e rastreadas. Marcar um módulo com _@Global()_ torna seus exports disponíveis em toda a aplicação sem exigir imports explícitos, mas isso deve ser reservado exclusivamente para preocupações transversais como logging, analytics ou services de configuração compartilhada que genuinamente precisam ser acessados de todas as partes da base de código.

<Callout tone="warning" title="Use @Global() com moderação">
  Marcar muitos módulos como globais contorna o encapsulamento que torna o
  sistema de módulos valioso e dificulta o rastreamento de dependências à medida
  que o projeto cresce. Reserve *@Global()* exclusivamente para infraestrutura
  que todo módulo genuinamente precise, como logging ou analytics, e mantém os
  services de feature dentro do próprio módulo com exports explícitos.
</Callout>

```typescript filename="src/modules/analytics/analytics.module.ts"
import { Global, Module } from "@spraxium/common";
import { AnalyticsService } from "./analytics.service";

@Global()
@Module({
  providers: [AnalyticsService],
  exports: [AnalyticsService],
})
export class AnalyticsModule {}
```

Use módulos globais com moderação porque eles contornam as fronteiras de encapsulamento que tornam o sistema de módulos valioso em primeiro lugar. Services específicos de feature devem sempre permanecer locais ao seu próprio módulo com exports explícitos, e apenas infraestrutura verdadeiramente universal deve ser global. Se você perceber que está marcando mais de dois ou três módulos como globais, geralmente indica que as fronteiras dos módulos precisam ser redesenhadas em vez de contornadas.

## Regras para manter módulos sustentáveis

Manter módulos saudáveis ao longo do tempo exige seguir algumas diretrizes práticas que previnem a complexidade de crescer sorrateiramente conforme o bot evolui. Mantenha uma fronteira de negócio por pasta de módulo, e exporte apenas o que outros módulos devem consumir enquanto mantém todos os detalhes de implementação internos privados. Separe declarações de command e listener dos internals de service, e prefira muitos módulos pequenos e focados compostos pelo _AppModule_ em vez de poucos módulos grandes e compartilhados que tentam fazer demais. Conforme o bot amadurece, é normal que um módulo existente se divida em dois mais focados quando um único domínio cresce além do seu escopo original, e esse tipo de divisão é sinal de crescimento saudável em vez de uma carga de refatoração.

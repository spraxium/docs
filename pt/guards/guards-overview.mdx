## O que é um guard

Um guard é uma classe que responde uma única pergunta antes de qualquer comando ou listener rodar: essa execução deve continuar? Se o guard diz sim, o handler segue normalmente. Se o guard diz não, o framework para a execução imediatamente e o método do handler nunca roda. Esse é o conceito central por trás dos guards no Spraxium, é tudo nessa categoria é construído em cima dessa decisão simples de permitir ou negar.

Na prática, isso significa que você não precisa escrever checagens de permissão, validações de role ou lógica de rate limit dentro de cada método handler. Em vez disso, você escreve essa lógica uma única vez em uma classe guard e aplica onde for necessário. O guard roda automaticamente antes do handler, e o handler lida apenas com lógica de negócio porque a fronteira de autorização já aconteceu. Isso mantém seus command handlers limpos e sua política de acesso centralizada em classes dedicadas e reutilizáveis.

Guards também se integram com a camada de exceptions. Quando um guard nega execução, o framework pode renderizar uma resposta pro usuário através de exception layouts sem que o handler precise saber sobre isso. Essa separação é o que faz dos guards uma fronteira arquitetural de verdade, é não apenas uma função utilitária que você chama manualmente no início de cada handler.

## A interface SpraxiumGuard

Todo guard implementa a interface _SpraxiumGuard_, que exige um único método chamado _canActivate_. Esse método recebe o _ExecutionContext_ atual e deve retornar um boolean ou uma promise que resolve pra boolean. Se o método retorna true, a execução continua pro próximo guard ou pro handler. Se retorna false, a execução para e o framework trata como uma negação.

```typescript filename="guild-only.guard.ts"
import { Guard } from "@spraxium/common";
import type { ExecutionContext, SpraxiumGuard } from "@spraxium/common";

@Guard()
export class GuildOnlyGuard implements SpraxiumGuard {
  canActivate(ctx: ExecutionContext): boolean {
    return ctx.getGuildId() !== null;
  }
}
```

O método _canActivate_ suporta retornos síncronos e assíncronos. Se seu guard precisa checar um banco de dados, chamar uma API externa ou fazer qualquer operação async pra decidir se a execução deve continuar, você pode retornar uma _Promise_ e o framework vai aguardar o resultado antes de tomar a decisão. Essa flexibilidade significa que guards funcionam igualmente bem pra checagens instantâneas como verificar contexto de guild e pra checagens mais pesadas como consultar um banco de dados de assinaturas.

## O ExecutionContext

O _ExecutionContext_ é o objeto passado pro método _canActivate_ de todo guard. Ele encapsula uma interaction ou message do Discord e fornece uma API unificada pra que seu guard não precise saber se o comando foi acionado por um slash command ou um prefix command. Cada método do contexto extrai uma informação específica sobre quem acionou a execução e onde aconteceu.

você pode estar se perguntando por que _getUserId_ e _getUser_ existem separadamente. O motivo é performance: se seu guard só precisa comparar o ID do usuário contra uma lista de owner IDs, chamar _getUserId_ evita resolver o objeto User completo. Se seu guard precisa de mais informações como a data de criação da conta, _getUser_ te da o objeto completo. Use o que fizer sentido pra checagem específica do seu guard.

## O pipeline de execução

Guards não rodam em ordem aleatória. O Spraxium avalia em três camadas, sempre na mesma sequencia: guards globais primeiro, depois guards de classe, depois guards de método. Se todos os guards em todas as três camadas retornam true, o método handler executa. Se qualquer guard retorna false ou lança uma exception em qualquer ponto da cadeia, a execução para imediatamente e todos os guards restantes são pulados.

<Steps>
  <Step title="Guards globais rodam primeiro">
    Sao registrados durante o bootstrap da aplicação com *useGlobalGuards* na
    instância *SpraxiumApplication*. Eles se aplicam a todo comando e todo
    listener com guard na aplicação inteira, o que os torna o lugar certo pra
    regras amplas como bloquear contas de bot, restringir acesso durante
    manutenção ou aplicar rate limit global.
  </Step>
  <Step title="Guards de classe rodam em segundo">
    Sao aplicados com *UseGuards* na classe do handler ou listener. Eles se
    aplicam a todo método dentro daquela classe, o que é útil quando todos os
    comandos de um módulo compartilham o mesmo requisito de acesso. Um exemplo
    típico e aplicar um guard de role na classe pra que todo subcommand de um
    handler de moderação exija a role Moderator.
  </Step>
  <Step title="Guards de método rodam por ultimo">
    Sao aplicados com *UseGuards* diretamente em um método específico. Eles se
    aplicam apenas aquele método, o que permite adicionar requisitos mais
    rigorosos pra operações sensiveis como banir usuários ou resetar dados sem
    afetar o resto dos métodos da classe.
  </Step>
  <Step title="Handler executa">
    Somente depois que todo guard em toda camada retornou true o framework chama
    o método handler de fato. Se seu código chega ao handler, você pode ter
    certeza de que todas as políticas de acesso configuradas já foram
    satisfeitas pro contexto de execução atual.
  </Step>
</Steps>

Esse pipeline usa avaliação short-circuit. No momento em que qualquer guard nega a execução, o framework para de avaliar e vai direto pro tratamento de negação. Isso é importante pra performance porque você deve colocar checagens baratas antes das caras sempre que possível. Por exemplo, uma simples checagem de guild-only que le uma propriedade deve vir antes de uma consulta ao banco que verifica status de assinatura premium.

<Callout tone="info" title="Guard versus lógica de negócio">
  Um guard responde "isso pode rodar?" e nada mais. O handler responde "o que
  deve acontecer?". Mantenha essas responsabilidades separadas pra que guards
  fiquem pequenos, reutilizáveis e fáceis de testar independentemente.
</Callout>

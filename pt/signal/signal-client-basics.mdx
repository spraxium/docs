## Papel do emissor

O *Signal Client* existe para publicar eventos de serviços externos em uma fronteira de integração orientada ao bot. Mesmo quando o emissor e um endpoint de API, um worker cron ou uma Server Action do Next.js, o objetivo operacional e sempre o mesmo: acionar comportamento do bot de forma controlada e auditável. O pacote cuida da construção de envelope, assinatura HMAC, rotação de webhooks e absorção de rate-limit para que o código produtor foque na lógica de negócio.

Essa perspectiva evita que times transformem o client em uma ferramenta de mensageria generica. O client e mais valioso quando seus serviços precisam de uma forma segura e consistente de notificar domínios do bot sobre mudanças de estado sem se acoplar a internals do gateway Discord.

## Contrato do construtor

O *SignalClient* recebe um único objeto *SignalClientConfig* e um *FallbackStore* customizado opcional. O construtor cria um *SignalEnvelopeBuilder* interno para assinatura HMAC, um *WebhookPool* para entrega round-robin e, quando habilitado, um *FallbackWorker* que inicia imediatamente. Todo campo além de `webhookUrls` e `hmacSecret` possui um default sensato, então uma configuração mínima só requer esses dois valores.

```typescript filename="signal-client.setup.ts"
import { SignalClient } from '@spraxium/signal-client';

const client = new SignalClient({
  webhookUrls: [
    process.env.SIGNAL_WEBHOOK_URL_1!, process.env.SIGNAL_WEBHOOK_URL_2!, ], hmacSecret: process.env.SIGNAL_HMAC_SECRET!, rateLimitStrategy: 'skip', });
```

O segredo compartilhado em `hmacSecret` deve corresponder exatamente a configuração bot-side. Se os valores divergirem, o bot rejeita todo envelope na verificação HMAC (regra R10) e nenhuma execução de listener acontece.

## Construção e assinatura de envelope

Toda chamada a *send* delega internamente ao *SignalEnvelopeBuilder*, que produz um envelope completo e criptograficamente assinado. O builder gera um UUID aleatorio como nonce, captura o timestamp atual como *sentAt* e computa a assinatura HMAC-SHA256 sobre a concatenação `nonce.sentAt.JSON.stringify(payload)`. Essa formula e idêntica a usada por *@spraxium/signal* no lado receptor, então ambos os lados concordam no contrato de assinatura sem nenhuma negociação adicional.

O envelope resultante sempre carrega versão de protocolo `1`, o nome do evento, o guild ID alvo e o payload junto com os campos criptograficos. O código produtor nunca precisa interagir com o builder diretamente porque *send* encapsula todo o fluxo.

## O método send

*send* e a API primaria. Ele aceita um nome de evento, um guild ID, um payload opcional e *SendOptions* opcionais. O fluxo interno constrói o envelope, escolhe o próximo webhook disponível do pool e entrega via HTTP POST. Se o webhook retornar 429, o pool o marca como rate-limited e tenta imediatamente o próximo candidato. Quando todos os webhooks estão esgotados, o comportamento depende da *rateLimitStrategy* configurada.

```typescript filename="emit-event.ts"
await client.send('billing.status.changed', guildId, {
  plan: 'premium', expiresAt: '2025-12-31', });
```

A interface *SendOptions* expõe dois campos opcionais. O campo *webhookIndex* força entrega por um webhook específico, ignorando a rotação round-robin. O campo *skipFallback* e usado internamente pelo *FallbackWorker* para evitar loops de re-enqueue durante tentativas de retry.

## Rotação do pool de webhooks

O *WebhookPool* distribui envios entre webhooks configurados usando um cursor round-robin. Quando um webhook retorna HTTP 429, ele e marcado como rate-limited pela duração especificada no header `Retry-After`, e o pool automaticamente o ignora ate o cooldown expirar. Adicionar múltiplas URLs de webhook melhora significativamente a resiliencia em bursts porque o client sempre tem endpoints alternativos para tentar.

O pool expõe varios métodos de inspeção úteis para monitoramento e debug. Você pode acessar o pool a qualquer momento pela propriedade *pool* na instância do client.

## Estrategias de rate-limit

Quando todos os webhooks do pool estão rate-limited simultaneamente, o client precisa decidir o que fazer. O campo *rateLimitStrategy* controla esse comportamento. Ambas as estrategias são escolhas válidas dependendo da arquitetura do seu serviço.

<Table>
  <TableHead>
    <TableRow>
      <TableCell header>Estrategia</TableCell>
      <TableCell header>Comportamento</TableCell>
      <TableCell header>Melhor cenário</TableCell>
    </TableRow>
  </TableHead>
  <TableBody>
    <TableRow>
      <TableCell>`'skip'`</TableCell>
      <TableCell>Lança *AllWebhooksRateLimitedError* imediatamente. O erro inclui o status completo do pool com timestamps de cooldown para que o chamador decida quando tentar novamente.</TableCell>
      <TableCell>Serviços que já possuem sua própria orquestração de retry ou infraestrutura de fila.</TableCell>
    </TableRow>
    <TableRow>
      <TableCell>`'wait'`</TableCell>
      <TableCell>Espera ate o webhook mais próximo sair do cooldown (via *msUntilAvailable*), depois tenta a entrega exatamente uma vez. Se a nova tentativa também falhar, lança exceção.</TableCell>
      <TableCell>Absorção de bursts curtos dentro do client sem lógica externa de retry.</TableCell>
    </TableRow>
  </TableBody>
</Table>

O default e `'skip'` porque mantém o client não bloqueante. Para a maioria dos contextos de Server Action ou API handler onde breve latência e aceitavel, `'wait'` oferece melhor ergonomia absorvendo picos transientes de rate-limit de forma transparente.

## Tipos de erro

O client expõe duas classes de erro que fornecem informação estruturada sobre falhas. Ambas estendem `Error` e carregam campos legíveis por maquina para tratamento programatico.

*SignalClientError* e lançado em respostas HTTP não-2xx e não-429. Ele expõe o *statusCode* e o *body* da resposta como propriedades, facilitando o log de falhas de transporte com contexto completo. Esse erro indica que o próprio endpoint do webhook teve um problema, não uma situação de rate-limit.

*AllWebhooksRateLimitedError* estende *SignalClientError* com `statusCode` de `429` e expõe um array *poolStatus*. Cada entrada no array e um snapshot *WebhookStatus* com *index*, *url*, flag *available* e timestamp *rateLimitedUntil*. Esse erro só e lançado na estrategia `'skip'` ou quando `'wait'` esgota sua única tentativa.

## Emissor prático: Server Actions do Next.js

Um padrão comum em produção e um painel admin construido com Next.js onde uma Server Action atualiza dados de negócio e emite um evento Signal para o bot. Essa abordagem mantém mutações de UI e reações do bot fracamente acopladas enquanto preserva um único fluxo de operador. A Server Action cuida de autorização e persistencia dentro da sua aplicação web, e o *Signal Client* cuida da entrega assíncrona no runtime do bot.

<CodeTabs>
  <Tab label="Server Action">
    ```typescript filename="app/actions/update-prefix.ts"
    'use server';

    import { SignalClient } from '@spraxium/signal-client';

    const client = new SignalClient({
      webhookUrls: [process.env.SIGNAL_WEBHOOK_URL_1!], hmacSecret: process.env.SIGNAL_HMAC_SECRET!, rateLimitStrategy: 'wait', });

    export async function updatePrefix(guildId: string, prefix: string) {
      // persista no seu banco de dados primeiro
      await db.guilds.update(guildId, { prefix });

      // depois emita o evento para o bot
      await client.send('config.update', guildId, {
        prefix, source: 'admin-panel', });
    }
    ```
  </Tab>
  <Tab label="API Route Handler">
    ```typescript filename="app/api/signal/route.ts"
    import { SignalClient } from '@spraxium/signal-client';
    import { NextResponse } from 'next/server';

    const client = new SignalClient({
      webhookUrls: [
        process.env.SIGNAL_WEBHOOK_URL_1!, process.env.SIGNAL_WEBHOOK_URL_2!, ], hmacSecret: process.env.SIGNAL_HMAC_SECRET!, rateLimitStrategy: 'skip', });

    export async function POST(req: Request) {
      const { event, guildId, payload } = await req.json();
      await client.send(event, guildId, payload);
      return NextResponse.json({ ok: true });
    }
    ```
  </Tab>
</CodeTabs>

## Diretrizes de modelagem de eventos

Prefira nomes orientados ao domínio que descrevam fatos de negócio em vez de ações de transporte. Um nome como *billing.status.changed* e mais fácil de versionar e raciocinar do que labels orientadas a ação que vazam detalhes internos de implementação. Mantenha nomes de evento em formato lowercase, separados por ponto, para manter consistencia com as convenções usadas em todo o sistema Signal.

Payloads devem ser intencionalmente pequenos e estaveis. Inclua apenas campos que o bot realmente precisa para tomar decisões e evolua o schema com mudanças aditivas sempre que possivel. Remover ou renomear um campo existente e uma breaking change que requer coordenar deploys do emissor e listener simultaneamente.

Use o filtro *events* no *FallbackConfig* para focar garantias de confiabilidade nos eventos mais críticos. Eventos que não aparecem na lista de filtro ainda são entregues pelo caminho normal, mas não serão enfileirados no store de fallback em caso de falha.

## Passo a passo do fluxo de entrega

<Steps>
  <Step title="Construir o envelope assinado">
    O método *send* delega ao *SignalEnvelopeBuilder*. O builder gera um nonce via `crypto.randomUUID()`, captura o `Date.now()` atual como *sentAt*, computa a assinatura HMAC-SHA256 e retorna um envelope completo com versão de protocolo `1`.
  </Step>
  <Step title="Selecionar o próximo webhook">
    *WebhookPool.next* retorna o próximo webhook disponível em ordem round-robin. Se *SendOptions.webhookIndex* foi fornecido, o pool retorna aquela entrada específica via *get*, ignorando a rotação. Se todos os webhooks estão rate-limited e nenhum índice foi forcado, o fluxo pula para o passo cinco.
  </Step>
  <Step title="Entregar via HTTP POST">
    O client envia uma requisição POST para a URL do webhook com o envelope serializado em JSON dentro do campo content de mensagem Discord. Se a resposta for 2xx, a entrega esta completa e o método retorna com sucesso.
  </Step>
  <Step title="Tratar respostas 429">
    Quando o webhook retorna HTTP 429, o pool marca aquele índice como rate-limited pela duração parseada do header `Retry-After` (default 1000 ms). O client tenta imediatamente o próximo webhook disponível. Essa rotação continua ate que um webhook tenha sucesso ou todos estejam esgotados.
  </Step>
  <Step title="Aplicar estrategia de rate-limit">
    Quando não restam webhooks, a *rateLimitStrategy* decide o resultado. Com `'skip'`, um *AllWebhooksRateLimitedError* e lançado imediatamente. Com `'wait'`, o client espera pelo cooldown mais próximo e tenta uma vez. Se o fallback estiver habilitado e o erro for capturavel, o envelope e enfileirado no store de fallback para retry posterior.
  </Step>
</Steps>

## Quando este modelo não e a ferramenta certa

Se você precisa de confirmação sincrona imediata da lógica do bot dentro do mesmo ciclo de vida da requisição, Signal geralmente não e o contrato certo. Chamadas diretas de serviço continuam mais simples e mais fáceis de rastrear para comunicação in-process dentro de uma única fronteira de serviço. Signal e projetado para entrega de eventos assíncrona e fire-and-forget onde o produtor não precisa de resposta do consumidor.

Da mesma forma, se seus eventos requerem garantias estritas de ordenação, uma fila dedicada como RabbitMQ ou Redis Streams e uma escolha melhor. Signal entrega mensagens tao rápido quanto o pool de webhooks permite, e a ordenação de retry após falhas e best-effort em vez de estritamente sequencial.

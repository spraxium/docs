## Por que Zod pertence a fronteira Signal

Listeners do bot são onde eventos externos se tornam efeitos colaterais internos. Mudanças de cargo, atualizações de configuração, ações de moderação e mutações de billing todas começam com um payload recebido. Usar Zod nessa fronteira permite rejeitar dados malformados antes que cheguem aos serviços de negócio. Isso não e opcional para bots em produção porque o payload origina de sistemas externos que podem evoluir independentemente, enviar formatos inesperados ou conter valores inválidos devido a bugs no código emissor.

O decorator *@OnSignal* aceita uma opção `schema` que recebe qualquer schema Zod. Quando presente, o *SignalProcessor* chama *safeParse* no payload do envelope antes de invocar o handler. Se a validação falhar, o handler nunca e chamado e um log diagnóstico e emitido quando o modo debug esta ativado. Se nenhum schema for configurado, o payload bruto e passado sem alteração, o que e útil para handlers que fazem sua própria lógica de parsing.

<Callout tone="success" title="Valide na fronteira, não dentro dos handlers">
  Anexar o schema ao <code>@OnSignal</code> garante que toda invocação do handler
  receba dados validados e tipados. Isso elimina verificações defensivas dentro do código de negócio
  e torna as assinaturas dos handlers confiáveis.
</Callout>

## Design de schema e configuração de listener

O padrão recomendado e definir schemas em arquivos dedicados e importa-los nas classes de listener. Essa separação mantém schemas reutilizaveis entre handlers e testaveis independentemente. Cada arquivo de schema exporta tanto a constante Zod quanto o tipo TypeScript inferido.

<CodeTabs>
  <Tab label="Schema" language="typescript">
```typescript filename="src/modules/signal/schemas/config-update.schema.ts"
import { z } from 'zod';

export const ConfigUpdateSchema = z.object({
  prefix: z.string().min(1).max(5), language: z.enum(['en', 'pt', 'es']).optional(), updatedBy: z.string().min(17).max(20), });

export type ConfigUpdatePayload = z.infer<typeof ConfigUpdateSchema>;
```
  </Tab>
  <Tab label="Listener" language="typescript">
```typescript filename="src/modules/signal/listeners/guild-config.signal.ts"
import { SignalListener, OnSignal } from '@spraxium/signal';
import type { SignalEnvelope } from '@spraxium/signal';
import {
  ConfigUpdateSchema, type ConfigUpdatePayload, } from '../schemas/config-update.schema';

@SignalListener()
export class GuildConfigSignal {
  @OnSignal('config.update', { schema: ConfigUpdateSchema })
  async onConfigUpdate(
    payload: ConfigUpdatePayload, envelope: SignalEnvelope, ): Promise<void> {
    // payload esta validado e fortemente tipado
    // envelope.guildId fornece contexto de tenant
  }
}
```
  </Tab>
</CodeTabs>

## Fluxo de validação no processador

Quando o *SignalProcessor* recebe um envelope validado, ele verifica se o handler resolvido possui um schema anexado. Se possuir, o processador chama *schema.safeParse(envelope.payload)* e inspeciona o resultado. Em caso de sucesso, *result.data* substitui o payload bruto e e passado ao handler. Em caso de falha, o handler e ignorado completamente e uma entrada de log diagnóstico e emitida com os detalhes dos problemas Zod.

Esse fluxo significa que seu handler tem a garantia de receber dados que correspondem ao schema. Não ha necessidade de blocos try-catch em torno do acesso ao payload, não ha necessidade de type assertions manuais e não ha risco de propriedades undefined causando erros de runtime. O schema atua como contrato de compile-time e runtime simultaneamente.

<Steps>
  <Step title="Verificação de presença do schema">
    O processador verifica se o handler possui um schema Zod. Se nenhum schema estiver anexado, o *envelope.payload* bruto e passado ao handler sem alteração. Isso permite que handlers que não precisam de validação pulem a sobrecarga completamente.
  </Step>
  <Step title="Execução do safe parse">
    O processador chama *schema.safeParse(envelope.payload)* que nunca lança exceção. Ele retorna uma union discriminada: `{ success: true, data: T }` com o payload parseado e tipado, ou `{ success: false, error: ZodError }` com os problemas de validação.
  </Step>
  <Step title="Caminho de sucesso">
    Na validação bem-sucedida, *result.data* substitui o payload bruto. Essa versão parseada tem defaults aplicados, transforms executados e tipos estreitados. O handler recebe esse valor tipado como primeiro argumento.
  </Step>
  <Step title="Caminho de falha">
    Na validação falha, o handler nunca e invocado. Se o modo debug estiver ativado, o processador loga os problemas Zod como um array JSON, incluindo o path, tipo esperado e valor recebido para cada campo com falha. O envelope e descartado silenciosamente para prevenir dados corrompidos de chegar a lógica de negócio.
  </Step>
</Steps>

## Múltiplos handlers com schemas diferentes

Uma única classe *@SignalListener* pode conter múltiplos métodos *@OnSignal* com nomes de evento e schemas diferentes. Cada handler e registrado independentemente no *SignalRouter*, e cada schema e validado independentemente durante o despacho. Isso permite agrupar handlers de domínio relacionados em uma classe enquanto mantém contratos de payload estritos por evento.

```typescript filename="src/modules/signal/listeners/guild-config.signal.ts"
import { SignalListener, OnSignal } from '@spraxium/signal';
import type { SignalEnvelope } from '@spraxium/signal';
import { z } from 'zod';

const PrefixSchema = z.object({
  prefix: z.string().min(1).max(5), });

const LanguageSchema = z.object({
  language: z.enum(['en', 'pt', 'es']), updatedBy: z.string(), });

@SignalListener()
export class GuildConfigSignal {
  @OnSignal('config.prefix.update', { schema: PrefixSchema })
  async onPrefixUpdate(
    payload: z.infer<typeof PrefixSchema>, envelope: SignalEnvelope, ): Promise<void> {
    // Trata mudancas de prefixo
  }

  @OnSignal('config.language.update', { schema: LanguageSchema })
  async onLanguageUpdate(
    payload: z.infer<typeof LanguageSchema>, envelope: SignalEnvelope, ): Promise<void> {
    // Trata mudancas de idioma
  }
}
```

## Padrões avancados de schema

Para eventos de domínio complexos, Zod suporta padrões poderosos de composição que mantêm schemas DRY e expressivos. Discriminated unions permitem validar diferentes formatos de payload baseados em um campo discriminador compartilhado. Intersections permitem combinar campos base com extensões específicas do evento.

<CodeTabs>
  <Tab label="Union discriminada" language="typescript">
```typescript filename="src/modules/signal/schemas/billing.schema.ts"
import { z } from 'zod';

const ActivatedSchema = z.object({
  action: z.literal('activated'), planId: z.string(), userId: z.string(), expiresAt: z.number(), });

const CancelledSchema = z.object({
  action: z.literal('cancelled'), userId: z.string(), reason: z.string().optional(), });

export const BillingEventSchema = z.discriminatedUnion('action', [
  ActivatedSchema, CancelledSchema, ]);

export type BillingEventPayload = z.infer<typeof BillingEventSchema>;
```
  </Tab>
  <Tab label="Base + extensão" language="typescript">
```typescript filename="src/modules/signal/schemas/moderation.schema.ts"
import { z } from 'zod';

const BaseModerationSchema = z.object({
  moderatorId: z.string(), targetUserId: z.string(), reason: z.string().min(1), });

export const BanSchema = BaseModerationSchema.extend({
  duration: z.number().positive().optional(), });

export const MuteSchema = BaseModerationSchema.extend({
  duration: z.number().positive(), channelIds: z.array(z.string()).optional(), });
```
  </Tab>
</CodeTabs>

## Responsabilidades de retry

O pacote bot-side *@spraxium/signal* não inclui fila de retry interna ou worker. Se uma validação Zod falhar, o envelope e descartado e nenhuma tentativa automática e feita. Isso e por design porque o bot deve permanecer um consumidor deterministico de eventos. A lógica de retry pertence ao lado emissor, onde o pacote *@spraxium/signal-client* fornece persistencia de fallback com backoff exponencial.

Essa separação mantém cada lado focado em sua responsabilidade. O bot valida e despacha envelopes confiáveis. O emissor cuida de confiabilidade de entrega, agendamento de retries e monitoramento de dead-letter. Se você precisar reprocessar um envelope que falhou, o emissor deve corrigir o formato do payload e reenvia-lo, em vez de o bot tolerar dados inválidos.

<Callout tone="warning" title="A responsabilidade de retry e do lado produtor">
  O bot-side <code>@spraxium/signal</code> intencionalmente não faz retry de payloads que falharam. Se
  retries de entrega são necessarios, configure-os no lado emissor usando as configurações de fallback
  do <code>@spraxium/signal-client</code> com store de arquivo ou Redis.
</Callout>

## Testando com createSignalPayload

O pacote `@spraxium/signal` exporta um helper *createSignalPayload* que constrói envelopes não assinados para fins de teste. Esse helper gera um UUID nonce e um timestamp atual, retornando um *UnsignedEnvelope* que pode ser usado em testes unitarios para verificar comportamento dos handlers sem montar o pipeline Signal completo.

```typescript filename="test/guild-config.signal.test.ts"
import { createSignalPayload } from '@spraxium/signal';

const testEnvelope = createSignalPayload('config.update', '123456789012345678', {
  prefix: '!', language: 'en', });

// testEnvelope possui: v, event, guildId, nonce, sentAt, payload
// Sem campo signature (nao assinado, apenas para testes)
```
